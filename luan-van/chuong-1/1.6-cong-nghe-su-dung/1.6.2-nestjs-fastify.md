# 1.6.2. NestJS và Fastify

**NestJS** là framework phát triển server-side cho Node.js, được xây dựng hoàn toàn trên TypeScript và lấy cảm hứng mạnh mẽ từ kiến trúc Angular. NestJS ra đời năm 2017 với mục tiêu giải quyết vấn đề mà Express để ngỏ: Express là một micro-framework tối giản, cho phép tự do tổ chức code theo bất kỳ cấu trúc nào — điều này lợi thế với dự án nhỏ nhưng dễ dẫn đến codebase không nhất quán khi dự án lớn và nhiều người tham gia. NestJS áp đặt một cấu trúc rõ ràng dựa trên các khái niệm Module, Controller, Service, Guard, Interceptor, và Pipe — mỗi loại có trách nhiệm được định nghĩa chặt chẽ, làm cho codebase dễ đọc, dễ test, và dễ mở rộng theo thời gian.

---

## Kiến Trúc NestJS

### Module

**Module** là đơn vị tổ chức cơ bản của NestJS — tương đương NgModule trong Angular. Mỗi module đóng gói một tính năng hoặc domain logic: khai báo Controller và Service thuộc về nó, import các module khác mà nó cần, và export những Provider mà module khác có thể inject. `DaPlex-API` được tổ chức thành các module theo domain: `AuthModule` (xác thực, phát hành token), `MediaModule` (quản lý phim và series, bao gồm cả BullMQ consumer lắng nghe tiến trình encode), `UserModule` (hồ sơ người dùng), `ExternalStoragesModule` (tích hợp đa cloud storage), `RolesModule` (quản lý vai trò và permission), cùng nhiều module chuyên biệt cho từng storage provider.

### Controller và Service

**Controller** chịu trách nhiệm nhận HTTP request, trích xuất tham số, và trả về HTTP response. Controller không chứa business logic — nó chỉ là tầng routing và serialization. `MediaController` là controller có nhiều kết nối nhất trong `DaPlex-API` với 56 liên kết trong knowledge graph, phản ánh vai trò trung tâm của nó trong hệ thống: xử lý các endpoint CRUD cho phim, series, episode, và cung cấp URL phát lại. **Service** chứa toàn bộ business logic — `MediaService` với 121 kết nối là node có degree cao nhất trong toàn bộ codebase DaPlex, cho thấy nó là điểm tích hợp trung tâm: phối hợp giữa database (Mongoose), hàng đợi encode (BullMQ), và storage layer.

### Guard, Interceptor, và Pipe

| Thành Phần | Triển Khai Trong DaPlex-API | Mô Tả |
|---|---|---|
| **Guard** | `JwtAuthGuard`, `RolesGuard` | Quyết định request có được phép thực thi hay không — xác thực JWT (RS256), tra cứu user từ MongoDB, kiểm tra permission bits (`ADMINISTRATOR`, `MANAGE_MEDIA`). Triển khai interface `CanActivate`, áp dụng qua `@UseGuards()` |
| **Interceptor** | Response envelope, request logger | Bao bọc request-response để chuẩn hóa format trả về (`{ success, data, message }`), log thời gian xử lý, và transform dữ liệu trước khi serialize JSON |
| **Pipe** | `ValidationPipe` (global) | Validation và transformation tham số đầu vào — kết hợp với `class-validator` để tự động validate DTO theo decorator annotation, từ chối request sai format trước khi chạm đến Service |

## Fastify Làm HTTP Adapter

NestJS mặc định sử dụng **Express** làm HTTP adapter nhưng hỗ trợ thay thế bằng **Fastify** thông qua `@nestjs/platform-fastify`. Fastify là HTTP framework Node.js hiệu năng cao, được tối ưu hóa về tốc độ parse JSON và serialize response thông qua `fast-json-stringify` và `ajv` — hai thư viện thực hiện JSON serialization/validation dựa trên JSON Schema thay vì duyệt object thông thường. Theo benchmark chính thức, Fastify xử lý khoảng 30.000–75.000 request/giây tùy cấu hình, so với Express khoảng 10.000–15.000 request/giây trong điều kiện tương đương.

> Lý do DaPlex chọn Fastify thay vì Express không chỉ là hiệu năng thuần túy. Fastify có hệ thống **plugin** chính thức với lifecycle hook đầy đủ (onRequest, preHandler, onSend, onError), hỗ trợ **schema-based validation** tích hợp, và **logger** tích hợp dựa trên `pino` — một trong những logger Node.js nhanh nhất hiện tại. `DaPlex-API` tận dụng pino logger để ghi structured log (JSON format) với request ID tự động, tiện lợi cho việc trace request qua distributed system và tích hợp với log aggregation service. Một điểm khác biệt quan trọng là Fastify hỗ trợ **async/await** native và không yêu cầu callback-style như Express, giảm nguy cơ lỗi unhandled rejection.

## TypeScript End-to-End

Một lợi thế đáng kể của việc chọn NestJS (TypeScript) cùng với Angular (TypeScript) là khả năng chia sẻ type definition giữa frontend và backend. `DaPlex-API` định nghĩa các DTO và interface mô tả cấu trúc API response — ví dụ `MediaResponseDto` mô tả object phim trả về với đầy đủ field và kiểu dữ liệu. Những definition này có thể được chia sẻ với `daplex-dune-v2` thông qua package hoặc git submodule, đảm bảo khi API thay đổi cấu trúc response, TypeScript compiler phát hiện ngay lập tức tất cả các Component frontend đang dùng field đó và cần cập nhật — thay vì phát hiện lỗi lúc runtime qua lỗi `undefined is not an object`.

## Lý Do Chọn NestJS

> So với Express thuần, NestJS mang lại cấu trúc nhất quán từ đầu, quan trọng với dự án có nhiều module (auth, media, user, storage, transcoder) cần maintain lâu dài. So với Hapi.js — một lựa chọn opinionated khác — NestJS có cộng đồng lớn hơn đáng kể, tài liệu đầy đủ hơn, và tích hợp tốt hơn với các công cụ trong hệ sinh thái TypeScript như `class-validator`, `class-transformer`, và `@nestjs/swagger` để tự động tạo tài liệu OpenAPI. Khả năng kết hợp với Fastify giúp NestJS vừa giữ được cấu trúc của framework opinionated vừa không đánh đổi hiệu năng — một yếu tố không thể bỏ qua khi API server phải xử lý đồng thời nhiều request upload và streaming từ người dùng.
