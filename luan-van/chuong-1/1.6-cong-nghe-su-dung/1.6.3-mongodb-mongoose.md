# 1.6.3. MongoDB và Mongoose

**MongoDB** là hệ quản trị cơ sở dữ liệu NoSQL theo mô hình document, được phát triển bởi MongoDB Inc. và ra mắt năm 2009. Thay vì tổ chức dữ liệu theo hàng và cột như cơ sở dữ liệu quan hệ (RDBMS), MongoDB lưu trữ dữ liệu dưới dạng **document** BSON (Binary JSON) — một định dạng nhị phân tương thích JSON hỗ trợ kiểu dữ liệu phong phú hơn, bao gồm ObjectId, Date, Binary, và Decimal128. Mỗi document là một đối tượng tự mô tả (self-describing), có thể chứa các trường lồng nhau (nested fields) và mảng với độ sâu tùy ý, không yêu cầu tất cả document trong cùng một collection phải có cùng cấu trúc.

---

## Tại Sao MongoDB Phù Hợp Với Hệ Thống VoD

Dữ liệu media trong hệ thống VoD có đặc điểm tự nhiên phù hợp với mô hình document. Thông tin về một bộ phim không chỉ là các trường đơn giản (tên, năm phát hành, điểm đánh giá) mà bao gồm nhiều cấu trúc lồng nhau và mảng: danh sách thể loại (genres), danh sách diễn viên kèm vai diễn và hình ảnh (cast array), thông tin đạo diễn, metadata từ nhiều nguồn (TMDB ID, TVDB ID, IMDB ID), danh sách ngôn ngữ audio và phụ đề có sẵn, và các mức chất lượng encode với thông tin bitrate/codec tương ứng.

Trong RDBMS, mô hình hóa cấu trúc này đòi hỏi nhiều bảng và JOIN phức tạp: bảng `movies`, bảng `genres`, bảng `movie_genres` (many-to-many), bảng `people`, bảng `movie_cast`, bảng `video_qualities`, v.v. Mỗi query lấy thông tin đầy đủ một bộ phim sẽ JOIN ít nhất 4–5 bảng, tốn kém về mặt query planning và index lookup. MongoDB cho phép nhúng (embed) toàn bộ thông tin liên quan vào một document, làm cho query đọc chỉ cần truy cập một collection duy nhất — phù hợp với pattern đọc chủ yếu (read-heavy) của hệ thống VoD nơi tỷ lệ đọc/ghi có thể lên đến 100:1.

Ngoài ra, metadata từ TMDB và TVDB có cấu trúc có thể thay đổi theo phiên bản API — các trường mới có thể được thêm hoặc trường cũ thay đổi kiểu dữ liệu. Schema linh hoạt của MongoDB cho phép DaPlex lưu trực tiếp metadata thô từ API bên ngoài mà không cần migration phức tạp khi cấu trúc thay đổi, điều mà RDBMS xử lý khó khăn hơn nhiều khi đã có dữ liệu production.

## Mongoose ODM

**Mongoose** là Object-Document Mapper (ODM) cho MongoDB trên Node.js, cung cấp layer trừu tượng với schema definition, validation, middleware (hooks), và query builder. Mặc dù MongoDB bản thân không bắt buộc schema, Mongoose áp đặt schema ở tầng application — mỗi collection được đại diện bởi một `Schema` định nghĩa các trường, kiểu dữ liệu, validation rule, và default value. Điều này cung cấp sự an toàn kiểu dữ liệu ở tầng ODM mà không mất tính linh hoạt của MongoDB khi cần thiết (Mongoose hỗ trợ `Mixed` type và `strict: false` cho các trường động).

Trong `DaPlex-API`, mỗi entity chính có một Mongoose Schema và Model tương ứng. `MediaService` tương tác với Model thông qua các phương thức Mongoose như `find()`, `findById()`, `findOneAndUpdate()`, và `aggregate()` — tất cả đều trả về Promise hoặc Mongoose Query object có thể await trong async/await context của NestJS.

## Thiết Kế Schema Media

Schema cho entity `Media` (đại diện cho một bộ phim hoặc series) trong DaPlex nhúng trực tiếp thông tin thể loại, diễn viên, và các mức chất lượng encode vào document thay vì tạo collection riêng:

```typescript
const MediaSchema = new Schema({
  title:       { type: String, required: true, index: true },
  type:        { type: String, enum: ['movie', 'series'], required: true },
  year:        { type: Number },
  overview:    { type: String },
  genres:      [{ type: String }],
  cast: [{
    name:      { type: String },
    character: { type: String },
    profilePath: { type: String },
    order:     { type: Number },
  }],
  externalIds: {
    tmdbId:    { type: Number, sparse: true },
    tvdbId:    { type: Number, sparse: true },
    imdbId:    { type: String, sparse: true },
  },
  qualities: [{
    codec:       { type: String, enum: ['h264', 'h265', 'vp9', 'av1'] },
    resolution:  { type: String },
    bitrate:     { type: Number },
    mpdUrl:      { type: String },
    status:      { type: String, enum: ['pending', 'processing', 'ready', 'failed'] },
  }],
  posterPath:   { type: String },
  backdropPath: { type: String },
  status:       { type: String, enum: ['draft', 'processing', 'published'], default: 'draft' },
}, { timestamps: true });
```

Mảng `qualities` nhúng thông tin về từng phiên bản encode đã hoàn thành, bao gồm codec, độ phân giải, và URL của file MPD trên cloud storage. Khi `DaPlex-Transcoder` hoàn thành encode một chất lượng, nó cập nhật phần tử tương ứng trong mảng này thông qua `MediaService` — tránh cần một collection `VideoQuality` riêng biệt và join phức tạp khi client hỏi "bộ phim này có những mức chất lượng nào đang sẵn sàng phát?".

## Indexing và Hiệu Năng

MongoDB hỗ trợ nhiều loại index để tăng tốc độ truy vấn. `DaPlex-API` tạo các index sau trên collection `media`:

| Loại Index | Trường | Mục Đích |
|---|---|---|
| **Single-field index** | `title`, `type` | Tăng tốc tìm kiếm theo tên phim và lọc theo loại (phim/series) |
| **Text index** | `title`, `overview` | Hỗ trợ full-text search với toán tử `$text`, tìm kiếm từ khóa mà không cần regex scan toàn collection |
| **Compound index** | `{ status: 1, type: 1, year: -1 }` | Tối ưu query trang chủ lấy danh sách phim `published` sắp xếp theo năm phát hành giảm dần |
| **Sparse index** | `externalIds.tmdbId`, `externalIds.imdbId` | Chỉ tạo entry cho document có trường đó, tiết kiệm không gian khi nhiều document không có TMDB/IMDB ID |

Ngoài indexing, `DaPlex-API` sử dụng **projection** để chỉ lấy các trường cần thiết thay vì toàn bộ document — ví dụ, endpoint lấy danh sách phim chỉ cần `title`, `year`, `posterPath`, và `status`, không cần mảng `cast` có thể chứa 50+ phần tử và mảng `qualities` với đầy đủ URL.

## Aggregation Pipeline

MongoDB **Aggregation Pipeline** cho phép xử lý dữ liệu qua chuỗi các stage biến đổi document, tương đương HAVING/GROUP BY/JOIN trong SQL nhưng linh hoạt hơn. `DaPlex-API` sử dụng aggregation cho các query phức tạp như: đếm số phim theo thể loại (dùng `$unwind` trên mảng genres, sau đó `$group`), lấy danh sách phim cùng thể loại với phim đang xem (`$match` + `$sample` để lấy ngẫu nhiên), và thống kê dashboard admin (tổng phim, dung lượng lưu trữ, tình trạng encode theo trạng thái).

## Lý Do Chọn MongoDB

> Quyết định sử dụng MongoDB thay vì PostgreSQL — lựa chọn phổ biến cho ứng dụng web hiện đại — dựa trên đặc điểm của dữ liệu media. Metadata phim từ TMDB/TVDB có cấu trúc không đồng nhất (series có season/episode, phim có runtime nhưng series thì không), và thường xuyên thay đổi khi API nguồn cập nhật thêm trường mới. Schema linh hoạt của MongoDB xử lý điều này tự nhiên, trong khi PostgreSQL với JSONB column cũng linh hoạt nhưng mất đi lợi thế của query planner quan hệ khi dữ liệu quan trọng nằm trong JSON. Mặt khác, DaPlex không có yêu cầu transaction phức tạp hay foreign key constraint nghiêm ngặt giữa các entity — những điểm mạnh cốt lõi của RDBMS — nên sự đánh đổi giữa tính nhất quán ACID của PostgreSQL và tính linh hoạt schema của MongoDB nghiêng về phía MongoDB cho bài toán này.
