# 3.3. ERD — Sơ Đồ Quan Hệ Thực Thể

Phần này ghi lại cấu trúc field và quan hệ giữa 18 collection của DaPlex, tổ chức theo bảy domain chức năng và hai mức chi tiết: ERD per-domain và ERD tổng hợp toàn cảnh. Nội dung bổ sung cho phần 3.2 — nếu 3.2 giải thích lý do thiết kế thì 3.3 ghi lại chính xác schema thực tế.

---

## Tổng Quan và Ngữ Cảnh Hệ Thống

Phần này trình bày sơ đồ quan hệ thực thể (ERD) đầy đủ của cơ sở dữ liệu DaPlex theo hai cấp độ: ERD phân rã theo domain để làm rõ quan hệ nội bộ từng nhóm, và ERD tổng hợp để thấy toàn cảnh 18 collection cùng lúc. Cách tiếp cận hai cấp này bổ sung cho phần 3.2 — trong khi 3.2 giải thích nguyên lý thiết kế và lý do lựa chọn (why), phần 3.3 ghi lại chính xác cấu trúc field và quan hệ giữa các collection (what).

Toàn bộ 18 collection của DaPlex được nhóm thành bảy domain chức năng theo mức độ gắn kết ngữ nghĩa. **Auth/User Domain** quản lý danh tính và phân quyền; **Content Domain** là lõi catalog nội dung phim và series; **Taxonomy Domain** cung cấp các chiều phân loại (thể loại, hãng sản xuất, tag); **User Activity Domain** ghi lại hành vi người dùng (lịch sử xem, đánh giá, playlist); **Storage Domain** theo dõi file vật lý trên object storage; **System Config Domain** lưu cấu hình hệ thống toàn cục. Riêng **Audit/Notification Domain** được tách sang database connection thứ hai (DATABASE_B) do đặc trưng append-only và nhu cầu retention policy độc lập.

**Quy ước biểu diễn:** `ref:X` là tham chiếu đến collection X (MongoDB DBRef qua Mongoose `populate`); `[ref:X]` là mảng tham chiếu; `embed:X` là subdocument nhúng trong document cha (`_id: false`, không có collection riêng); `[embed:X]` là mảng subdocument nhúng. Tất cả `_id` đều là BigInt Snowflake ID trừ Collection `Counter` dùng string key.

## Nhóm 1 — Auth/User Domain

Auth/User Domain gồm hai collection `User` và `Role` liên kết với nhau qua quan hệ many-to-many. Quan hệ này được biểu diễn theo cả hai chiều: `User.roles[]` lưu danh sách role của user, còn `Role.users[]` là backreference ngược lại. Trong thực tế query, backreference `Role.users[]` luôn được exclude khi load user cho authentication guard (`populate('roles', { users: 0 })`) để tránh load tập dữ liệu lớn không cần thiết. Field `permissions: number` của Role dùng bitfield — mỗi bit tương ứng một permission, cho phép kiểm tra tập hợp permission bằng phép AND một lần thay vì iterate qua danh sách.

## Nhóm 2 — Content Domain

Content Domain là domain phức tạp nhất với sáu collection và hai embedded subdocument. Collection `Media` đóng vai trò trung tâm — nó sử dụng single-collection polymorphism để chứa cả phim lẻ (với embedded `Movie`) lẫn series (với embedded `TVShow`), tránh duplicate logic query khi search/filter đa kiểu. `TVEpisode` có collection riêng thay vì nhúng vào `TVShow` vì mỗi tập có thể được query độc lập (tìm tập theo số, sort, filter theo pStatus). `Credit` tồn tại như một bảng nối giữa `Media` và `People`, lưu thêm thông tin vai trò (`job`, `character`) mà không thể đặt vào bên trong `People`.

## Nhóm 3 — Taxonomy Domain

Taxonomy Domain gồm năm collection phục vụ phân loại nội dung: `Genre`, `Production`, `MediaTag`, `MediaCollection`, `ChapterType`. Tất cả đều có quan hệ many-to-many hai chiều với `Media`.

> DaPlex lưu ID array ở cả hai phía (`Media.genres[]` và `Genre.media[]`) thay vì dùng junction collection. Cách này phù hợp với workload đọc nhiều: load một `Media` document kèm tất cả taxonomy trong một query `populate`, không cần join.

`MediaCollection` là grouping logic (Trending, Featured, Thể loại nổi bật) do admin tạo và curate thủ công, khác với `Genre` là phân loại nội dung. `ChapterType` định nghĩa các loại timestamp (Intro, Outro, Recap, Credits) được đánh dấu trong `Movie.chapters[]` và `TVEpisode.chapters[]`.

## Nhóm 4 — User Activity Domain

User Activity Domain ghi lại hành vi người xem thông qua ba collection: `History` (lịch sử xem và resume playback), `Rating` (đánh giá nội dung), `Playlist` (danh sách tự tạo). `History` là collection quan trọng nhất về mặt hiệu suất — mỗi lần người dùng xem đến giây thứ N, vị trí đó được upsert vào `History.time` để hỗ trợ resume. Field `watched: number` đếm số lần xem hoàn chỉnh, dùng để tính trending. `Rating` có index compound unique `{ media: 1, user: 1 }` — mỗi user chỉ có một đánh giá mỗi media; khi user đánh giá lại, document cũ được update và Media `ratingScore`/`ratingCount` được điều chỉnh theo delta.

> `Playlist` dùng embedded `PlaylistItem[]` thay vì collection riêng vì item không bao giờ cần query độc lập — chúng luôn được load trong ngữ cảnh playlist.

## Nhóm 5 — Storage Domain

Storage Domain quản lý vòng đời file vật lý từ lúc upload đến khi serve. `ExternalStorage` lưu credentials (encrypted) và cấu hình cho từng object storage provider (Azure Blob, Cloudflare R2, AWS S3, OneDrive, ...). `MediaStorage` là record theo dõi một file cụ thể trên storage — một media có thể có nhiều `MediaStorage` document: một `type=source` cho file gốc, nhiều `type=stream_video` cho các bản encode (360p/720p/1080p H.264, ...), nhiều `type=stream_audio` cho audio tracks. ## Luồng Dữ Liệu Xem Video

Để minh họa cách các collection tương tác trong một tình huống thực tế, xét luồng dữ liệu khi người dùng mở trang xem một tập phim series. Client gửi request lên API với `mediaId` và `episodeId`. API thực hiện hai query song song: load `Media` document để lấy metadata series (tên, mô tả, genres, pStatus) và load `TVEpisode` document để lấy thông tin tập (số tập, tên, pStatus). Sau khi kiểm tra visibility và pStatus, API query `MediaStorage` với điều kiện `{ episode: episodeId, type: MANIFEST }` để lấy danh sách file stream đã encode. Với mỗi `MediaStorage` document, API resolve CDN URL từ `ExternalStorage.publicUrl` kết hợp với `MediaStorage.path` và trả về `StreamManifest` JSON cho client.

Đồng thời, nếu user đã đăng nhập, API upsert document `History` (`{ user, media }`) với thời gian bắt đầu xem. Phía client load thêm document `Rating` (`{ user, media }`) nếu tồn tại để hiển thị đánh giá hiện tại của user. Khi user dừng xem ở giây thứ N, client gửi request cập nhật `History.time = N` và `History.paused = true`. Nếu user xem hết tập, `History.watched` được tăng thêm 1 và `Media.views`/`TVEpisode.views` được increment atomically.

Thứ nhất, **nhiều quan hệ many-to-many được lưu trực tiếp dưới dạng mảng ID** trong document (`Media.genres[]`, `Media.tags[]`, `Role.users[]`) thay vì qua junction table như SQL — điều này hợp lý với workload đọc nhiều của VoD platform vì toàn bộ metadata được load trong một query duy nhất. Thứ hai, **các subdocument embedded** (`Movie`, `TVShow`, `PlaylistItem`, `MediaStorageStream`) không có collection riêng, nghĩa là dữ liệu của chúng luôn được load cùng document cha — đây là lựa chọn có chủ đích cho dữ liệu không bao giờ cần query độc lập mà luôn cần trong ngữ cảnh của document cha.
