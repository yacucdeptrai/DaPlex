# 3.2. Thiết Kế Cơ Sở Dữ Liệu

Phần này trình bày các quyết định thiết kế cơ sở dữ liệu của DaPlex — từ lựa chọn MongoDB Atlas làm hệ quản trị chính, các nguyên tắc schema áp dụng nhất quán, đến cách phân vùng collection theo hai database connection. Hiểu rõ thiết kế này là nền tảng để đọc ERD chi tiết ở phần 3.3 và lý giải các quyết định kiến trúc trong phần triển khai.

---

## Lựa Chọn MongoDB và Nguyên Tắc Thiết Kế Schema

DaPlex sử dụng MongoDB Atlas làm cơ sở dữ liệu chính cho toàn bộ dữ liệu nghiệp vụ. Lựa chọn này xuất phát từ đặc trưng của domain nội dung video: một `Media` document cần nhúng (embed) nhiều cấu trúc con có schema khác nhau tùy theo loại nội dung — bộ phim nhúng `Movie` subdocument với pointer đến file nguồn, series nhúng `TVShow` subdocument với mảng tham chiếu đến các tập phim.

> Mô hình dữ liệu dạng document của MongoDB cho phép biểu diễn cấu trúc phân cấp này tự nhiên hơn so với relational model đòi hỏi nhiều JOIN table.

Ba nguyên tắc thiết kế schema được tuân thủ nhất quán trong toàn bộ cơ sở dữ liệu DaPlex.

**Thứ nhất: BigInt Snowflake ID.** Toàn bộ collection dùng `_id: BigInt` thay vì ObjectId mặc định của MongoDB. ID được sinh bởi `createSnowFlakeId()` — một hàm tiện ích tạo 64-bit Snowflake ID có tính chất tăng đơn điệu theo thời gian, đảm bảo sắp xếp tự nhiên theo thứ tự tạo. Snowflake ID không phụ thuộc vào MongoDB server (generated tại application layer), cho phép insert phân tán mà không có ID collision. Để serialize BigInt qua JSON, `applyBigIntPatches()` được gọi ở tầng application để override `JSON.stringify()`.

**Thứ hai: Pattern TrackableDoc.** Các collection quan trọng (Media, TVEpisode, Role, Setting) extend lớp `TrackableDoc<T>` — lớp này thêm field `_original` lưu snapshot state trước khi thay đổi. Hook Mongoose `post('init')` tự động populate `_original` khi document được load từ database. `AuditLogModule` so sánh `_original` với state mới để generate `AuditLogChange[]`, tạo ra audit trail chính xác không cần thêm logic trong service.

**Thứ ba: Đa ngôn ngữ qua `_translations` map.** Các collection cần i18n (Media, TVEpisode, Genre) chứa field `_translations: { [lang: string]: TranslatedFields }`. Ví dụ `media._translations.vi` chứa `title`, `slug`, và `overview` bằng tiếng Việt.

> Thay vì tạo collection riêng cho translation (pattern thường dùng trong SQL), DaPlex nhúng trực tiếp vào document cha — đọc nhanh hơn và phù hợp với workload read-heavy.

## Phân Vùng Hai Database Connection

Tất cả collection được phân vào hai MongoDB connection. **DATABASE_A** (biến `DATABASE_URL`) là database chính chứa toàn bộ dữ liệu nghiệp vụ: User, Role, Media, TVEpisode, MediaStorage, ExternalStorage, Genre, Production, MediaTag, MediaCollection, ChapterType, Playlist, History, Rating, Setting, DriveSession. **DATABASE_B** (biến `DATABASE_URL_B`) chứa hai collection có đặc trưng khác: `AuditLog` và `Notification` — cả hai đều là append-only (không update), tăng trưởng liên tục theo thời gian, và có thể áp dụng retention policy riêng (xóa log cũ hơn 1 năm, xóa notification đã đọc sau 30 ngày) mà không ảnh hưởng đến DATABASE_A. Cách phân vùng này còn cho phép backup DATABASE_A và DATABASE_B theo tần suất và chiến lược riêng biệt.

## Collection User và Role: Hệ Thống Xác Thực và Phân Quyền

`User` là collection trung tâm của hệ thống xác thực. Mỗi document User có các field định danh bắt buộc (`username`, `email` — đều có index `unique`), `password` lưu dạng bcrypt hash, và hai trạng thái boolean quan trọng: `verified` (đã xác thực email) và `banned` (bị khóa tài khoản). Field `owner: boolean` (optional) đánh dấu tài khoản owner đặc biệt — chỉ có một user có `owner: true` trong toàn hệ thống; user này có `hasPermission = true` mặc định trong mọi `RolesGuard`, không phụ thuộc vào `roles` hay `permissions`. Field `activationCode` và `recoveryCode` là các mã dùng một lần cho luồng xác thực email và đặt lại mật khẩu — sau khi dùng, hai field này được `$unset` khỏi document. `avatar` và `banner` được lưu dạng embedded `UserFile` subdocument (chứa URL và metadata) thay vì chỉ lưu URL string, cho phép lưu thêm thông tin như kích thước file và storage provider.

```
User {
  _id: BigInt (Snowflake)
  username: string (unique, indexed)
  email: string (unique, indexed)
  password: string (bcrypt hash)
  nickname?: string
  birthdate: ShortDate { year, month, day }
  about?: string
  roles: [ref:Role] (populated on auth)
  verified: boolean = false
  banned: boolean = false
  owner?: boolean
  activationCode?: string (unset sau khi dùng)
  recoveryCode?: string (unset sau khi dùng)
  avatar?: UserFile { name, mimeType, size, ... }
  banner?: UserFile
  settings: UserSettings (embedded)
  lastActiveAt: Date
  createdAt, updatedAt (timestamps)
}
```

`Role` là collection định nghĩa vai trò với hệ thống phân quyền bitfield. Field `permissions: number` lưu tập hợp permission dưới dạng bitmask — ví dụ `0x3` có nghĩa là bit 0 và bit 1 đều bật, tương ứng với hai permission cụ thể. `PermissionsService.hasPermission()` dùng phép AND bitwise để kiểm tra từng role: `role.permissions & UserPermission.EDIT_MEDIA`. Field `position: number` xác định thứ tự ưu tiên khi hiển thị nhiều role — role có position nhỏ hơn hiện thị trước. Mảng `users` là backreference từ Role về User, được exclude mặc định trong `findUserAuthGuard()` (`populate('roles', { users: 0 })`).

```
Role {
  _id: BigInt (Snowflake)
  name: string (indexed)
  color?: number (màu hex)
  users: [ref:User] (exclude khi populate auth)
  permissions: number (bitfield)
  position: number (indexed)
  createdAt, updatedAt (timestamps)
  _original (TrackableDoc)
}
```

## Collection Media: Catalog Nội Dung Trung Tâm

`Media` là collection phức tạp nhất trong hệ thống, phục vụ cả phim lẻ (`type: 'movie'`) lẫn series (`type: 'tv'`).

> Thiết kế dùng một collection duy nhất thay vì tách Movie/Series ra hai collection riêng vì phần lớn field là chung (title, genres, credits, ratings, visibility), và việc search/filter đa kiểu (vừa phim vừa series theo genre) hiệu quả hơn trên một collection.

Trường `pStatus: number` là **public status** — trạng thái hiển thị với người dùng cuối (đang phát, kết thúc, sắp ra mắt). Trường `status: string` là **processing status** — trạng thái xử lý nội bộ (upload pending, encoding, ready). Hai trường này tách biệt có chủ đích: một media có thể đang encode (status = 'encoding') nhưng vẫn hiển thị public với tập cũ đã sẵn sàng (pStatus = 'ongoing').

Hệ thống đếm view được denormalize trực tiếp vào `Media` document thay vì tính từ collection `History`. Bốn counter `views`, `dailyViews`, `weeklyViews`, `monthlyViews` được update bằng MongoDB `$inc` atomic khi user xem — đọc cực nhanh (O(1), không cần aggregate), đánh đổi bằng write amplification nhỏ. `ratingCount`, `ratingScore`, và `ratingAverage` cũng được denormalize tương tự từ `Rating` collection.

`visibility: number` (enum `MediaVisibility`) kiểm soát ai có thể xem — PUBLIC, UNLISTED, PRIVATE, hay chỉ member cụ thể. Field `addedBy` (ref User) track ai đã tạo media document. Mảng `inCollections` là backreference từ `MediaCollection` document — thay vì query collection để tìm media nào nằm trong collection nào, có thể đọc trực tiếp từ `media.inCollections`.

```
Media {
  _id: BigInt (Snowflake)
  type: 'movie' | 'tv'
  title: string (text index)
  originalTitle?: string
  slug: string (text index + _translations.vi.slug)
  overview: string
  poster?: MediaFile { name, mimeType, size, ... }
  backdrop?: MediaFile
  originalLang?: string (indexed)
  genres: [ref:Genre] (indexed)
  studios: [ref:Production]
  producers: [ref:Production]
  tags: [ref:MediaTag] (indexed)
  credits: [ref:Credit]
  runtime: number (phút)
  movie?: Movie (embedded, nếu type='movie')
  tv?: TVShow (embedded, nếu type='tv')
  videos: [MediaVideo] (trailer, clip)
  adult: boolean
  releaseDate: ShortDate (year+month+day indexed)
  status: string (processing status)
  inCollections: [ref:MediaCollection]
  externalIds: MediaExternalIds (tmdbId, imdbId, ...)
  scanner: MediaScannerData (scan metadata)
  views, dailyViews, weeklyViews, monthlyViews: number (indexed)
  ratingCount, ratingScore, ratingAverage: number (indexed)
  pStatus: number (public status)
  visibility: number
  addedBy: ref:User
  _translations: { [lang]: { title, slug, overview } }
  createdAt, updatedAt (timestamps + indexes)
  _original (TrackableDoc)
}
```

`Movie` (embedded, `_id: false`) chỉ xuất hiện trong document `Media` có `type: 'movie'`. Field `source` là tham chiếu đến `MediaStorage` document — file nguồn gốc trước khi encode. Mảng `tJobs` lưu BullMQ job ID đang chạy để có thể cancel. `TVShow` (embedded, `_id: false`) chỉ xuất hiện trong Media có `type: 'tv'`, chứa `episodeCount`, con trỏ đến `lastEpisode` (tập mới nhất đã phát), `pLastEpisode` (tập mới nhất đã publish công khai), và mảng `episodes` (tham chiếu tất cả tập phim).

## Collection TVEpisode: Tập Phim Series

`TVEpisode` là collection độc lập (không phải subdocument) vì mỗi tập phim có thể có file nguồn riêng (`source`), subtitle riêng, chapter riêng, và cần track trạng thái encode riêng. Quan hệ với `Media` là `media: ref:Media` — field này được index cùng với `epNumber` (index compound `{media: 1, epNumber: 1}`) để query "tất cả tập của media X sắp xếp theo số tập" hiệu quả.

Tương tự `Media`, `TVEpisode` cũng có `status` (processing), `pStatus` (public visibility), `visibility`, `views`, và `_translations` cho i18n. Cấu trúc này cho phép từng tập được quản lý phân quyền riêng — một tập preview có thể `visibility: PUBLIC` trong khi tập khác `visibility: MEMBER_ONLY`.

## Collection MediaStorage: Quản Lý File Vật Lý

`MediaStorage` là collection trung tâm cho file management — đây là điểm kết nối giữa `Media`/`TVEpisode` với cloud storage provider. Mỗi document `MediaStorage` đại diện cho một **file cụ thể trên cloud storage** — có thể là file nguồn gốc, một track video encoded, một track audio, hay file subtitle.

Field `type: number` (enum `MediaStorageType`) xác định vai trò của file: source video, stream video (encoded), stream audio, subtitle, hoặc thumbnail. Field `quality: number` (optional, chỉ có ở stream video) chứa resolution — 1080, 720, 480, v.v. Field `hdrFormat: number` chứa HDR format nếu có (HDR10, Dolby Vision, HLG). Field `streams` là mảng embedded `MediaStorageStream` — sau khi encode xong, từng track con (360p, 480p, 720p, 1080p) được lưu như một element trong mảng này, mỗi element chứa `type`, `quality`, `codec`, `channels`, `mimeType`, `size`.

`linkedStorage` là tham chiếu đến một `ExternalStorage` thứ hai — được dùng trong trường hợp file nguồn nằm trên storage A (ví dụ OneDrive cá nhân), nhưng encode output cần upload lên storage B (Cloudflare R2 production). Đây là "linked source storage" pattern cho phép tách storage upload từ user khỏi storage production.

```
MediaStorage {
  _id: BigInt (Snowflake)
  type: MediaStorageType (source, stream_video, stream_audio, subtitle, ...)
  name: string (tên file)
  path: string (đường dẫn trên storage)
  quality?: number (resolution, chỉ stream_video)
  hdrFormat?: number
  mimeType: string
  size: number (bytes)
  streams: [MediaStorageStream] (embedded, sau khi encode)
  options?: MediaSourceOptions
  media: ref:Media
  episode?: ref:TVEpisode
  storage: ref:ExternalStorage (storage chứa output)
  linkedStorage?: ref:ExternalStorage (storage chứa source)
}

MediaStorageStream (embedded) {
  _id: BigInt
  type: MediaStorageType
  name: string
  quality?: number
  codec?: number
  hdrFormat?: number
  channels?: number (audio)
  mimeType: string
  size: number
}
```

## Collection ExternalStorage: Quản Lý Cloud Provider

`ExternalStorage` lưu credentials và metadata của từng cloud storage provider được cấu hình. Field `kind: number` (enum `CloudStorage`) xác định loại provider: Azure Blob, Cloudflare R2, S3, OneDrive, Google Drive, Dropbox. Credentials (`clientId`, `clientSecret`, `accessToken`, `refreshToken`) được lưu encrypted bằng `CRYPTO_SECRET_KEY` — `StringCrypto` utility mã hóa/giải mã trong `ExternalStoragesService`. Field `expiry` lưu thời hạn hết hạn của `accessToken` (chỉ cần cho OAuth providers như OneDrive — `refreshToken` chỉ bắt buộc khi `kind === ONEDRIVE`).

`publicUrl` và `secondPublicUrl` là URL CDN public tương ứng với provider, được dùng khi generate URL phát video. `used: number` là dung lượng đã dùng (bytes), cập nhật sau mỗi upload/delete. Mảng `files` là backreference đến tất cả `MediaStorage` document đang dùng storage này.

## Collection Setting: Cấu Hình Hệ Thống

`Setting` là một document singleton (thực tế chỉ có một record) chứa toàn bộ cấu hình vận hành của hệ thống. `owner: ref:User` là người tạo cấu hình này. Sáu field `ExternalStorage` ref chỉ định storage mặc định cho từng loại file: `mediaPosterStorage` (ảnh bìa), `mediaBackdropStorage` (ảnh nền), `tvEpisodeStillStorage` (ảnh thumbnail tập phim), `mediaSourceStorages[]` (nơi lưu file nguồn), `mediaSubtitleStorages[]` (nơi lưu subtitle), `linkedMediaSourceStorages[]` (source storage của user linked).

`defaultVideoCodecs: number` là bitmask chỉ định codec nào được bật mặc định khi tạo encode job mới. Các field `audioParams`, `videoH264Params`, v.v. là string FFmpeg tham số override env var của Transcoder — đây là cơ chế cho phép admin thay đổi preset encode mà không cần restart Transcoder. `videoQualityList: number[]` và `videoNextGenQualityList: number[]` là danh sách resolution target (ví dụ `[1080, 720, 480, 360]`). `videoEncodingSettings: [EncodingSetting]` là mảng embedded per-quality override (ví dụ: ở 480p dùng CRF cao hơn để tiết kiệm dung lượng).

## Collection AuditLog và Notification: DATABASE_B

`AuditLog` sử dụng polymorphic reference với pattern `refPath`: field `target: BigInt` là ID của object được audit, và `targetRef: string` cho biết target thuộc collection nào (ví dụ `'Media'`, `'Role'`, `'User'`). Mongoose dùng `refPath: 'targetRef'` để populate đúng model tương ứng. Field `changes: [AuditLogChange]` lưu mảng thay đổi từng field (field name, old value, new value), được generate bởi `TrackableDoc` comparison trong service. Document `AuditLog` là append-only — chỉ insert, không bao giờ update hay delete trong quá trình hoạt động bình thường.

`Notification` cũng dùng polymorphic reference nhưng hiện chỉ hỗ trợ `targetRef: 'Media'`. Field `type: number` xác định loại thông báo (tập mới, cập nhật metadata, v.v.). Field `read: boolean = false` được update khi user đọc notification. Cả hai collection này trong DATABASE_B có thể bị truncate hoặc archive độc lập với DATABASE_A.

## Collection History, Rating, DriveSession, Counter

`History` lưu lịch sử xem của từng user. Ba field chính: `time: number` (vị trí dừng lại tính bằng giây, dùng cho resume playback), `paused: boolean` (đang dừng hay đang xem), và `watched: number` (số lần xem). Compound index `{user: 1, media: 1}` tăng tốc query "lịch sử xem của user X với media Y". Index `{date: 1}` hỗ trợ query "xem gần đây nhất".

`Rating` là quan hệ nhiều-nhiều giữa User và Media với compound unique index `{media: 1, user: 1}` — đảm bảo mỗi user chỉ có một rating cho mỗi media. `score: number` là điểm rating. Sau mỗi insert/update Rating, service cập nhật `media.ratingCount`, `media.ratingScore`, và `media.ratingAverage` bằng $inc và $set atomic.

`DriveSession` là session upload tạm thời — khi user bắt đầu upload file nguồn, một DriveSession document được tạo với `expiry: Date`. Transcoder job data truyền `driveSessionId`; nếu session hết hạn trước khi encode bắt đầu, job sẽ bị reject. Sau khi encode hoàn tất, DriveSession bị xóa.

`Counter` là collection đơn giản dùng cho auto-increment sequence (nếu cần ID tuần tự như số tập phim). Schema có `_id: string` (tên sequence) và `seq: number`, được update bằng `findOneAndUpdate({ _id: name }, { $inc: { seq: 1 } }, { upsert: true, new: true })`.

## Tổng Hợp Index Quan Trọng

Thiết kế index trong DaPlex tập trung vào ba nhóm query chính: tìm kiếm nội dung, sắp xếp theo popularity, và lookup theo quan hệ.

| Collection | Index | Mục Đích |
|---|---|---|
| `Media` | `{ slug: 'text', '_translations.vi.slug': 'text' }` | Full-text search theo slug |
| `Media` | `{ genres: 1 }`, `{ tags: 1 }` | Filter theo thể loại, tag |
| `Media` | `{ views: 1 }`, `{ dailyViews: 1 }`, `{ weeklyViews: 1 }`, `{ monthlyViews: 1 }` | Sort trending, most-viewed |
| `Media` | `{ ratingAverage: 1 }` | Sort top-rated |
| `Media` | `{ 'releaseDate.year': 1, 'releaseDate.month': 1 }` | Filter theo năm/tháng |
| `Media` | `{ createdAt: 1 }`, `{ updatedAt: 1 }` | Sort mới nhất, cập nhật gần đây |
| `TVEpisode` | `{ media: 1, epNumber: 1 }` | Lấy tập phim theo số thứ tự |
| `Role` | `{ name: 1 }`, `{ position: 1 }` | Sort role theo vị trí |
| `History` | `{ user: 1 }`, `{ media: 1 }`, `{ date: 1 }` | Lịch sử xem |
| `Rating` | `{ media: 1, user: 1 }` (unique) | Prevent duplicate rating |
| `Genre` | `{ '_translations.vi.name': 1 }` (unique, sparse) | Unique translated genre name |
| `User` | `{ username: 1 }`, `{ email: 1 }` (unique) | Login lookup |

*Hình 3.7: Quan hệ giữa các collection trong DATABASE_A và DATABASE_B — Media là entity trung tâm kết nối hầu hết các domain khác*

Hình 3.7 cho thấy `Media` là entity trung tâm trong cơ sở dữ liệu DaPlex. Hầu hết các domain khác đều kết nối về `Media`: `History` và `Rating` theo dõi tương tác của User với Media; `Genre`, `Production`, `MediaTag` là metadata phân loại Media; `MediaStorage` lưu file vật lý của Media; `TVEpisode` là phần mở rộng của Media khi type là series. Thiết kế này phản ánh thực tế nghiệp vụ: nội dung video là lý do tồn tại của toàn bộ hệ thống, và mọi tính năng đều xoay quanh việc quản lý và phân phối nội dung đó.
