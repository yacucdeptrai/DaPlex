# 4.3.2. Module Xác Thực và Phân Quyền

`AuthModule` là nền tảng bảo mật của DaPlex-API, cung cấp toàn bộ vòng đời xác thực người dùng và cơ chế phân quyền dựa trên bitmask. Thiết kế xác thực ảnh hưởng trực tiếp đến bảo mật và trải nghiệm người dùng trên toàn hệ thống, từ luồng đăng nhập đến kiểm soát quyền truy cập từng endpoint. Mục này đi qua JWT/Redis token model, cơ chế refresh rotation, AuthGuard với cache, RBAC bitmask, rate limiting hai chế độ, và flow xác nhận email.

---

## Tổng Quan AuthModule

`AuthModule` (`src/resources/auth/`) là module trung tâm của toàn hệ thống bảo mật DaPlex-API, cung cấp đầy đủ vòng đời xác thực người dùng: đăng ký, đăng nhập, refresh token, thu hồi phiên, xác nhận email, và khôi phục mật khẩu. Module này được export rộng rãi — `AuthService`, `PermissionsModule`, và `Redis2ndCacheModule` được export để các module khác như `MediaModule`, `UsersModule` có thể sử dụng `AuthGuard` và kiểm tra permission mà không cần import lại từ đầu.

Thiết kế xác thực của DaPlex dựa trên mô hình **JWT access token kết hợp opaque refresh token trong Redis**. Access token là JWT ngắn hạn (mặc định 900 giây = 15 phút) chứa payload `{ _id: string }` — chỉ lưu User ID, không lưu roles hay permissions để tránh stale cache khi admin thay đổi quyền. Refresh token **không phải JWT** mà là một chuỗi ngẫu nhiên 32 ký tự sinh bởi `nanoid(32)` — quyết định này được ghi chú trong comment code `DEPRECATED, SWITCHED FROM JWT TO NANOID`.

> Lý do chuyển sang nanoid: refresh token JWT có thể bị decode để lấy thông tin user ID, trong khi opaque token là string vô nghĩa không tiết lộ thông tin — mọi validation đều phải qua Redis lookup, không thể bypass bằng cách tự ký token giả.

## Luồng Đăng Nhập và Tạo JWT Token

`AuthService.authenticate()` thực hiện đăng nhập theo bốn bước: (1) query MongoDB tìm user theo email, populate `roles` với các field cần thiết (`_id`, `name`, `color`, `permissions`, `position`) và sort theo `position`; (2) nếu không tìm thấy user, throw `HttpException` với `StatusCode.EMAIL_NOT_EXIST`; (3) `bcrypt.compare(password, user.password)` so sánh mật khẩu nhập vào với hash trong database — nếu sai throw `StatusCode.INCORRECT_PASSWORD`; (4) gọi `createJwtToken(user)` để tạo cặp token.

`createJwtToken()` tạo hai token song song bằng `Promise.all()`: sign JWT access token với `jwtService.signAsync({ _id: user._id.toString() }, { secret, expiresIn })` và generate refresh token bằng `nanoid(32)`. Refresh token được lưu vào Redis với key `refresh_token:{token}` → value `{ _id: user._id.toString() }` và TTL bằng `REFRESH_TOKEN_EXPIRY` (mặc định 2592000 giây = 30 ngày). Hàm trả về instance `Jwt` entity chứa cả hai token và `UserDetails` payload.

`AuthController.signIn()` nhận `Jwt` từ service rồi thực hiện thêm: `response.setCookie('refresh_token', jwt.refreshToken, { maxAge: jwt.refreshTokenExpiry })` để ghi refresh token vào **HttpOnly cookie** (JavaScript không thể đọc, bảo vệ khỏi XSS), và `response.setCookie('authenticated', 'true', { httpOnly: false, maxAge })` để ghi một cookie không HttpOnly — Angular đọc cookie này khi khởi động để biết người dùng đã đăng nhập và trigger refresh token tự động. Response JSON trả về chỉ chứa `accessToken` và `payload` (UserDetails) — `refreshToken` và `refreshTokenExpiry` bị exclude bởi `@Exclude({ toPlainOnly: true })` trong `Jwt` entity.

## Cơ Chế Refresh Token Rotation

`AuthController.refreshToken()` đọc cookie `refresh_token` từ Fastify request, truyền vào `AuthService.refreshToken()`. Service tìm refresh token trong Redis theo key `refresh_token:{token}` — nếu không tìm thấy (đã expire hoặc bị revoke), throw `StatusCode.TOKEN_REVOKED`. Nếu tìm thấy, service **ngay lập tức set TTL của key còn lại xuống 60 giây** (`redisCacheService.set(key, payload, 60)`) trước khi tạo token mới. Kỹ thuật này xử lý race condition: nếu client gửi hai request refresh đồng thời (ví dụ tab mobile và tab desktop), request đầu tiên thành công và set TTL còn 60s, request thứ hai vẫn tìm được key (trong 60s) và cũng thành công nhận token mới — tránh trường hợp một refresh invalidate session của tab kia. Sau 60 giây, key cũ expire tự nhiên.

Sau khi verify token, service load user từ MongoDB (`findById` với populate roles), tạo cặp token mới, và xóa refresh token cũ không còn cần thiết (vì token mới đã được ghi vào Redis). Controller set lại cả hai cookie với token mới và trả về response. Mô hình **refresh token rotation** này đảm bảo mỗi refresh token chỉ dùng một lần — nếu attacker đánh cắp refresh token và dùng trước user, user sẽ không thể refresh và biết session bị xâm phạm.

## AuthGuard — Xác Thực Và Cache Người Dùng

`AuthGuard` trong `src/resources/auth/guards/auth.guard.ts` là NestJS guard áp dụng trên các route yêu cầu xác thực. `canActivate()` đọc header `Authorization` từ request Fastify. Nếu header rỗng: kiểm tra `authGuardOptions.anonymous` từ `@AuthGuardOptions()` decorator — nếu route cho phép anonymous, set `request.user = { isAnonymous: true }` và return true (cho phép truy cập với user anonymous); nếu không, throw `StatusCode.NULL_AUTHORIZATION`. Cơ chế anonymous mode quan trọng cho các route như "xem danh sách media" — user chưa đăng nhập vẫn xem được, nhưng service biết là anonymous để bỏ qua check watch history hay visibility restriction cho member.

Sau khi verify JWT thành công (`jwtService.verifyAsync()` throw nếu token hết hạn hoặc sai signature), guard gọi `findUserAuthGuard(BigInt(payload._id))`. Phương thức này dùng `redis2ndCacheService.wrap()` với TTL 300 giây — nếu key `user_auth_guard:{id}` đã có trong Redis, trả về ngay; nếu không, chạy callback: query MongoDB với `findOneAndUpdate({ lastActiveAt: new Date() })` (update thời gian active cuối), select các field cần thiết, populate roles, rồi `plainToInstance(AuthUserDto)` và `scanPermission()`. Kết quả được cache 5 phút. Ưu điểm: mỗi request không cần round-trip MongoDB; nhược điểm: thay đổi role/ban user có độ trễ tối đa 5 phút mới có hiệu lực.

> Trade-off này chấp nhận được với VoD platform — không như hệ thống tài chính cần revoke permission tức thì.

Sau khi nhận `AuthUserDto` từ cache, guard kiểm tra `user.banned` — nếu bị ban throw `StatusCode.USER_BANNED` (403 Forbidden). Set `request.user = user` để controller và các downstream decorator có thể access qua `@AuthUser() user: AuthUserDto`. Nếu có exception không phải `HttpException` (ví dụ JWT library error), guard catch và throw `StatusCode.UNAUTHORIZED` generic để không leak thông tin kỹ thuật ra ngoài.

## Hệ Thống Phân Quyền RBAC — Permission Bitmask

`PermissionsService` implement Role-Based Access Control dùng bitmask integer. Mỗi role trong database có field `permissions: number` — một 32-bit integer trong đó mỗi bit đại diện cho một quyền:

| Permission | Bit | Giá trị thập phân |
|---|---|---|
| `MANAGE_MEDIA` | 0 | 1 |
| `MANAGE_USERS` | 1 | 2 |
| `ADMINISTRATOR` | 2 | 4 |
| *(các permission khác)* | ... | ... |

Một role có thể gán nhiều permission bằng bitwise OR: role `Content Manager` với `permissions = 3` (`0b011`) có cả `MANAGE_MEDIA` và `MANAGE_USERS`. Kiểm tra permission dùng bitwise AND: `role.permissions & UserPermission.MANAGE_MEDIA` trả về non-zero nếu bit đó được set. `PermissionsService.scanPermission()` enumerate toàn bộ `UserPermission` enum keys, check từng permission qua mọi role của user, và trả về `granted: number[]` — mảng các permission value được phép. Mảng này được cache trong `AuthUserDto.granted` để các guard và service kiểm tra mà không phải duyệt roles lại.

`PermissionsService.hasPermission(user, permissions[])` là helper nhanh: return true nếu user có bất kỳ permission nào trong mảng đầu vào (OR logic), hoặc nếu user có role `ADMINISTRATOR` (bypass mọi kiểm tra). `canEditRole()` và `canEditPermissions()` implement privilege escalation protection: user chỉ có thể edit role có `position` cao hơn (số nhỏ hơn = quyền cao hơn) và chỉ có thể grant permission mà chính mình đang có — không thể tự nâng quyền bằng cách edit role của mình.

## RateLimitInterceptor — Hai Chế Độ Đếm

`RateLimitInterceptor` là NestJS interceptor (không phải guard) để linh hoạt hơn trong việc quyết định khi nào increment counter. Được cấu hình qua `@RateLimitOptions()` với bốn tham số:

**`catchMode: 'success'`** (dùng cho sign-up, gửi email xác nhận, gửi email reset): counter chỉ tăng khi request **thành công**. Pipeline: request đến → check counter < limit → xử lý → nếu response OK: `redis.set(key, count+1, ttl)`. Ứng dụng: gửi email xác nhận tối đa 1 lần mỗi 2 phút — nếu gửi thất bại (server email down), counter không tăng và user có thể thử lại ngay.

**`catchMode: 'error'`** (dùng cho sign-in): counter chỉ tăng khi có lỗi **4xx** (không phải 429). Pipeline: request đến → check counter → nếu đăng nhập sai: `redis.set(key, count+1, ttl)` → nếu đăng nhập đúng: `redis.del(key)` (reset counter). Ứng dụng: chỉ đếm lần nhập sai mật khẩu, đăng nhập đúng xóa counter. Sau 5 lần sai trong 5 phút, response trả về `{ continueWithCaptcha: true }` — frontend hiển thị reCAPTCHA; nếu body chứa field `captcha` hợp lệ, cho phép tiếp tục dù đã vượt limit.

Key Redis cho rate limit là `rate_limit:{id}:{url}:{method}:{catchMode}` — trong đó `id` là user ID nếu đã đăng nhập, hoặc IP address nếu chưa đăng nhập. Dùng IP cho sign-in không đăng nhập là đúng logic: attacker brute force từ một IP sẽ bị block, user hợp lệ từ IP khác không bị ảnh hưởng. `retryAfterSeconds` lấy từ `redis.ttl(key)` và trả về trong response body để frontend hiển thị countdown.

## Flow Xác Nhận Email và Khôi Phục Mật Khẩu

Cả hai flow dùng pattern tương tự: tạo mã ngẫu nhiên (`nanoid(8)` — 8 ký tự), lưu vào user document trong MongoDB, gửi email qua SendGrid với URL có tham số `id` và `code`. User click link, frontend gửi `POST /api/auth/confirm-email` với `{ id, activationCode }` — service dùng `findOneAndUpdate({ _id: id, activationCode }, { $set: { verified: true }, $unset: { activationCode: 1 } })` để atomic verify và xóa code, nếu không match throw `StatusCode.INVALID_CODE`.

Đặc điểm quan trọng: `sendConfirmEmail` và `passwordRecovery` đều bị rate limit `{ catchMode: 'success', ttl: 120, limit: 1 }` — tối đa 1 email mỗi 2 phút, counter tăng khi thành công. Controller `sendConfirmationEmail` yêu cầu `@UseGuards(AuthGuard)` — chỉ user đã đăng nhập mới gửi lại email xác nhận (tránh spam email người khác), còn `passwordRecovery` không yêu cầu đăng nhập (user quên mật khẩu không đăng nhập được). Email template được quản lý trong SendGrid dashboard, code chỉ truyền data (`recipient_name`, `button_url`) vào template ID qua `SendgridTemplate` enum.
