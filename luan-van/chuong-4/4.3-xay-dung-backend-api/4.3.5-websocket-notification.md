# 4.3.5. Hệ Thống Thông Báo Real-Time

DaPlex-API tích hợp thông báo real-time qua WebSocket để admin dashboard luôn đồng bộ trạng thái mà không cần polling. Kiến trúc này quan trọng vì quy trình encode video có thể kéo dài hàng chục phút — admin cần biết ngay khi hoàn thành hoặc thất bại bất kể đang ở trang nào. Mục này trình bày `WsAdminGateway`, cơ chế xác thực kết nối và phân phòng, hai enum trung tâm, pattern self-exclusion qua `X-Socket-ID`, thông báo cá nhân kết quả encode, và trạng thái hiện tại của `NotificationModule`.

---

## Tổng Quan WsAdminGateway

Thông báo real-time trong DaPlex-API được thực hiện qua `WsAdminGateway` (`src/resources/ws-admin/ws-admin.gateway.ts`) sử dụng Socket.IO namespace `ws/admin`. Gateway được tích hợp vào `AppSocketModule` — một module riêng biệt chỉ import `WsAdminModule`, được `AppModule` import song song với `AppRoutingModule` — cho phép WebSocket server chạy trên cùng cổng Fastify (port 3000) mà không cần port riêng. Khác với nhiều hệ thống WebSocket tách biệt thành microservice, kiến trúc DaPlex gắn gateway ngay trong API process để gateway có thể trực tiếp inject và sử dụng `AuthService`, tránh round-trip xác thực qua network.

`@WebSocketGateway({ namespace: 'ws/admin', cors: { origin: '*' }, transports: ['websocket'] })` cấu hình ba điểm quan trọng:

| Tham số | Mô tả |
|---|---|
| `namespace: 'ws/admin'` | Tách biệt không gian với các namespace khác nếu có trong tương lai |
| `cors: { origin: '*' }` | Cho phép cross-origin connection từ Angular frontend (cần thiết vì frontend có thể chạy trên domain khác trong production) |
| `transports: ['websocket']` | Chỉ chấp nhận WebSocket transport thuần, loại bỏ HTTP long-polling fallback của Socket.IO — quyết định này giảm overhead và làm đơn giản hóa reverse proxy configuration (Nginx chỉ cần forward WebSocket upgrade header, không cần xử lý polling request) |

---

## Xác Thực Kết Nối và Phân Phòng

`WsAdminGateway` implement `OnGatewayConnection` — NestJS lifecycle hook gọi `handleConnection(client: Socket)` ngay khi socket kết nối, trước khi bất kỳ message event nào được xử lý. Method này đọc `client.handshake.auth['token']` — Angular client gửi access token qua Socket.IO `auth` option khi khởi tạo connection (`socket = io('/ws/admin', { auth: { token: accessToken } })`). Service gọi `authService.verifyAccessToken(accessToken)` để verify JWT; nếu token hợp lệ, client được join vào room `user:id:{user._id}` — room riêng theo user ID để gửi thông báo cá nhân hóa. Nếu token không hợp lệ hoặc hết hạn, `client.disconnect(true)` ngắt kết nối ngay lập tức, không emit lỗi để tránh tiết lộ thông tin. Toàn bộ logic xác thực nằm trong lifecycle hook, không phải guard — vì guard chỉ apply cho HTTP handler, còn WebSocket connection lifecycle cần implement interface riêng.

Sau khi kết nối thành công và join room cá nhân, Angular admin client tự chủ động join các room admin cần thiết thông qua event `room:join`. `handleRoomJoin(@ConnectedSocket() client, @MessageBody() payload: string)` đơn giản gọi `client.join(payload)` và return `true` (acknowledgement). Tương tự, `room:leave` gọi `client.leave(payload)`. Thiết kế "client-driven room joining" cho phép frontend quyết định đăng ký nhận notification nào theo ngữ cảnh — ví dụ khi admin đang xem trang danh sách media thì join `admin:media:list`, khi mở chi tiết media `id=123` thì join `admin:media:details:123`. Khi điều hướng đi, Angular gọi `room:leave` để unsubscribe, tránh nhận notification dư thừa và giữ số lượng client trong mỗi room đúng với số người thực sự đang xem.

## SocketRoom và SocketMessage — Hai Enum Trung Tâm

Hệ thống room-based notification của DaPlex được hình thức hóa qua hai enum tập trung trong `src/enums/`. `SocketRoom` định nghĩa tất cả room pattern: `USER_ID = 'user:id'` (prefix cho room cá nhân, suffix là `_id` của user), và một tập các admin room: `ADMIN_MEDIA_LIST = 'admin:media:list'` (ai đang xem danh sách media), `ADMIN_MEDIA_DETAILS = 'admin:media:details'` (prefix cho room chi tiết một media cụ thể), `ADMIN_EPISODE_DETAILS = 'admin:episode:details'` (prefix cho room chi tiết tập phim), cùng với room tương tự cho collection, genre, production, tag, chapter-type. Admin room có hai loại: room danh sách (broadcast khi có thay đổi ảnh hưởng đến list view) và room chi tiết (broadcast khi thay đổi một entity cụ thể, suffix là Snowflake ID).

`SocketMessage` enum liệt kê 20 message type khác nhau, phản ánh toàn bộ các sự kiện có thể xảy ra trong quá trình quản lý nội dung: các `REFRESH_*` event (media, TV episode, videos, subtitles, chapters, collections, genres, productions, tags, chapter-types) báo hiệu frontend fetch lại data từ REST API; các `SAVE_*/DELETE_*_SOURCE` event báo trạng thái upload nguồn; `ADD_MOVIE_STREAM/ADD_TV_STREAM` báo stream mới sẵn sàng; `MEDIA_PROCESSING_SUCCESS/FAILURE` báo kết quả encode cho user đã upload. Triết lý thiết kế là **event-driven refetch**: thay vì server push toàn bộ data object qua WebSocket (tốn payload), server chỉ emit một event kèm `{ mediaId }` báo hiệu frontend rằng entity đó đã thay đổi, Angular component lắng nghe event rồi tự gọi REST API để lấy data mới. Cách này đơn giản hóa đồng bộ state — frontend không cần merge partial update mà chỉ refetch toàn bộ entity.

## Cơ Chế Self-Exclusion — X-Socket-ID Header

Một pattern quan trọng xuất hiện xuyên suốt `MediaService` là cơ chế loại trừ chính mình khi broadcast: thay vì luôn emit từ `this.wsAdminGateway.server` (broadcast tới tất cả client trong room bao gồm cả người thực hiện thao tác), service kiểm tra `headers.socketId`:

```typescript
const ioEmitter = (headers.socketId && this.wsAdminGateway.server.sockets.get(headers.socketId))
  || this.wsAdminGateway.server;
ioEmitter.to(SocketRoom.ADMIN_MEDIA_LIST).emit(SocketMessage.REFRESH_MEDIA);
```

`headers.socketId` được extract từ `X-Socket-ID` HTTP header — Angular `BaseUrlInterceptor` tự động thêm header này vào mọi API request nếu user đang có WebSocket connection. Nếu socket ID hợp lệ và `server.sockets.get(headers.socketId)` trả về socket object của người thực hiện thao tác, emit từ socket đó thay vì từ server. Trong Socket.IO, khi emit từ một socket, mặc định socket đó không nhận event — hành vi này tạo ra self-exclusion tự nhiên. Admin vừa thêm genre sẽ không nhận `REFRESH_GENRES` event vì UI của họ đã được cập nhật ngay khi API trả về response; chỉ các tab admin khác đang mở cùng trang mới nhận event và fetch lại data.

Pattern này đặc biệt quan trọng cho trải nghiệm người dùng: nếu admin A và admin B đều đang xem danh sách media, khi A sửa một media, B nhận notification và tự động refresh list, còn A không bị interrupt vì UI của A đã correct sau khi PATCH request thành công.

## Thông Báo Kết Quả Encode Cho User

Hai socket message đặc biệt hướng đến user cá nhân thay vì admin room là `MEDIA_PROCESSING_SUCCESS` và `MEDIA_PROCESSING_FAILURE`. Khi `MediaResultConsumer` xử lý job `finished-encoding`, sau khi cập nhật `pStatus = DONE` trong MongoDB, `handleMovieStreamQueueDone()` kiểm tra `mediaQueueResultDto.isPrimary` — flag này đánh dấu đây là codec primary (H264, codec đầu tiên hoàn thành, có nghĩa media đã có thể phát). Nếu là primary codec, service emit hai event: đến `user:id:{mediaQueueResultDto._id}` (Snowflake ID của admin đã upload) qua `MEDIA_PROCESSING_SUCCESS`, và đến admin rooms qua `REFRESH_MEDIA` để cập nhật status display. Tương tự, `handleMovieStreamQueueError()` emit `MEDIA_PROCESSING_FAILURE` đến room cá nhân của user.

Mô hình này cho phép admin đóng trang upload và tiếp tục làm việc — khi encode xong, họ nhận thông báo popup ngay trên bất kỳ trang admin nào đang mở. Angular `WsService` lắng nghe `MEDIA_PROCESSING_SUCCESS` event trên room `user:id:{userId}` và trigger notification tương ứng. Room `user:id` được join ngay khi WebSocket kết nối thành công (`handleConnection` join room này cho mọi authenticated user), nên không cần admin chủ động join — thông báo cá nhân luôn hoạt động độc lập với navigation state.

## NotificationModule — Trạng Thái Phát Triển

`NotificationModule` (`src/resources/notification/`) tồn tại trong codebase với `Notification` schema trong `DATABASE_B` (database phụ), nhưng `NotificationService` hiện chỉ ở dạng scaffold — các method `create/findAll/findOne/update/remove` trả về placeholder string thay vì logic thực sự. Hệ thống thông báo hiện tại của DaPlex hoạt động hoàn toàn qua WebSocket real-time (các `SocketMessage` event mô tả ở trên) mà không có persistent notification storage.

> Đây là trade-off có chủ đích trong giai đoạn phát triển hiện tại: real-time WebSocket đáp ứng nhu cầu cốt lõi của admin dashboard (biết ngay khi encode xong, biết ngay khi co-admin sửa nội dung), còn in-app notification inbox (thông báo lưu trữ, đánh dấu đã đọc) là tính năng dự kiến cho phiên bản sau khi có thêm yêu cầu từ người dùng.
