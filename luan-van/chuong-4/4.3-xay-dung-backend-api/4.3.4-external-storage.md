# 4.3.4. Tích Hợp External Storage

DaPlex-API thiết kế tầng lưu trữ theo mô hình multi-provider để không gắn cứng vào bất kỳ cloud vendor nào và cho phép linh hoạt phân bổ workload giữa nhiều provider với chi phí tối ưu. Đây là điểm khác biệt quan trọng cho nền tảng tự triển khai — người dùng có thể kết hợp nhiều storage miễn phí hoặc chi phí thấp thay vì phụ thuộc một nhà cung cấp duy nhất. Mục này trình bày schema bảo mật credentials, cơ chế upload session, FilerService cho local storage, và quản lý vòng đời file trong từng storage.

---

## Tổng Quan Kiến Trúc Multi-Provider Storage

DaPlex-API thiết kế tầng lưu trữ theo mô hình **multi-provider**: thay vì gắn cứng vào một cloud storage duy nhất, hệ thống hỗ trợ đồng thời nhiều provider khác nhau thông qua một abstraction layer, cho phép admin đăng ký nhiều tài khoản storage và linh hoạt phân bổ workload giữa chúng. Thiết kế này đặc biệt phù hợp với VoD platform tự triển khai — người dùng có thể tận dụng dung lượng miễn phí hoặc chi phí thấp từ nhiều nhà cung cấp (OneDrive 5GB miễn phí, Google Drive 15GB miễn phí, Backblaze B2 10GB miễn phí, Cloudflare R2 10GB miễn phí), hoặc chạy server lưu trữ riêng qua DaPlex-Filer.

`ExternalStoragesModule` (`src/resources/external-storages/`) quản lý các cấu hình storage này trong MongoDB thông qua collection `externalstorage`. `ExternalStoragesService` cung cấp toàn bộ các thao tác CRUD và các utility method mà các service khác (đặc biệt là `MediaService`, `OnedriveService`, `S3Service`, `FilerService`) gọi để tra cứu cấu hình trước khi thực hiện thao tác với storage. Các module provider cụ thể nằm trong `src/common/modules/`: `CloudflareR2Module`, `S3Module`, `OnedriveModule`, `FilerModule`, `GoogleDriveModule`, `DropboxModule`, `AzureBlobModule`, `ImageKitModule`, `ImgurModule` — mỗi module implement logic giao tiếp riêng với API của từng provider.

## Schema ExternalStorage — Thiết Kế Bảo Mật Credentials

`ExternalStorage` schema (`src/schemas/external-storage.schema.ts`) lưu trữ toàn bộ thông tin cấu hình một storage instance. Field `kind` là `CloudStorage` enum phân biệt loại provider:

| Giá trị | Provider |
|---|---|
| `CLOUDINARY = 1` | Cloudinary |
| `IMAGEKIT = 2` | ImageKit |
| `GOOGLE_DRIVE = 3` | Google Drive |
| `IMGUR = 4` | Imgur |
| `DROPBOX = 5` | Dropbox |
| `ONEDRIVE = 6` | OneDrive |
| `S3 = 7` | S3-compatible (AWS S3, Backblaze B2, MinIO...) |
| `FILER = 8` | DaPlex-Filer (local file server) |

Hai field `clientId` và `clientSecret` lưu credentials của OAuth2 app hoặc API key tương ứng với từng provider.

Điểm đặc biệt quan trọng về bảo mật: `clientSecret` **không bao giờ lưu dưới dạng plaintext**. Trong `ExternalStoragesService.create()`, trước khi lưu vào MongoDB, service khởi tạo `new StringCrypto(configService.get('CRYPTO_SECRET_KEY'))` và gọi `stringCrypto.encrypt(addStorageDto.clientSecret)` để mã hóa AES. Tương tự với `update()`. Khi các provider service cần dùng secret, họ gọi `externalStoragesService.decryptToken(storage)` — method này decrypt `clientSecret` in-place và set `storage._decrypted = true` để tránh decrypt lại lần thứ hai trong cùng một request lifecycle. Mô hình "encrypt at rest, decrypt on use" này bảo vệ credentials ngay cả khi database bị compromise — attacker không thể sử dụng credentials mà không có `CRYPTO_SECRET_KEY`.

| Field | Mô tả |
|---|---|
| `accessToken` / `refreshToken` / `expiry` | OAuth2 token **runtime** sau khi user hoàn tất authorization flow — thay đổi mỗi khi token được refresh, khác với `clientId/clientSecret` là credentials tĩnh |
| `folderId` / `folderName` | Xác định thư mục đích trong cloud storage |
| `publicUrl` / `secondPublicUrl` | Domain CDN dùng để xây dựng URL stream cho client |
| `inStorage` | `MediaStorageType` enum đánh dấu role của storage trong hệ thống — ví dụ `SOURCE` đánh dấu storage dùng để upload file video nguồn |
| `files[]` | Mảng reference đến các `MediaStorage` document đang nằm trong storage này |
| `used` | Tổng bytes đã dùng — cho phép theo dõi usage và hiển thị trên admin dashboard |

## Cơ Chế Upload Session — OneDrive và S3

Hai provider được sử dụng nhiều nhất cho video upload là OneDrive và S3-compatible. Cả hai đều hỗ trợ **resumable upload** — đặc biệt quan trọng cho file video lớn có thể vài chục GB.

`OnedriveService.createUploadSession()` giao tiếp với Microsoft Graph API để tạo upload session: service tra cứu storage đang được chỉ định là `SOURCE` qua `settingsService.findMediaSourceStorage()`, kiểm tra token còn hạn không (`storage.expiry < new Date()`), nếu hết hạn tự động gọi `refreshToken()` trước. Sau đó POST đến `graph.microsoft.com/v1.0/{urlPath}:/{encodedFilePath}:/createUploadSession` để nhận `uploadUrl` — URL tạm thời do Microsoft cấp phát cho phép client upload trực tiếp bằng HTTP PUT mà không cần đi qua API server DaPlex. Session trả về `{ url, storage: storage._id }` — sau khi client upload xong, gọi `saveMovieSource()` với `sessionId` để kích hoạt encode. Logic token refresh trong `OnedriveService` implement retry: nếu nhận 401 Unauthorized từ Graph API lần đầu, tự refresh token rồi retry lần hai trước khi throw lỗi — xử lý trường hợp token expire trong khoảng thời gian ngắn giữa khi check expiry và khi gọi API.

`S3Service` làm việc với bất kỳ provider nào implement S3-compatible API: AWS S3, Backblaze B2, MinIO, Wasabi, IDrive e2. Service tự implement AWS Signature Version 4 (HMAC-SHA256) vì không dùng AWS SDK — tất cả request đều là raw HTTP call qua `HttpService`. Method `findInStorages()` nhận danh sách storage và path, lần lượt thử `headObject()` trên từng storage với retry logic — dùng khi cần tìm file encode kết quả trên một trong nhiều S3 storage đã cấu hình.

`CloudflareR2Service` khác biệt với `S3Service` ở chỗ credentials lấy từ **environment variable** (`CLOUDFLARE_R2_API_URL`, `CLOUDFLARE_R2_S3_KEY`, `CLOUDFLARE_R2_S3_SECRET`) thay vì từ `ExternalStorage` document. Đây là provider "hệ thống" — dùng cho poster, backdrop, thumbnail (ảnh nhỏ, upload từ API server) — trong khi `S3Service` được dùng để Transcoder upload video segment (video lớn, upload từ server Transcoder riêng). Cloudflare R2 cũng implement S3v4 signing nhưng cache chữ ký theo ngày (`cachedSignatureKey`, `cachedSignatureDate`) để không tính lại HMAC mỗi request.

## FilerService — Tích Hợp Local File Server

`FilerService` là provider đặc biệt kết nối với **DaPlex-Filer** — một service lưu trữ local chạy trên cùng server hoặc mạng nội bộ. Không giống các cloud provider dùng OAuth2, Filer xác thực bằng JWT: `FilerService.refreshToken()` gọi `jwtService.signAsync({}, { secret: storage.clientSecret, expiresIn: 4 * 60 * 60 })` — ký JWT với `clientSecret` là shared secret giữa API và Filer server, hạn 4 giờ. Token được làm mới sớm 5 phút trước khi hết hạn (`expiresIn - 300`) để tránh token expire trong giữa một batch upload. Sau khi ký, `expiry` được ghi ngược về `ExternalStorage` document và vào cache của `SettingsService` để không cần query MongoDB lại trong vòng TTL.

> Mô hình JWT-based authentication cho Filer cho phép deploy self-hosted mà không cần tích hợp OAuth2 provider bên ngoài — chỉ cần cấu hình shared secret. Đây là lựa chọn thực tế cho người dùng muốn giữ toàn bộ dữ liệu on-premise: video upload lên Filer (disk nội bộ), encode bởi Transcoder trên cùng máy chủ, stream qua CDN riêng.

## Quản Lý Storage Usage và Vòng Đời File

`ExternalStoragesService` cung cấp các method utility cho `MediaService` để theo dõi vòng đời file trong storage:

`addFileToStorage(storageId, fileId, fileSize, session)` gọi `$push: { files: fileId }` và `$inc: { used: fileSize }` trong cùng một MongoDB atomic update — thêm reference đến file mới và tăng bộ đếm bytes đã dùng. `deleteFileFromStorage(storageId, fileId, fileSize, session)` là thao tác đối xứng dùng `$pull` và `$inc: { used: -fileSize }`. Cả hai đều nhận `ClientSession` để tham gia vào MongoDB transaction lớn hơn (ví dụ khi xóa media, cần xóa file và update storage usage trong cùng transaction với xóa media document).

Cơ chế bảo vệ xóa storage: `remove()` từ chối xóa nếu `storage.files?.length` không rỗng — admin phải xóa tất cả media liên kết với storage trước. Điều này ngăn chặn tình trạng "dangling reference" khi storage config bị xóa nhưng MediaStorage documents vẫn còn reference đến storage ID đó. `addSettingStorage/deleteSettingStorage` quản lý field `inStorage` — khi admin chỉ định một storage làm nguồn upload chính, `addSettingStorage()` ghi `MediaStorageType.SOURCE` vào document đó; khi thay đổi, `deleteSettingStorage()` xóa field này để tránh conflit.

## Bảo Mật API — Owner-Only Access

Toàn bộ endpoint của `ExternalStoragesController` yêu cầu `@RolesGuardOptions({ requireOwner: true })` thay vì `UserPermission.MANAGE_MEDIA` thông thường. `requireOwner` là một cấp quyền cao hơn `ADMINISTRATOR` — chỉ có account chủ sở hữu của instance DaPlex mới có thể đọc, tạo, sửa, hoặc xóa storage configuration.

> Quyết định này có lý do rõ ràng về bảo mật: storage configuration chứa credentials của cloud storage — nếu admin thường có thể đọc, họ sẽ thấy thông tin tài khoản OneDrive, S3 key/secret của chủ hệ thống. Tách biệt `owner` khỏi `admin` cho phép chia sẻ quyền quản lý nội dung media (`MANAGE_MEDIA`) mà không lộ thông tin cơ sở hạ tầng.
