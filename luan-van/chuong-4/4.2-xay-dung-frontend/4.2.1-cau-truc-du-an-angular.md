# 4.2.1. Cấu Trúc Dự Án Angular

Mục này mô tả cách tổ chức mã nguồn của dự án `daplex-dune-v2`, từ cấu trúc thư mục cấp cao nhất đến phân tầng chi tiết bên trong `src/app/`. Việc hiểu rõ cấu trúc dự án là nền tảng để nắm bắt luồng dữ liệu, phạm vi dependency, và quy ước đặt tên được áp dụng nhất quán trong toàn bộ codebase. Mục này bao gồm: tổ chức thư mục, cấu hình TypeScript/Angular, phân tích chi tiết `core/` và `shared/`, cấu hình Service Worker, và hằng số toàn cục.

---

## Tổ Chức Thư Mục Cấp Cao Nhất

Dự án `daplex-dune-v2` tuân theo cấu trúc chuẩn Angular CLI với một số điều chỉnh để phù hợp với quy mô ứng dụng. Tại thư mục gốc, ngoài thư mục `src/` chứa mã nguồn chính, dự án có một tập hợp các file cấu hình quan trọng: `angular.json` định nghĩa toàn bộ vòng đời build (builder, asset, style, environment replacement), `tsconfig.json` cấu hình TypeScript 5.9.3 cho toàn bộ project, `tailwind.config.js` cấu hình Tailwind CSS utility classes, `ngsw-config.json` quy định chiến lược cache của Service Worker, `transloco.config.js` cấu hình Transloco i18n, và `purgecss.config.js` loại bỏ CSS không dùng trong production build để giảm kích thước stylesheet.

Bên trong `src/`, cấu trúc được phân chia thành: thư mục `app/` chứa toàn bộ mã Angular, thư mục `assets/` chứa tài nguyên tĩnh (i18n, icons, images, fonts), thư mục `environments/` chứa hai file cấu hình (`environment.ts` cho development, `environment.prod.ts` cho production), và các file cấp root như `main.ts` (entry point), `index.html` (shell HTML), `styles.scss` (global styles), `manifest.webmanifest` (PWA manifest), `robots.txt`, và `_redirects` (dùng với Cloudflare Pages/Netlify để đảm bảo SPA routing). File `polyfills.ts` dù vẫn tồn tại nhưng không còn cần thiết kể từ Angular 21 sử dụng native browser APIs.

Thư mục `assets/i18n/` chứa hai file JSON ngôn ngữ (`en.json`, `vi.json`) được load động bởi Transloco — mỗi file có cấu trúc key-value phẳng với các namespace như `pageTitles`, `buttons`, `errors`, `messages`, giúp dịch toàn bộ UI mà không cần rebuild. Thư mục `assets/icons/` chứa SVG icon set riêng của dự án, trong khi font chữ phụ cho subtitle (như Noto Sans JP) được load từ CDN (`cdn.jsdelivr.net/gh/daplex-site/subtitle-fonts`) thay vì đóng gói vào bundle — chỉ tải khi người dùng xem video có subtitle.

## Cấu Hình TypeScript và Angular

File `tsconfig.json` cấu hình TypeScript với mức độ type safety cao nhất. Target compile là `ES2022` với `moduleResolution: "bundler"` — lựa chọn phù hợp với Vite bundler, cho phép resolution logic giống Node.js nhưng không cần file extension trong import. Bảng sau tổng hợp các option đáng chú ý:

| Option | Giá trị | Mục đích |
|---|---|---|
| `strict` | `true` | Bật toàn bộ strict flags (`strictNullChecks`, `noImplicitAny`, `strictFunctionTypes`...) |
| `noImplicitOverride` | `true` | Yêu cầu đánh dấu rõ ràng khi override method ở subclass |
| `noPropertyAccessFromIndexSignature` | `true` | Ngăn truy cập thuộc tính tùy tiện qua index signature |
| `noImplicitReturns` | `true` | Đảm bảo mọi code path đều có return statement |
| `experimentalDecorators` | `true` | Bắt buộc cho Angular decorators kiểu cũ (`@Component`, `@Injectable`) |
| `useDefineForClassFields` | `false` | Cần thiết để DI của Angular hoạt động đúng với cơ chế initialize class field |
| `strictTemplates` | `true` | Bắt lỗi type trong HTML template tại compile time thay vì runtime |
| `strictInjectionParameters` | `true` | Kiểm tra type của tham số injection tại compile time |

Trong `angular.json`, schematic mặc định cho component được đặt `changeDetection: "OnPush"` và `standalone: false` — mọi component được tạo mới bằng `ng generate component` sẽ tự động dùng OnPush Change Detection và khai báo trong NgModule thay vì standalone.

> OnPush là chế độ change detection hiệu quả hơn mặc định: Angular chỉ kiểm tra component khi Input thay đổi reference, khi có event từ template, hoặc khi Observable emit — không quét toàn bộ cây component sau mỗi microtask như chế độ Default. Điều này đặc biệt quan trọng với các trang có nhiều item như danh sách media (100+ card) hay trang admin quản lý playlist.

## Cấu Trúc Thư Mục `app/core`

Thư mục `app/core` là trái tim của ứng dụng, chứa tất cả code infrastructure chạy ở tầng singleton. Phân tích chi tiết từng subfolder cho thấy sự phân chia trách nhiệm rõ ràng:

**`core/services/`** chứa 13 injectable service, mỗi service ứng với một domain nghiệp vụ:

| Service | Chức năng |
|---|---|
| `AuthService` | Quản lý phiên xác thực và JWT |
| `MediaService` | CRUD media, fetch catalog |
| `HistoryService` | Lịch sử xem và watch time |
| `RatingsService` | Đánh giá sao |
| `PlaylistsService` | Tạo/sửa/xóa playlist và playlist item |
| `CollectionService` | Bộ sưu tập media |
| `GenresService` | Quản lý thể loại |
| `TagsService` | Quản lý tag |
| `ProductionsService` | Metadata nhà sản xuất |
| `UsersService` | Quản lý user profile |
| `AuditLogService` | Nhật ký kiểm toán |
| `QueueUploadService` | Hàng đợi upload file phân đoạn |
| `ItemDataService` | In-memory data passing giữa route |
| `DestroyService` | Subject emit khi component destroy, dùng với `takeUntil` để tự động unsubscribe |

**`core/models/`** định nghĩa hơn 40 TypeScript class/interface ánh xạ trực tiếp với response schema của API: `MediaModel`, `MovieModel`, `TvShowModel`, `TvEpisodeModel`, `UserModel`, `RoleModel`, `MediaStreamModel`, `MediaStreamSubtitleModel`, `MediaVideoModel`, `HistoryModel`, `PlaylistModel`, `RatingModel`, v.v. Mỗi model được dùng để type-check response data từ HTTP client, đảm bảo frontend và backend đồng thuận về cấu trúc dữ liệu. Đây là lợi thế rõ ràng của việc dùng TypeScript full-stack — model definition ở frontend và DTO ở backend có thể được kiểm tra tay để đảm bảo nhất quán.

**`core/dto/`** chứa các Data Transfer Object cho request body, tổ chức theo domain: `dto/auth/` (SignInDto, SignUpDto, ConfirmEmailDto...), `dto/media/` (CreateMediaDto, UpdateMediaDto, CursorPageMediaDto, FindMediaStreamDto...), `dto/playlists/`, `dto/collections/`, `dto/history/`, `dto/ratings/`, `dto/tags/`, `dto/genres/`, `dto/productions/`, `dto/users/`. DTOs phân biệt với models ở chỗ chúng mô tả dữ liệu đi **lên** server (request), trong khi models mô tả dữ liệu đi **xuống** client (response).

**`core/enums/`** tập trung 25+ enum constant dùng toàn ứng dụng: `MediaType` (Movie/TvShow), `MediaStatus`, `MediaVisibility`, `MediaStorageType`, `VideoCodec` (H264/H265/VP9/AV1), `AudioCodec`, `UserPermission` (bitmask), `SocketMessage`, `SocketRoom`, `StorageKey`, `AppErrorCode`, `MediaBreakpoints`, `QueueUploadStatus`, v.v. Tập trung enum tại một nơi ngăn chặn magic string và đảm bảo đồng nhất giữa các component.

**`core/utils/`** chứa hơn 15 utility function và class không phụ thuộc Angular. Utils được export qua barrel file `utils/index.ts` để import gọn hơn.

| Utility | Chức năng |
|---|---|
| `StreamManifestHelper` | Convert DASH manifest sang M3U8 data URI cho Vidstack |
| `SubtitleStyleHelper` | Tạo CSS từ subtitle display preferences |
| `DateFnsHelper` | Format date với locale vi/en dùng date-fns |
| `ObjectHelper` | Các hàm tiện ích xử lý object |
| `NumberHelper` | Các hàm tiện ích xử lý số |
| `FormHelper` | Các hàm tiện ích xử lý reactive form |
| `DeviceHelper` | Detect browser/device type |
| `BlobBuilder` | Xây dựng Blob từ dữ liệu nhị phân |
| `ThumbhashHelper` | Generate progressive image placeholder từ thumbhash |

**`core/interfaces/`** nhóm TypeScript interface theo context sử dụng. Phân chia này tránh file interface khổng lồ và giúp IDE auto-import chính xác hơn.

| Subfolder | Nội dung |
|---|---|
| `interfaces/video-player/` | StreamManifest, LocalPlayerSettings, ManifestConverterOptions |
| `interfaces/ws/` | WebSocket event payload types |
| `interfaces/forms/` | Typed ReactiveForm interfaces |
| `interfaces/events/` | Component event output types |
| `interfaces/primeng/` | SelectOption, DataMenuItem wrappers |

## Cấu Hình Service Worker (PWA)

File `ngsw-config.json` định nghĩa hai asset group với chiến lược cache khác nhau. Group `app` với `installMode: "prefetch"` cache toàn bộ bundle chính (`index.html`, `*.js`, `*.css`, favicon) ngay khi Service Worker cài đặt lần đầu — người dùng có thể load lại trang mà không cần kết nối mạng sau lần truy cập đầu tiên. Group `assets` với `installMode: "lazy"` chỉ cache file khi nó được request lần đầu, bao gồm ảnh, font, SVG, và các file tĩnh khác — chiến lược lazy phù hợp vì không muốn prefetch toàn bộ ảnh thumbnail của catalog media vào cache ngay từ đầu.

Kết hợp với `updateMode: "prefetch"` cho group assets, khi có version mới của ứng dụng, Angular Service Worker tự động cập nhật cache bundle trong nền — người dùng sẽ nhận prompt "cập nhật ứng dụng" mà không cần reload tay. `manifest.webmanifest` định nghĩa metadata PWA (name, theme color `#1976d2`, icons) để trình duyệt có thể hiển thị lựa chọn "Add to Home Screen" trên mobile và desktop.

## Hằng Số Toàn Cục (config.ts)

File `src/environments/config.ts` tập trung tất cả constant không thay đổi theo môi trường (development/production). Bảng sau liệt kê các constant chính được định nghĩa trong file này:

| Constant | Giá trị | Ý nghĩa |
|---|---|---|
| Giới hạn poster | 3 MiB | Kích thước tối đa file ảnh poster |
| Giới hạn backdrop | 5 MiB | Kích thước tối đa file backdrop |
| Giới hạn avatar | 4 MiB | Kích thước tối đa file avatar |
| Giới hạn video nguồn | 40 GiB | Kích thước tối đa file video upload |
| Kích thước tối thiểu poster | 500×750px (2:3) | Ràng buộc tỉ lệ và độ phân giải ảnh poster |
| Kích thước tối thiểu backdrop | 1280×720px (16:9) | Ràng buộc tỉ lệ và độ phân giải backdrop |
| `QUEUE_UPLOAD_CHUNK_SIZE` | 10 MiB | Kích thước mỗi chunk khi upload phân đoạn |
| `QUEUE_UPLOAD_RETRIES` | 5 | Số lần retry upload thất bại (delay 3 giây/lần) |
| `SIGN_IN_LIMIT_COUNT` | 5 | Số lần đăng nhập sai tối đa trong TTL |
| `SIGN_IN_LIMIT_TTL` | 300 giây | Cửa sổ thời gian áp dụng giới hạn đăng nhập |
| `HTTP_CACHE_TTL` | 300000 ms (5 phút) | Thời gian cache HTTP client |

Việc tách constant ra khỏi service và component vào một file duy nhất có hai lợi ích: thứ nhất, dễ điều chỉnh business rule (ví dụ thay đổi giới hạn kích thước ảnh) mà không cần tìm trong nhiều file; thứ hai, constant được import trực tiếp bởi cả component lẫn validator — ví dụ `FileUploadComponent` dùng `UPLOAD_POSTER_SIZE` để validate trước khi upload, còn `MaxFileSizeValidator` cũng dùng cùng constant đó để validate trong reactive form, đảm bảo cả UI feedback và form validation dùng cùng một giá trị.

## Tổ Chức Shared Layer

Thư mục `shared/` được chia thành sáu nhóm rõ ràng. Mỗi nhóm sẽ được import vào feature module thông qua `SharedModule` hoặc import trực tiếp component/directive tương ứng, tùy thuộc feature module đó cần gì.

| Thư mục | Nội dung |
|---|---|
| `components/` | 21 component UI tái sử dụng |
| `dialogs/` | Các modal/dialog độc lập (dialog thêm vào playlist, dialog xác nhận xóa) |
| `directives/` | Angular directive tái sử dụng (infinite scroll, click outside) |
| `pipes/` | Angular pipe (format duration, format file size) |
| `layouts/` | Hai shell layout: `home-layout/` và `admin-layout/` |
| `modules/` | Hiện tại chứa `ws/` module |

`HomeLayoutComponent` trong `shared/layouts/home-layout/` đóng vai trò shell cho toàn bộ public-facing interface: nó render `home-header` (thanh điều hướng trên), `home-footer` (footer), `slide-menu` (menu trượt mobile), và `<router-outlet>` ở giữa nơi các route con của HomeModule, MediaModule, UsersModule render vào. `AdminLayoutComponent` tương tự nhưng render sidebar admin với menu điều hướng dọc thay vì header ngang. Việc tách layout ra khỏi AppComponent cho phép các route không thuộc layout nào (ví dụ trang lỗi 404, trang đăng nhập fullscreen) render trực tiếp tại AppComponent mà không có header/footer.
