# 4.4. Xây Dựng DaPlex-Transcoder (Media Server)

Chương này trình bày kiến trúc và cơ chế hoạt động của DaPlex-Transcoder — thành phần chịu trách nhiệm encode video thành các luồng DASH đa codec và đa bitrate. Transcoder được thiết kế như một process độc lập với pipeline encode rõ ràng, cơ chế cancel, và khả năng điều phối khi triển khai nhiều instance. Các phần bên dưới đi sâu vào từng lớp kiến trúc: worker, pipeline encode, và cơ chế cancel/điều phối.

---

## Tổng Quan Kiến Trúc

DaPlex-Transcoder là ứng dụng NestJS độc lập chạy trên Fastify, lắng nghe cổng 3001, chịu trách nhiệm hoàn toàn cho quá trình encode video từ file nguồn thành các luồng DASH phù hợp nhiều codec và bitrate.

> Tách biệt Transcoder thành một process riêng là quyết định kiến trúc quan trọng — encode video là tác vụ CPU-intensive có thể kéo dài hàng chục phút đến vài giờ, chạy chung với API sẽ làm giảm khả năng phục vụ HTTP request. Với kiến trúc tách biệt, API và Transcoder có thể deploy trên hai máy chủ khác nhau: API trên server có network tốt để phục vụ stream, Transcoder trên server có CPU mạnh (thậm chí GPU cho hardware encode) để xử lý video.

Transcoder kết nối vào cùng `REDIS_QUEUE_URL` với API để consume encode jobs từ bốn BullMQ queue (`video-transcode:1/2/3/4`). Kết nối MongoDB (`DATABASE_URL`) không được giữ thường trực mà chỉ mở tạm thời ở đầu mỗi job để đọc cấu hình và ngắt kết nối ngay sau đó — giảm số lượng connection đồng thời đến database. Hệ thống logging dùng `WinstonModule` với hai transport: console output (với timestamp và millisecond timing) và `DailyRotateFile` lưu log theo ngày vào thư mục `logs/`, giữ tối đa 300 ngày, format JSON để dễ parse — phục vụ debugging lỗi encode sau này.

`AppModule` của Transcoder import ba module nghiệp vụ: `VideoModule` (chứa 4 BullMQ worker consume encode jobs và `VideoService` thực thi encode), `VideoCancelModule` (chứa worker riêng xử lý tín hiệu hủy), và `TranscoderApiModule` (HTTP client kiểm tra trạng thái Transcoder phụ). `ScheduleModule.forRoot()` kích hoạt cron job để cleanup.

## Worker Architecture — BaseVideoConsumer và Bốn Codec Worker

`BaseVideoConsumer` là abstract class extend `WorkerHost` cung cấp logic lifecycle chung cho bốn codec worker. Ngoài việc inject Winston logger, class định nghĩa ba utility method: `pauseWorker()`, `resumeWorker()`, và `closeWorker()` để điều khiển worker state từ bên ngoài — ví dụ khi admin muốn tạm dừng encode để bảo trì server. Ba `@OnWorkerEvent` handler log trạng thái paused/resumed/closed ra Winston.

Bốn class con `VideoCosumerH264`, `VideoCosumerH265`, `VideoCosumerVP9`, `VideoCosumerAV1` đều decorate `@Processor('video-transcode:N', { concurrency: 1 })` — `concurrency: 1` đảm bảo mỗi worker chỉ xử lý một job tại một thời điểm. Mỗi class `process()` method chỉ làm một việc: ủy thác cho `videoService.transcode(job, VideoCodec.N)`.

> Thiết kế này tách logic worker lifecycle khỏi logic encode — toàn bộ encode logic tập trung trong `VideoService`, dễ test và dễ đọc hơn so với nhét tất cả vào consumer.

## Luồng Encode — VideoService.transcode()

`VideoService.transcode(job, codec)` là method trung tâm của toàn bộ Transcoder, thực hiện một pipeline dài từ khi nhận job đến khi publish kết quả. Pipeline có thể chia thành bảy giai đoạn:

**Giai đoạn 1 — Kiểm tra cancel.** Method đầu tiên kiểm tra `CanceledJobIds[]` in-memory — nếu job ID có trong danh sách (do `VideoCancelConsumer` thêm vào trước đó), trả về `{}` ngay lập tức mà không thực hiện bất kỳ thao tác nào. Danh sách được lọc: chỉ giữ lại các ID lớn hơn job ID hiện tại (Snowflake ID có thứ tự thời gian, ID nhỏ hơn là job cũ hơn đã xử lý xong).

**Giai đoạn 2 — Kết nối MongoDB và đọc cấu hình.** `mongoose.connect(DATABASE_URL, { useBigInt64: true })` mở connection tạm thời. Service đọc `appSettings` (cấu hình encode từ admin dashboard), `mediaInfo` (lấy `originalLang` để gán cho audio track), và `streamStorage` (lấy `publicUrl` để build URL CDN cho input file khi `UseURLInput=true`). Sau khi đọc xong, `mongoose.disconnect()` ngay — connection chỉ tồn tại trong thời gian ngắn đầu job. Encoding parameters ưu tiên từ `appSettings` nếu admin đã custom, ngược lại fallback về constants trong `config.ts`.

**Giai đoạn 3 — Kiểm tra interruption và download file nguồn.** Nếu thư mục `transcodeDir = ${TRANSCODE_DIR}/${job.id}` đã tồn tại từ lần chạy trước (server bị restart giữa chừng), Transcoder publish `retry-encoding` result job để API biết, xóa thư mục cũ, rồi bắt đầu lại từ đầu. Sau đó tạo thư mục mới và download file nguồn via `rcloneHelper.downloadFile()`. Nếu bật `USE_URL_INPUT=true` (truy cập trực tiếp file nguồn qua URL CDN thay vì download về local), việc download chỉ xảy ra khi encode H264 (vì H264 cần encode audio trước) — các codec còn lại skip download, đọc file qua URL trực tiếp, tiết kiệm đáng kể thời gian và dung lượng đĩa khi encode nhiều codec cùng một file nguồn.

**Giai đoạn 4 — Phân tích file nguồn.** `FFprobe` đọc thông tin stream (video/audio tracks, codec, fps, bitrate, color space), `MediaInfo` đọc thêm metadata chi tiết hơn (frame rate chính xác từ `FrameRate` field). Hai tool bổ sung cho nhau vì FFprobe đôi khi không đủ thông tin cho các container phức tạp. Nếu không tìm thấy video track hoặc audio track, Transcoder throw `UnrecoverableError` để BullMQ không retry (lỗi do file nguồn bị corrupt, retry cũng không giải quyết được). `validateSourceQuality()` so sánh chiều cao nguồn với `qualityList` (H264 dùng `ENCODING_QUALITY`, các codec khác dùng `NEXT_GEN_ENCODING_QUALITY` với ít quality tier hơn để tiết kiệm thời gian), lọc ra danh sách chất lượng thực sự có thể encode.

**Giai đoạn 5 — Publish update-source và encode audio.** Sau khi có thông tin source, Transcoder publish job `update-source` lên `VIDEO_TRANSCODE_RESULT` queue để API cập nhật metadata (duration, resolution thực tế). Audio encoding chỉ thực hiện một lần cho codec H264 — kết quả audio track (AAC + Opus) được dùng chung cho tất cả codec video. Logic chọn track: ưu tiên surround track (>2 channels) làm default audio, stereo/mono làm secondary; nếu admin cấu hình `selectAudioTracks` (index cụ thể) và `extraAudioTracks` (ngôn ngữ phụ như commentary), service encode thêm các track đó. Mỗi track encode hai dạng: AAC (cho HLS compatibility) và Opus (cho DASH performance trên trình phát web).

**Giai đoạn 6 — Remux và encode video.** Nếu bật `SPLIT_ENCODING=true` kết hợp `USE_URL_INPUT=true` và file nguồn không phải MKV, service remux sang MKV trước bằng ffmpeg (copy stream, không re-encode), upload lên storage, rồi dùng URL CDN của MKV này làm input cho encode. Lý do: MKV container hỗ trợ seeking tốt hơn, cho phép encode nhiều codec song song cùng file mà không cần mỗi codec tự download bản copy riêng. Encode video gọi `encodeByCodec()` với tham số phù hợp cho từng codec — H264 dùng `VIDEO_H264_PARAMS`, H265 dùng `VIDEO_H265_PARAMS`, VP9 dùng `VIDEO_VP9_PARAMS`, AV1 dùng `VIDEO_AV1_PARAMS`. Trước khi encode H265/VP9/AV1, service kiểm tra HDR metadata (`hdrMetadataHelper.getHdrMetadata()`) — nếu là HDR video, truyền thêm tone-mapping params để preserve HDR trong output.

**Giai đoạn 7 — Thumbnail sprite và xác nhận upload.** Chỉ sau khi H264 encode xong, service gọi `generateSprites()` tạo hai bộ thumbnail grid: size M (160×160 pixel, grid 10×10, 100 frame mỗi page) và size L (320×320 pixel, grid 5×5, 25 frame mỗi page). Thumbnail sprite là ảnh JPEG ghép nhiều frame thành một grid — trình phát dùng để hiển thị preview khi user hover lên progress bar mà không cần tải từng frame riêng. Sau khi upload thumbnails, service kiểm tra số file đã upload so với `totalExpectedFiles` (= số quality tier + 1 source + 3 audio files) — nếu thiếu, retry check tối đa 5 lần (S3-compatible API có thể có latency listing).

## Cơ Chế Cancel và Điều Phối Đa Transcoder

`VideoCancelConsumer` xử lý `@Processor(TaskQueue.VIDEO_CANCEL, { concurrency: 1 })` — một queue riêng cho tín hiệu hủy. Khi API enqueue job cancel (khi admin click "Hủy encode"), `process()` gọi `videoService.addToCanceled(job.data)` để thêm job ID vào `CanceledJobIds[]`, rồi `job.discard()` để BullMQ không retry. Khi `VideoService.transcode()` đang chạy và gặp tín hiệu cancel (được throw từ `ffmpegHelper` khi phát hiện `CanceledJobIds`), nó catch lỗi, log, và return sớm. Thiết kế in-memory list hoạt động tốt cho single-instance Transcoder nhưng có hạn chế: nếu chạy nhiều instance, cancel signal chỉ đến một instance.

> Đây là trade-off chấp nhận được vì Transcoder thường deploy một instance.

`TranscoderApiService.checkAndWaitForTranscoderPriority()` giải quyết trường hợp deploy hai Transcoder instance song song. Trước khi bắt đầu remux (bước CPU-intensive nhất), service polling `PRIMARY_TRANSCODER_URL/video/transcoder-priority` — endpoint này trả về `{ priority: number }` phản ánh Transcoder đó có đang bận không (`priority > 0`). Nếu instance chính đang bận, instance phụ wait 10 giây và poll lại, tối đa 30 lần trước khi tiếp tục. Cơ chế này tránh hai instance cạnh tranh CPU của cùng một máy chủ khi Transcoder và Transcoder phụ cùng chạy trên một server.

`DaplexApiService.ensureProducerAppIsOnline()` là safety check trước khi Transcoder publish result job về API. Service ping URL của API (từ `job.data.producerUrl`), retry tối đa 10 lần với khoảng cách 30 giây, để đảm bảo API đang online và sẵn sàng nhận result. Nếu file `BYPASS_PRODUCER_CHECK_FILE` tồn tại trên disk, check này bị bypass — cho phép operator tắt check trong môi trường development hoặc khi API offline có chủ đích. Validation URL được thực hiện qua whitelist `DAPLEX_API_DOMAINS` env var (đọc từ `producers-domains.txt`) để tránh SSRF — Transcoder không gọi ra URL tùy ý mà người dùng cung cấp.
