# 4.5.3. Đóng Gói CMAF và StreamManifest

Mục này mô tả cách DaPlex đóng gói video/audio đã encode thành định dạng CMAF single-file và xây dựng file manifest JSON tích hợp thông tin DASH lẫn HLS. CMAF cho phép một file MP4 phục vụ cả hai chuẩn streaming mà không phải lưu nội dung hai lần, trong khi StreamManifest JSON loại bỏ hoàn toàn nhu cầu server-side MPD generation. Mục bao gồm lý do chọn CMAF, lệnh MP4Box, quản lý đĩa, cấu trúc MPD và m3u8 output, StreamManifest JSON schema, và quy trình lưu manifest sau khi tất cả quality tier hoàn thành.

---

## CMAF — Common Media Application Format và Lý Do Chọn

Sau khi FFmpeg tạo ra file MP4 "thô" chứa video hoặc audio đã encode, bước tiếp theo là đóng gói lại file này thành định dạng **CMAF (Common Media Application Format)** trước khi upload lên storage. CMAF là một profile của ISOBMFF (ISO Base Media File Format) được Apple, Microsoft, và DASH-IF cùng xây dựng nhằm tạo ra định dạng container duy nhất tương thích với cả DASH và HLS — hai chuẩn streaming chính hiện nay. Trước CMAF, HLS yêu cầu MPEG-TS container còn DASH yêu cầu fragmented MP4 (fMP4), buộc nhà phát hành phải mã hóa hoặc lưu trữ nội dung hai lần. CMAF giải quyết vấn đề này bằng cách sử dụng fragmented MP4 duy nhất mà cả HLS (từ v7) lẫn DASH đều đọc được, giảm chi phí lưu trữ và đơn giản hóa pipeline.

DaPlex chọn variant đặc biệt của CMAF là **single-file CMAF** (còn gọi là "On-Demand profile" trong thuật ngữ DASH): toàn bộ video được đóng gói thành **một file MP4 duy nhất** thay vì chia thành hàng trăm segment file nhỏ. Trong single-file CMAF, file MP4 được tổ chức thành nhiều fragment, mỗi fragment là một đơn vị segment độc lập. Hai byte range trong file đóng vai trò quan trọng: **initRange** là offset của initialization segment (chứa codec parameters, track info) và **indexRange** là offset của segment index (Side Index Box — chứa timing và offset của từng segment fragment trong file). Trình phát không cần tải toàn bộ file — chỉ cần tải initRange và indexRange qua HTTP Range request, rồi tính toán offset của từng segment cần phát và request đúng byte range đó.

> Cách tiếp cận này giảm số lượng file cần quản lý trên cloud storage từ hàng trăm (segment-based) xuống còn một file cho mỗi quality tier.

## createMP4BoxPackArgs — Lệnh Đóng Gói

`createMP4BoxPackArgs()` tạo args cho MP4Box binary từ GPAC project. Lệnh hoàn chỉnh có dạng:

```
MP4Box -dash 6000 -profile onDemand -segment-name "{tempName}$Init=$" -out "{playlistName}:dual" "{input}"
```

| Tham số | Giá trị ví dụ | Mô tả |
|---|---|---|
| `-dash` | `6000` | Segment duration 6000ms (6 giây). GOP = fps×2 đảm bảo mỗi segment chứa đúng 3 GOP (6÷2=3), mọi segment boundary đều là keyframe. Nếu GOP không khớp, MP4Box vẫn tạo segment nhưng gây "seeking artifact" khi switch quality tier. |
| `-profile` | `onDemand` | DASH On-Demand profile — single-file CMAF, khác với `-profile live` tạo segment riêng biệt. Phù hợp VoD vì toàn bộ nội dung đã có sẵn, không cần sliding window. |
| `-segment-name` | `"{tempName}$Init=$"` | Tên initialization segment nhúng trong file. Windows dùng `$Init=$`; Linux cần escape `\$Init=\$` vì POSIX shell expand `$`. Service kiểm tra `process.platform === 'win32'` để chọn cú pháp đúng. |
| `-out` | `"{playlistName}:dual"` | Suffix `:dual` yêu cầu MP4Box sinh ra cả DASH MPD lẫn HLS m3u8 từ cùng một file nguồn. Kết quả: (1) MP4 CMAF, (2) `.mpd` DASH manifest XML, (3) `_1.m3u8` HLS byte-range playlist. |

## prepareMediaFile — Quản Lý Đĩa và Đổi Tên

`prepareMediaFile()` bọc toàn bộ quá trình packaging với logic kiểm tra dung lượng đĩa trước khi thực thi. Lý do cần kiểm tra: MP4Box đọc file input và tạo file output mới hoàn toàn — trong quá trình đó, cần dung lượng đĩa gấp đôi file gốc. Nếu không kiểm tra và server hết đĩa giữa chừng, MP4Box tạo ra file output không hoàn chỉnh nhưng không báo lỗi rõ ràng. `diskSpaceUtil.hasFreeSpaceToCopyFile()` kiểm tra dung lượng available tại thư mục đích so với kích thước file input.

Nếu không đủ dung lượng, service xóa file nguồn gốc ban đầu (file video nguồn đã download) để giải phóng chỗ trước khi MP4Box chạy, ghi log cảnh báo. Sau khi MP4Box hoàn thành và xóa file input thô (encoded raw), nếu đã xóa file nguồn, service tự động download lại qua rclone (`rcloneHelper.downloadFile()`).

> Cơ chế "sacrifice and re-download" này đảm bảo packaging không bị stuck vì thiếu đĩa, đánh đổi bằng thời gian download lại — một trade-off chấp nhận được vì các codec tiếp theo (H265/VP9/AV1) có thể dùng `USE_URL_INPUT` để không cần download.

Sau khi MP4Box chạy xong, file output có tên tạm là `{tempFileName}.mp4`. Service đổi tên (`fileHelper.renameFile`) sang `{finalOutputName}.mp4` — bước rename này atomic hơn copy trên cùng filesystem, đảm bảo không có trạng thái file không hoàn chỉnh nào được đọc bởi bước tiếp theo.

## Cấu Trúc DASH MPD Output

File `.mpd` do MP4Box tạo ra là XML theo chuẩn MPEG-DASH. Các field quan trọng mà `StreamManifest` extract bao gồm:

| Field MPD | Mô tả |
|---|---|
| `MPD.mediaPresentationDuration` | Tổng thời lượng, dạng ISO 8601 như `PT1H23M45.6S` |
| `MPD.minBufferTime` | Thời gian buffer tối thiểu |
| `MPD.maxSubsegmentDuration` | Thời lượng segment tối đa — khớp với `-dash 6000` |
| `MPD.Period.duration` | Thời lượng period, thường bằng `mediaPresentationDuration` cho VoD một period |
| `MPD.Period.AdaptationSet.Representation.bandwidth` | Bitrate tính bằng bps |
| `MPD.Period.AdaptationSet.Representation.codecs` | Codec string chuẩn RFC 6381, ví dụ `avc1.64001f` cho H264 |
| `MPD.Period.AdaptationSet.Representation.SegmentBase.indexRange` | Byte range dạng `"688-1023"` |
| `MPD.Period.AdaptationSet.Representation.SegmentBase.Initialization.range` | Byte range dạng `"0-687"` |
| `AdaptationSet.par` | Picture aspect ratio, ví dụ `16:9` |
| `AdaptationSet.sar` | Sample aspect ratio — dùng để trình phát hiển thị đúng tỉ lệ khung hình |

## Cấu Trúc HLS m3u8 Output

File `_1.m3u8` do MP4Box tạo ra là HLS playlist version 7 với byte range segment references — không phải playlist với URL segment riêng biệt. Định dạng này tận dụng khả năng byte range của HLS (từ version 4, được Apple hỗ trợ từ iOS 5) để tham chiếu các vùng trong cùng file MP4 CMAF. Mỗi segment trong playlist có dạng:

```
#EXT-X-BYTERANGE:204800@512000
video_1080p.mp4
```

Số `204800` là length (bytes), `@512000` là offset trong file. Trình phát HLS tải đúng byte range đó từ file MP4 bằng HTTP Range header. `m3u8-parser` (thư viện NPM, phát triển bởi Video.js team) parse file này và trả về `parser.manifest.segments[]`, mỗi segment có `duration` (floating-point seconds), `byterange.length`, `byterange.offset`, và `timeline` (monotonic index). Parser cũng trả về `targetDuration` (ceiling của max segment duration, số nguyên) và `segments[0].map.byterange` (byte range của initialization segment — khớp với `initRange` từ MPD).

## StreamManifest — JSON Schema Tích Hợp DASH + HLS

`StreamManifest` class đóng vai trò là custom manifest builder, tổng hợp thông tin từ MPD và m3u8 thành một JSON document duy nhất. `HlsManifest` interface định nghĩa cấu trúc top-level:

| Field | Kiểu / Giá trị | Mô tả |
|---|---|---|
| `hlsVersion` | `7` | HLS version |
| `videoTracks` | `HlsVideoTrack[]` | Danh sách video track |
| `audioTracks` | `HlsAudioTrack[]` | Danh sách audio track |
| `totalDuration` | `float` | Tổng thời lượng tính bằng giây |
| `targetDuration` | `6` | Segment duration |
| `mediaSequence` | `0` | Bắt đầu từ 0 với VoD |
| `playlistType` | `'VOD'` | Loại playlist |

Mỗi `HlsVideoTrack` chứa: `codec` (RFC 6381 string từ MPD), `codecID` (VideoCodec enum integer), `width`, `height`, `par`, `sar`, `bandwidth` (bps), `duration` (giây), `format`, `language`, `mimeType`, `frameRate`, `uri` (đường dẫn tương đối `{streamId}/{fileName}` trong storage), và hai object DASH+HLS segment data. `dashSegment` (`DashSegmentGroup`) chứa `minBufferTime`, `mediaPresentationDuration`, `maxSubsegmentDuration`, `indexRange: { start, end }`, `initRange: { start, end }` — các số nguyên parse từ chuỗi "start-end" trong MPD. `hlsSegment` (`HlsSegmentGroup`) chứa `targetDuration`, `byterange` của initialization map, và `segments: HlsSegment[]` — mảng các segment với `timeline`, `duration`, và `byterange: { length, offset }`.

`HlsAudioTrack` có cấu trúc tương tự với các field bổ sung đặc thù audio: `name` (tên hiển thị như "AAC Stereo", "Opus 5.1"), `group: 'audio'`, `default`, `autoselect`, `language`, `title`, `channels`, `samplingRate`, `codecID` (AudioCodec bitmask). Method `getAudioName()` tạo tên hiển thị theo quy tắc: 1 kênh → `"{Format} Mono"`, 2 kênh → `"{Format} Stereo"`, nhiều hơn → `"{Format} {channels-1}.1"` (ví dụ 6 kênh → "AAC 5.1").

Cả hai `appendVideoPlaylist()` và `appendAudioPlaylist()` đều dùng `luxon.Duration.fromISO()` để convert chuỗi ISO 8601 duration (như `"PT1H23M45.600S"`) sang số giây floating-point. `luxon` được chọn vì parser ISO 8601 duration built-in của JavaScript không tồn tại — `Date.parse()` chỉ xử lý ISO 8601 datetime, không xử lý duration.

## saveManifestFile và Kết Thúc Một Codec

`saveManifestFile()` được `transcode()` gọi sau khi tất cả quality tier của một codec đã encode và upload xong. Method tạo Snowflake ID mới cho stream, gọi `manifest.saveFile(manifestFilePath)` để serialize `HlsManifest` object thành JSON và ghi ra đĩa, upload file JSON qua rclone, rồi publish job `add-stream-manifest` lên `VIDEO_TRANSCODE_RESULT` queue với payload gồm `streamId`, `fileName = manifest_{codec}.json`, `codec`, và `hdrFormat` (nếu là HDR content). API nhận job này, tạo `MediaStorage` document cho manifest file, và cập nhật trạng thái stream trong MongoDB.

File `manifest_{codec}.json` là artifact cuối cùng và quan trọng nhất của quá trình encode. Angular frontend không cần gọi API để lấy MPD XML — thay vào đó, `Vidstack` player đọc JSON manifest này, reconstruct DASH playlist client-side từ `dashSegment.initRange`, `dashSegment.indexRange`, và `uri`, sau đó request byte range HTTP trực tiếp từ CDN.

> Thiết kế này loại bỏ hoàn toàn server-side MPD generation, giảm tải cho API server và tăng khả năng cache vì manifest là static file.
