# 4.1. Môi Trường Phát Triển

Chương này mô tả môi trường phát triển cục bộ của hệ thống DaPlex, bao gồm các công cụ, runtime, dịch vụ hạ tầng, và quy trình cấu hình cần thiết để chạy toàn bộ pipeline từ upload đến phát video. Việc hiểu rõ môi trường phát triển là nền tảng để nắm được cách các thành phần tương tác và phụ thuộc lẫn nhau trong quá trình triển khai thực tế.

---

## Tổng Quan

Hệ thống DaPlex gồm bốn thành phần độc lập — daplex-dune-v2 (frontend Angular), DaPlex-API (backend NestJS), DaPlex-Transcoder (transcoder NestJS), và Redis — mỗi thành phần có tập phụ thuộc và cấu hình runtime riêng. Mặc dù vậy, cả bốn thành phần đều dùng chung một ngôn ngữ lập trình (TypeScript), một hệ sinh thái quản lý gói (npm), và một bộ quy ước tổ chức mã nguồn nhất quán. Sự đồng nhất này giảm thiểu chi phí chuyển ngữ cảnh khi làm việc trên nhiều thành phần đồng thời, đồng thời cho phép tái sử dụng kiến thức về kiểu dữ liệu và thiết kế API giữa frontend và backend.

Môi trường phát triển được thiết kế để chạy hoàn toàn trên máy local mà không phụ thuộc vào dịch vụ cloud — developer chỉ cần cài đặt Node.js, MongoDB, Redis, FFmpeg, và MP4Box trên máy local để có thể chạy đầy đủ toàn bộ pipeline từ upload file nguồn đến phát video DASH. Cấu hình cloud storage (Rclone, Azure Blob, Cloudflare R2...) là tùy chọn trong giai đoạn phát triển — Transcoder có thể cấu hình dùng local filesystem thay vì remote storage.

## Runtime và Ngôn Ngữ

**Node.js** là runtime duy nhất cho toàn bộ hệ thống server-side. Phiên bản tối thiểu là Node.js 20 LTS, tương thích với Angular CLI 21, NestJS 10, và các phụ thuộc native như `sharp` (xử lý ảnh), `bcrypt` (hashing mật khẩu). Node.js 22 LTS cũng được hỗ trợ. Việc sử dụng một phiên bản runtime nhất quán trên cả frontend lẫn backend loại bỏ rủi ro incompatibility khi chạy build script hoặc sử dụng monorepo tool.

**TypeScript** được dùng xuyên suốt với hai phiên bản cấu hình riêng theo đặc thù từng dự án. DaPlex-API và DaPlex-Transcoder dùng TypeScript 5.3.3 — phiên bản stable tại thời điểm phát triển NestJS 10 — với cấu hình `tsconfig.json` kế thừa từ `@nestjs/tsconfig-paths`. Frontend daplex-dune-v2 dùng TypeScript 5.9.3, phiên bản mới hơn được Angular CLI 21 yêu cầu để hỗ trợ các tính năng type inference nâng cao. Cả ba dự án đều bật `strict: true`, `noImplicitAny: true`, và `strictNullChecks: true` để đảm bảo type safety tối đa.

**npm** là package manager thống nhất cho cả ba dự án. Tệp `package-lock.json` được commit vào repository để đảm bảo reproducible install trên mọi máy — điều đặc biệt quan trọng với các gói fork (`@ktt45678/vidstack`, `@ktt45678/primeng`) không có trên npm registry chính thức mà phải cài từ Git hoặc local path.

## Công Cụ Phát Triển

**@nestjs/cli** phiên bản 10.3.0 là công cụ dòng lệnh chính để khởi tạo, build, và chạy DaPlex-API và DaPlex-Transcoder trong chế độ development. Lệnh `nest start --watch` kích hoạt hot-reload dựa trên `@swc/core` (DaPlex-Transcoder) hoặc TypeScript compiler watch mode (DaPlex-API), tự động recompile và restart khi file thay đổi. DaPlex-Transcoder cấu hình `@swc/cli` và `@swc/core` làm compiler thay cho tsc để tăng tốc độ recompile trong development.

**@angular/cli** phiên bản 21.2.7 quản lý toàn bộ vòng đời dự án frontend: `ng serve` khởi động development server có HMR (Hot Module Replacement) tại cổng 4200, `ng build` tạo production bundle, `ng test` chạy unit test với Karma/Jasmine. Angular 21 dùng Vite làm bundler mặc định thay cho Webpack, rút ngắn đáng kể thời gian cold start của `ng serve` so với các phiên bản trước.

**ESLint** và **Prettier** được cấu hình trong cả ba dự án để đảm bảo nhất quán về style code. DaPlex-API và DaPlex-Transcoder dùng `@typescript-eslint/eslint-plugin` 6.x với ruleset NestJS, còn frontend dùng Angular ESLint. Prettier đảm bảo format đồng nhất (dấu nháy đơn, trailing comma ES5, print width 120) giữa các file TypeScript, HTML template, và SCSS.

**Swagger UI** (`@nestjs/swagger` 7.2.0) tự động sinh tài liệu API từ decorator `@ApiProperty`, `@ApiOperation` và type annotation trong controller. Endpoint `/api/docs` trả về interactive UI để test API trực tiếp, đặc biệt hữu ích khi phát triển các endpoint upload và encode phức tạp.

## Cơ Sở Hạ Tầng Dịch Vụ

**MongoDB** là database chính lưu trữ toàn bộ dữ liệu metadata. DaPlex-API kết nối hai connection riêng biệt: `DATABASE_URL` trỏ đến DATABASE_A chứa 16 collection chính (User, Media, Role...), và `DATABASE_URL_B` trỏ đến DATABASE_B chứa AuditLog và Notification. Phiên bản tối thiểu là MongoDB 6.0 để hỗ trợ time-series collection và `$densify` aggregation. Trong development, cả hai connection có thể trỏ cùng một MongoDB instance với hai database name khác nhau.

**Redis** đảm nhiệm ba vai trò đồng thời: hàng đợi tác vụ bất đồng bộ (BullMQ), distributed cache (ioredis + cache-manager), và pub/sub channel (Socket.IO Redis adapter). BullMQ dùng Redis để lưu job definition, job state, và retry metadata — khi Transcoder restart sau crash, job đang chạy dở sẽ tự động được lấy lại từ hàng đợi `bull:*` trong Redis. Phiên bản tối thiểu là Redis 6.2 (hỗ trợ `GETDEL`, `LMPOP`).

**FFmpeg** là công cụ encode video cốt lõi. DaPlex dùng custom build tích hợp đầy đủ codec: `libx264` (H.264), `libx265` (H.265/HEVC), `libvpx-vp9` (VP9), `libsvtav1` (AV1 qua SVT-AV1 encoder), `libfdk_aac` (AAC VBR), `libopus` (Opus). Các codec thế hệ mới như SVT-AV1 thường không có trong FFmpeg binary từ package manager hệ điều hành, nên dự án cung cấp build script tại `FFmpeg-Builds-SVT-AV1-HDR/` dựa trên Docker để cross-compile cho Linux x64, Linux ARM64, và Windows. Ngoài ra, FFmpeg cần hỗ trợ `zscale` filter (libzimg) cho HDR tonemapping khi encode video HDR xuống SDR.

**MP4Box** (từ dự án GPAC) là công cụ đóng gói DASH. Sau khi FFmpeg encode ra các file MP4 thô, MP4Box đảm nhiệm tạo DASH manifest (`.mpd`) và fragment các MP4 thành SegmentBase profile với `initRange` và `indexRange` — định dạng `isoff-on-demand` phù hợp với VoD. Phiên bản tối thiểu là GPAC 2.2.

**Rclone** thực hiện đồng bộ file encode lên cloud storage. DaPlex-Transcoder gọi `rclone copy` sau khi MP4Box đóng gói xong, hỗ trợ đồng thời nhiều provider: Backblaze B2, Cloudflare R2, Amazon S3, Azure Blob, và bất kỳ provider nào tương thích S3. Rclone được cấu hình qua environment variable `RCLONE_CONFIG` trỏ đến file `rclone.conf` hoặc dùng cơ chế environment-based config để tránh lưu credential trong file.

## Biến Môi Trường

Cả DaPlex-API và DaPlex-Transcoder đọc cấu hình từ biến môi trường thông qua `@nestjs/config` với validation bằng class-validator tại thời điểm startup. Nếu biến bắt buộc thiếu hoặc sai format, ứng dụng từ chối khởi động và in error message rõ ràng thay vì crash sau khi đã nhận request. Bảng dưới liệt kê các biến môi trường quan trọng nhất:

| Biến | Component | Mô Tả | Ví Dụ |
|---|---|---|---|
| `PORT` | API, Transcoder | Cổng HTTP server | `3000`, `3001` |
| `DATABASE_URL` | API, Transcoder | MongoDB URI — DATABASE_A | `mongodb://localhost:27017/daplex` |
| `DATABASE_URL_B` | API | MongoDB URI — DATABASE_B | `mongodb://localhost:27017/daplex_b` |
| `REDIS_URL` | API, Transcoder | Redis URI | `redis://localhost:6379` |
| `JWT_SECRET` | API | Ký JWT access token | chuỗi ngẫu nhiên 64+ ký tự |
| `JWT_REFRESH_SECRET` | API | Ký JWT refresh token | chuỗi ngẫu nhiên 64+ ký tự |
| `ACCESS_TOKEN_EXPIRY` | API | Thời gian sống access token | `900` (giây) |
| `REFRESH_TOKEN_EXPIRY` | API | Thời gian sống refresh token | `2592000` (giây, 30 ngày) |
| `TRANSCODER_URL` | API | URL nội bộ Transcoder | `http://localhost:3001` |
| `API_URL` | Transcoder | URL nội bộ API để callback | `http://localhost:3000` |
| `TRANSCODER_SECRET` | API, Transcoder | Shared secret cho internal call | chuỗi ngẫu nhiên 32+ ký tự |
| `FFMPEG_PATH` | Transcoder | Đường dẫn ffmpeg binary | `/usr/local/bin/ffmpeg` |
| `FFPROBE_PATH` | Transcoder | Đường dẫn ffprobe binary | `/usr/local/bin/ffprobe` |
| `MP4BOX_PATH` | Transcoder | Đường dẫn MP4Box binary | `/usr/local/bin/MP4Box` |
| `RCLONE_PATH` | Transcoder | Đường dẫn rclone binary | `/usr/local/bin/rclone` |
| `TEMP_DIR` | Transcoder | Thư mục tạm encode | `/tmp/daplex-transcode` |

Frontend Angular không dùng biến môi trường runtime; thay vào đó dùng Angular environment file (`src/environments/environment.ts`) được bake vào bundle tại thời điểm build. Giá trị chính là `apiUrl` (URL của DaPlex-API) và các flag feature toggle.

## Quy Trình Khởi Động

Thứ tự khởi động đúng trên máy developer là: (1) khởi động MongoDB và Redis trước, (2) khởi động DaPlex-API (`npm run start:dev`), (3) khởi động DaPlex-Transcoder (`npm run start:dev`), (4) khởi động Angular dev server (`npm start`). DaPlex-API phải chạy trước DaPlex-Transcoder vì Transcoder đăng ký BullMQ worker và cần Redis sẵn sàng, đồng thời sẽ gửi HTTP callback về API khi encode hoàn thành — nếu API chưa lên, callback sẽ thất bại và encode job sẽ retry theo backoff policy của BullMQ.

Khi lần đầu chạy API, `mongoose` tự động tạo collection và index trong MongoDB theo schema định nghĩa trong các `*Schema` class. Không cần migration script riêng cho schema khởi tạo — đây là đặc trưng của MongoDB schemaless với Mongoose auto-create. Tuy nhiên, khi thêm index mới vào schema đã có dữ liệu, `autoIndex: false` được bật trong production để tránh rebuild index block I/O — admin cần chạy `db.collection.createIndex()` thủ công trong maintenance window.
