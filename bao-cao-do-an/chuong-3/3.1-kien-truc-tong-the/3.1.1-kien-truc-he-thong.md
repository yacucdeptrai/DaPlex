# 3.1.1. Kiến Trúc Hệ Thống Tổng Quan

Mục này mô tả cách DaPlex tổ chức mã nguồn và hạ tầng triển khai như một hệ thống nhiều thành phần độc lập. Việc hiểu rõ cấu trúc này là nền tảng để nắm kiến trúc chi tiết của từng thành phần ở các mục tiếp theo. Nội dung bao gồm chiến lược umbrella repository với git submodule, hai topology triển khai, phân vùng bốn kết nối Redis, hai kết nối MongoDB, và giao thức giao tiếp nội bộ giữa API và Transcoder.

---

## Tổ Chức Mã Nguồn: Umbrella Repository với Git Submodule

Toàn bộ mã nguồn DaPlex được tổ chức dưới dạng một **umbrella repository** — một repository "vỏ" trên GitHub không chứa code ứng dụng trực tiếp mà chỉ quản lý bốn **git submodule** con: `daplex-dune-v2` (Angular PWA), `DaPlex-API` (NestJS REST API), `DaPlex-Transcoder` (NestJS worker), và `Redis` (cấu hình Redis). Mỗi submodule là một git repository hoàn toàn độc lập với lịch sử commit, branch, tag, và CI/CD pipeline riêng. File `.gitmodules` tại gốc umbrella repo chỉ ghi nhận URL GitHub và đường dẫn mount của từng submodule, không lưu nội dung code.

> Quyết định dùng git submodule thay vì monorepo (Nx, Turborepo) hoặc multi-repo hoàn toàn tách biệt xuất phát từ ba ràng buộc thực tế của dự án. Thứ nhất, ba thành phần backend có **ngôn ngữ runtime khác nhau về môi trường chạy**: API và Transcoder cùng dùng NestJS nhưng cần `node_modules` độc lập vì dependency tree có thể phân kỳ theo thời gian (ví dụ Transcoder cần `fluent-ffmpeg` còn API không cần). Thứ hai, Transcoder thường được **triển khai trên một máy chủ vật lý khác** so với API — một VPS chuyên biệt có cấu hình CPU cao — nên việc clone và cập nhật riêng lẻ từng submodule quan trọng hơn là tích hợp build chung. Thứ ba, submodule `Redis` lưu các file cấu hình Redis (`redis.conf`) có thể được áp dụng độc lập trên hạ tầng không cần cài Node.js. Monorepo yêu cầu một tool build trung tâm và workspace protocol; điều này tạo ra coupling không cần thiết giữa các thành phần vốn chạy trên những máy chủ khác nhau.

Quy trình nhân bản dự án yêu cầu thêm tham số `--recursive` khi `git clone` để tải đồng thời cả bốn submodule. Nếu bỏ qua tham số này, thư mục `daplex-dune-v2/`, `DaPlex-API/`, `DaPlex-Transcoder/`, và `Redis/` sẽ tồn tại nhưng rỗng. Các thao tác cập nhật submodule cũng cần lệnh `git submodule update --remote` để kéo commit mới nhất từ branch chính của từng submodule. Trong quy trình CI/CD thực tế, pipeline của umbrella repo thường chỉ cần kích hoạt khi file `.gitmodules` hoặc các commit pointer của submodule thay đổi; mỗi submodule có pipeline riêng xử lý test, build, và deploy độc lập.

## Chiến Lược Triển Khai: VPS Đơn và Phân Tán

DaPlex hỗ trợ hai topology triển khai khác nhau mà không cần thay đổi code, chỉ cần điều chỉnh biến môi trường. Sự linh hoạt này là hệ quả trực tiếp của quyết định kiến trúc: API và Transcoder giao tiếp với nhau **hoàn toàn qua Redis và HTTP**, không có direct in-process call nào.

Trong cấu hình **VPS đơn** (single-node), toàn bộ bốn thành phần — DaPlex-API, DaPlex-Transcoder, Redis, và MongoDB (nếu self-hosted) — chạy trên cùng một máy chủ. DaPlex-API lắng nghe trên port cấu hình bởi `PORT` (thường 3000), còn DaPlex-Transcoder bind vào `ADDRESS=127.0.0.1` và `PORT=3001`, nghĩa là Transcoder **không expose ra ngoài Internet** — chỉ có thể gọi vào từ API trên cùng máy. Ba URL Redis (`REDIS_URL`, `REDIS_2ND_URL`, `REDIS_QUEUE_URL`, `REDIS_IO_URL`) đều trỏ về `127.0.0.1:6379` nhưng dùng **database index khác nhau** (ví dụ `/0`, `/1`, `/3`) để tránh xung đột key. Cấu hình VPS đơn phù hợp cho giai đoạn phát triển và deployment quy mô nhỏ, tiêu biểu là VPS 2–4 vCPU, 4–8 GB RAM.

Trong cấu hình **phân tán** (distributed), DaPlex-Transcoder được tách ra chạy trên một máy chủ riêng với cấu hình CPU cao hơn (8–16 vCPU), trong khi API tiếp tục phục vụ trên VPS nhỏ hơn. Cấu hình này đòi hỏi ba thay đổi: (1) `REDIS_QUEUE_URL` trên cả API và Transcoder đều trỏ về địa chỉ Redis server trung tâm thay vì localhost; (2) `DAPLEX_API_DOMAINS` trên Transcoder được set thành địa chỉ IP hoặc hostname của API server (để whitelist các HTTP callback từ Transcoder về API); (3) biến `ADDRESS` trên Transcoder được đặt thành `0.0.0.0` nếu cần API server gọi ngược lại Transcoder qua HTTP. Transcoder kết nối trực tiếp với MongoDB Atlas (biến `DATABASE_URL` trong Transcoder `.env`) để cập nhật trạng thái `Media` document sau khi encode xong, không nhất thiết phải đi qua API REST — đây là một thiết kế đặc biệt sẽ được phân tích kỹ hơn ở mục 3.1.3.

Ngoài hai topology trên, kiến trúc còn hỗ trợ **multi-transcoder**: nhiều instance Transcoder cùng lắng nghe trên BullMQ queue. Biến `PRIMARY_TRANSCODER_URL` trong Transcoder `.env` cho phép chỉ định một Transcoder "chính" để các instance phụ có thể forward hoặc delegate job. BullMQ đảm bảo một job chỉ được nhận bởi một worker duy nhất (at-most-once delivery với `concurrency: 1` trên mỗi queue), vì vậy việc thêm Transcoder instance mới chỉ đơn giản là thêm subscriber vào cùng một Redis queue mà không cần thay đổi cấu hình API.

## Phân Vùng Dữ Liệu và Bốn Kết Nối Redis

Một điểm thiết kế đặc biệt trong DaPlex-API là sử dụng **bốn URL Redis riêng biệt**, mỗi URL phục vụ một mục đích cụ thể và được import qua module NestJS độc lập. Sự tách biệt này không chỉ là quy ước đặt tên mà là ranh giới kiến trúc thực sự: mỗi module có schema key riêng, TTL policy riêng, và có thể trỏ về Redis instance khác nhau trong cấu hình production phức tạp.

| Biến | Vai Trò | Ghi Chú |
|---|---|---|
| `REDIS_URL` | **Authentication state** — lưu refresh token, TTL 30 ngày | Redis "quan trọng nhất": mất dữ liệu đồng nghĩa toàn bộ user phải đăng nhập lại |
| `REDIS_2ND_URL` | **Application cache tầng hai** (`Redis2ndCacheModule`) — kết quả `findUserAuthGuard()`, rate limit counter, lookup nặng, TTL 300 giây | Có thể xóa bất kỳ lúc nào; hệ thống tự reload từ MongoDB |
| `REDIS_QUEUE_URL` | **BullMQ job queue + RedisPubSub** — encode job metadata, job state machine (waiting → active → completed/failed); cũng dùng làm pub/sub bus cho cancel signal (`video-cancel`) và nhận kết quả encode từ Transcoder | Chia sẻ với DaPlex-Transcoder; cả hai bên phải trỏ về cùng instance và database index |
| `REDIS_IO_URL` | **Socket.IO adapter** — pub/sub bus để broadcast WebSocket event qua nhiều API instance | Cần thiết khi triển khai multi-instance API sau load balancer; hiện tại chưa kích hoạt trong `main.ts` |

Cách tách bốn kết nối Redis thay vì dùng chung một connection pool có hai lợi ích cụ thể. Đầu tiên, mỗi kết nối có thể được tối ưu riêng về pool size và timeout theo đặc tính workload: BullMQ queue cần connection bền vững hơn với retry policy phức tạp hơn so với application cache có thể tolerate connection drop. Thứ hai, khi Redis bị quá tải hoặc cần maintenance, có thể scale từng database index lên Redis Cluster hoặc chuyển sang Redis instance riêng mà không ảnh hưởng đến các vai trò còn lại.

## Kết Nối Hai MongoDB và Phân Chia Tập Dữ Liệu

DaPlex-API cấu hình hai MongoDB connection riêng biệt thông qua biến `DATABASE_URL` (kết nối `DATABASE_A`) và `DATABASE_URL_B` (kết nối `DATABASE_B`). Cả hai thường trỏ về cùng một MongoDB Atlas cluster nhưng có thể cấu hình về hai cluster tách biệt nếu cần. Trong `AppModule`, hai connection được đăng ký với `MongooseModule.forRootAsync()` sử dụng `connectionName` khác nhau để NestJS phân biệt khi inject. Các model được khai báo thuộc kết nối nào cụ thể thông qua `MongooseModule.forFeature([...], 'DATABASE_A')` hoặc `MongooseModule.forFeature([...], 'DATABASE_B')`.

> Lý do phân tách hai database connection — thay vì dùng chung một — liên quan đến **phân vùng quan tâm theo domain**: một database lưu dữ liệu nghiệp vụ chính (media catalog, user profile, roles, permissions, collections, ratings, history), còn database kia lưu dữ liệu vận hành và audit (audit log, notifications, scan jobs). Sự tách biệt này cho phép áp dụng các retention policy khác nhau, backup frequency khác nhau, và có thể chuyển sang dedicated cluster cho từng loại workload nếu cần scale. Trong môi trường phát triển đơn giản, cả hai biến có thể trỏ về cùng một MongoDB instance và khác nhau ở tên database trong connection string (ví dụ `mongodb://localhost/daplex_main` và `mongodb://localhost/daplex_audit`).

DaPlex-Transcoder cũng có kết nối MongoDB độc lập với biến `DATABASE_URL` trong `.env` của riêng nó. Điều này cho phép Transcoder cập nhật trực tiếp trạng thái `Media` document (trường `status`, `streams`, `storageObjectId`) sau khi encode xong mà không cần gọi ngược về API endpoint.

> **Trade-off:** Thiết kế này là một trade-off có chủ đích: tốc độ và independence của Transcoder được ưu tiên hơn tính thuần khiết của kiến trúc "chỉ có API mới ghi database". Hệ quả là Transcoder cần có access credentials MongoDB giống API, và bất kỳ thay đổi schema nào trên model `Media` đều cần được cập nhật đồng thời trong cả hai codebase.

## Giao Tiếp Nội Bộ: API–Transcoder Protocol

Quan hệ giữa DaPlex-API và DaPlex-Transcoder được xây dựng trên ba kênh giao tiếp bất đồng bộ, mỗi kênh phục vụ một chiều data flow khác nhau. Hiểu rõ ba kênh này là tiền đề để nắm kiến trúc của cả hai thành phần.

**Kênh 1: API → Transcoder qua BullMQ queue.** Khi admin xác nhận một upload session, `MediaService` trong DaPlex-API tạo BullMQ job với data `IVideoData` (chứa file path, codec target, resolution list, media ID) và đẩy vào queue `${TaskQueue.VIDEO_TRANSCODE}:${VideoCodec}` trên Redis. DaPlex-Transcoder, đang chạy consumer `VideoCosumerH264` (hay H265/VP9/AV1 tương ứng), nhận job qua `WorkerHost.process()` và bắt đầu pipeline encode. BullMQ với `concurrency: 1` đảm bảo xử lý tuần tự — tại bất kỳ thời điểm nào, một queue chỉ xử lý một job, tránh tình trạng hai job cùng saturate CPU.

**Kênh 2: Transcoder → API qua HTTP callback.** Sau khi encode và upload segment lên cloud storage thành công, Transcoder gọi `PATCH /api/media/:id/movie/streams` (hoặc endpoint tương ứng) để cập nhật API về kết quả. `DAPLEX_API_DOMAINS` whitelist trong Transcoder `.env` kiểm soát domain nào được phép nhận callback này — một cơ chế bảo mật tối thiểu ngăn Transcoder bị exploit để gọi webhook tùy ý. Trong cấu hình phân tán, Transcoder cần có đường mạng tới API server; trong cấu hình VPS đơn thì callback đi qua loopback.

**Kênh 3: API → Transcoder qua Redis Pub/Sub.** Khi admin hủy một encode job đang chạy, API không thể gọi trực tiếp vào process của Transcoder (không có gRPC hay shared memory). Thay vào đó, API publish một message lên channel `video-cancel` của Redis. `VideoCancelModule` trong Transcoder subscribe kênh này, nhận signal hủy, và gọi `BullMQ.getJob(jobId).moveToFailed()` sau đó dừng FFmpeg process đang chạy. Đây là lý do `RedisPubSubModule` chia sẻ `REDIS_QUEUE_URL` với BullMQ — hai cơ chế (queue và pub/sub) dùng chung một Redis connection nhưng hoạt động hoàn toàn độc lập.

## Bảng Tóm Tắt Biến Môi Trường Quan Trọng

Để cụ thể hóa các quan hệ kiến trúc đã mô tả, bảng dưới đây liệt kê các biến môi trường cốt lõi và vai trò của chúng trong từng topology triển khai:

| Biến Môi Trường | Component | Vai Trò | VPS Đơn | Phân Tán |
|---|---|---|---|---|
| `DATABASE_URL` | API + Transcoder | MongoDB kết nối chính | cùng cluster | cùng cluster |
| `DATABASE_URL_B` | API | MongoDB kết nối phụ (audit, notifications) | cùng instance | cùng instance |
| `REDIS_URL` | API | Refresh token storage | `127.0.0.1:6379/0` | redis-server |
| `REDIS_2ND_URL` | API | Application cache (permission, rate-limit) | `127.0.0.1:6379/1` | redis-server |
| `REDIS_QUEUE_URL` | API + Transcoder | BullMQ encode queue + RedisPubSub cancel/result (dùng chung) | `127.0.0.1:6379/3` | redis-server |
| `REDIS_IO_URL` | API | Socket.IO adapter (chưa kích hoạt) | `127.0.0.1:6379/2` | redis-server |
| `DAPLEX_API_DOMAINS` | Transcoder | Whitelist HTTP callback target | `127.0.0.1,localhost` | IP của API server |

*Bảng 3.1: Biến môi trường phân vùng theo topology triển khai — VPS đơn dùng loopback, phân tán dùng địa chỉ network*

Bảng 3.1 cho thấy rằng chuyển đổi từ VPS đơn sang phân tán về cơ bản là bài toán **network configuration**, không phải bài toán code. Ba biến có sự khác biệt quan trọng nhất là `REDIS_QUEUE_URL` (phải đồng bộ giữa API và Transcoder), `ADDRESS` (Transcoder từ loopback sang public interface), và `DAPLEX_API_DOMAINS` (từ localhost sang địa chỉ thực). Tất cả các biến còn lại đều là application config độc lập với topology.
