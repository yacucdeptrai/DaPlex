# 3.1.2. Kiến Trúc Backend API

Mục này mô tả kiến trúc nội bộ của DaPlex-API — từ điểm khởi động ứng dụng, cấu trúc phân cấp module NestJS, đến các tầng common modules dùng chung. Hiểu kiến trúc này giúp nắm cách một request HTTP đi qua hệ thống và tại sao mỗi thành phần được tách thành module độc lập. Nội dung gồm: Fastify adapter và cấu hình khởi động, ValidationPipe, AppModule, AppRoutingModule với 17 resource module, các common module, và AppSocketModule.

---

## Khởi Động Ứng Dụng và Fastify Adapter

Điểm khởi đầu của DaPlex-API là file `main.ts`, nơi NestJS application được khởi tạo với `NestFactory.create<NestFastifyApplication>(AppModule, new FastifyAdapter(...))` thay vì lời gọi `NestFactory.create()` thông thường vốn dùng Express.

> Quyết định thay thế Express bằng Fastify là lựa chọn kiến trúc có tính toán: Fastify được thiết kế từ đầu với focus vào throughput JSON và overhead thấp trên từng request, benchmark nội bộ của NestJS cho thấy Fastify đạt thông lượng cao hơn Express khoảng 2–3 lần trong workload JSON-heavy. DaPlex-API là ứng dụng hầu như chỉ xử lý JSON (metadata CRUD, authentication flow, job management) nên đây là trade-off hợp lý: đổi lấy một số plugin không tương thích Express để được throughput cao hơn.

`FastifyAdapter` được khởi tạo với hai tham số cấu hình quan trọng:

| Tham Số | Giá Trị | Mục Đích |
|---|---|---|
| `logger` | `{ level: isDev ? 'info' : 'warn' }` | Giảm log verbosity từ `info` xuống `warn` ở production để tránh I/O overhead do Fastify/Pino logging |
| `trustProxy` | `process.env.TRUST_PROXY === 'true'` | Tin tưởng `X-Forwarded-For` khi API chạy sau nginx/Cloudflare, cho phép rate limiter lấy IP thực; `false` cho VPS đơn, `true` cho production sau reverse proxy |

Một detail nhỏ nhưng quan trọng trong `main.ts` là block DNS override chỉ chạy trong development:

```typescript
if (isDev) {
  dns.setServers(['1.1.1.1:53', '8.8.8.8:53']);
}
```

MongoDB Atlas dùng SRV records cho connection string dạng `mongodb+srv://...`. Trên một số môi trường Windows local, DNS resolver mặc định có thể thất bại khi tra cứu SRV record của Atlas domain. Thay vì dùng connection string thuần IP (sẽ mất tính năng load balancing của Atlas), DaPlex chọn override DNS server về Cloudflare (`1.1.1.1`) và Google (`8.8.8.8`) — hai DNS public có SRV record support đáng tin cậy. Block này bị bỏ qua ở production vì production server thường có DNS được cấu hình đúng.

Sau khi khởi tạo application instance, `main.ts` thực hiện tuần tự các bước setup trước khi gọi `app.listen()`: đăng ký Swagger UI (chỉ trong `isDev`), cấu hình `ValidationPipe` global, bật CORS, đăng ký Fastify plugin `@fastify/multipart` và `@fastify/cookie`, sau đó wire class-validator vào DI container. Trình tự này có ý nghĩa: nếu Swagger được setup sau khi route đã mounted, có thể bỏ sót một số route; nếu ValidationPipe được đăng ký sau CORS, request không hợp lệ vẫn bị block đúng cách nhưng response sẽ không có CORS header.

## ValidationPipe và Chiến Lược Xử Lý Lỗi

`ValidationPipe` được cấu hình với hai flag: `whitelist: true` (tự động loại bỏ bất kỳ property nào không được khai báo trong DTO, ngăn mass assignment) và `stopAtFirstError: true` (dừng validation ngay sau lỗi đầu tiên thay vì thu thập tất cả lỗi).

> **Tại sao `stopAtFirstError`?** `stopAtFirstError` là lựa chọn phù hợp cho REST API hướng client vì client thường hiển thị từng lỗi một — thu thập toàn bộ lỗi sẽ tốn thêm compute mà không cải thiện UX.

Điểm đặc biệt là `exceptionFactory` tùy chỉnh thay vì dùng format lỗi mặc định của NestJS:

```typescript
exceptionFactory: (errors: ValidationError[]) => {
  let error = errors[0];
  while (!error?.constraints && error?.children.length) {
    error = error.children[0];
  }
  return new BadRequestException({
    code: Object.values<any>(error?.contexts || {})[0]?.code || -1,
    message: Object.values<any>(error?.constraints)[0] || ''
  });
}
```

Logic này duyệt xuống cây lỗi nested (khi DTO có object lồng nhau), lấy constraint đầu tiên bị vi phạm, và extract `code` từ `contexts` — một field tùy chỉnh mà validator decorator có thể inject. Kết quả là error response luôn có cấu trúc `{ code: number, message: string }` nhất quán, thay vì object `ValidationError` nhiều field mà class-validator trả về mặc định. Mã lỗi số (`code`) cho phép frontend phân biệt loại lỗi mà không cần parse chuỗi message.

## AppModule: Module Gốc và Cấu Hình Toàn Cục

`AppModule` là root module của ứng dụng — NestJS sẽ dùng nó như điểm khởi đầu để xây dựng toàn bộ dependency injection graph. Trong kiến trúc của DaPlex-API, `AppModule` giữ vai trò đặc biệt: nó **không chứa business logic** mà chỉ là nơi đăng ký các infrastructure module cấp hệ thống và hai module routing.

`ConfigModule.forRoot({ isGlobal: true, cache: true, validate })` là module được import đầu tiên và quan trọng nhất. `isGlobal: true` có nghĩa là `ConfigService` có thể được inject vào bất kỳ module nào mà không cần import `ConfigModule` lại ở cấp module con — đây là pattern chuẩn cho configuration trong NestJS monolith. `cache: true` khiến `ConfigService.get()` lưu kết quả vào bộ nhớ sau lần đọc đầu tiên, tránh phải đọc `process.env` nhiều lần. `validate` là hàm được import từ `env-validation.dto.ts`, sử dụng `class-validator` để kiểm tra toàn bộ biến môi trường **ngay khi application khởi động** — nếu thiếu bất kỳ biến bắt buộc nào (ví dụ `DATABASE_URL`, `REDIS_URL`, `COOKIE_DOMAIN`), app sẽ crash với error rõ ràng thay vì bị lỗi ngầm khi runtime.

Hai `MongooseModule.forRootAsync()` tiếp theo đăng ký hai kết nối MongoDB với `connectionName` tương ứng là `MongooseConnection.DATABASE_A` và `MongooseConnection.DATABASE_B`. Cả hai connection đều cấu hình `family: 4` (ép ioredis và MongoDB driver dùng IPv4, tránh lỗi DNS với IPv6 trên một số VPS) và `useBigInt64: true` (cho phép MongoDB driver serialize `Long` BSON type thành JavaScript `BigInt` thay vì `number`, tránh overflow khi xử lý các ID hoặc counter lớn). Hàm `applyBigIntPatches()` được gọi ở cấp module level trong `main.ts` hoàn thiện pipeline này bằng cách patch `JSON.stringify()` để serialize `BigInt` thành string thay vì throw error — vì `JSON.stringify(BigInt(1))` ném exception theo mặc định trong JavaScript.

`BullModule.forRootAsync()` setup global BullMQ connection từ `REDIS_QUEUE_URL`. Thay vì truyền connection string trực tiếp, code dùng `parseRedisUrl(url)` để tách riêng `host`, `port`, `password`, và `database` index, sau đó truyền vào `ioredis` connection object. Điều này cần thiết vì `ioredis` (client Redis mà BullMQ dùng) không phải lúc nào cũng parse đúng tất cả format URL Redis, đặc biệt khi URL có password hoặc database index (e.g. `redis://:password@host:6379/3`).

> Flag `enableOfflineQueue: false` có ý nghĩa quan trọng: khi Redis không khả dụng, BullMQ sẽ **từ chối các job mới ngay lập tức** (throw error) thay vì giữ chúng trong queue bộ nhớ vô thời hạn. Đây là fail-fast behavior: admin sẽ biết ngay rằng encode pipeline bị gián đoạn, thay vì job bị treo im lặng.

`ScheduleModule.forRoot()` kích hoạt NestJS Cron scheduler, sử dụng `@Cron()` decorator ở các service cần tác vụ định kỳ. Trong DaPlex-API, scheduler được dùng cho các tác vụ như dọn dẹp token hết hạn, đồng bộ metadata từ TMDB, và kiểm tra trạng thái encode job bị stale.

## AppRoutingModule: 17 Resource Modules

`AppRoutingModule` là module định nghĩa toàn bộ bề mặt REST API của DaPlex-API. Thay vì đăng ký route trực tiếp trong `AppModule`, DaPlex dùng `RouterModule.register(routes)` của NestJS core — một pattern cho phép cấu hình **prefix URL theo từng module** một cách declarative. Toàn bộ 17 resource module được mount dưới prefix `/api`, tạo nên không gian URL có cấu trúc rõ ràng.

| Prefix Route | Module | Chức Năng Chính |
|---|---|---|
| `/api/auth` | `AuthModule` | Đăng nhập, đăng ký, refresh token, xác thực email |
| `/api/users` | `UsersModule` | Quản lý profile người dùng, tìm kiếm, ban/unban |
| `/api/roles` | `RolesModule` | CRUD vai trò, phân quyền bitfield |
| `/api/media` | `MediaModule` | CRUD nội dung, upload session, encode job management |
| `/api/genres` | `GenresModule` | Thể loại nội dung |
| `/api/productions` | `ProductionsModule` | Hãng sản xuất, studio |
| `/api/collections` | `CollectionModule` | Playlist do admin tạo (featured, trending...) |
| `/api/tags` | `TagsModule` | Tag tùy chỉnh cho media |
| `/api/ratings` | `RatingsModule` | Đánh giá và review của người dùng |
| `/api/history` | `HistoryModule` | Lịch sử xem, resume playback position |
| `/api/playlists` | `PlaylistsModule` | Playlist do người dùng tạo |
| `/api/notification` | `NotificationModule` | Thông báo trong app và push notification |
| `/api/settings` | `SettingsModule` | Cấu hình hệ thống, storage provider |
| `/api/external-storages` | `ExternalStoragesModule` | Quản lý kết nối cloud storage (Azure, R2, S3...) |
| `/api/audit-log` | `AuditLogModule` | Lịch sử hành động admin |
| `/api/media-scanner` | `MediaScannerModule` | Quét metadata tự động từ TMDB/TVDB |

*Bảng 3.2: 17 resource module trong AppRoutingModule và prefix URL tương ứng*

Cách tổ chức module trong `AppRoutingModule` tuân theo nguyên tắc **module = domain boundary**: mỗi module chứa controller, service, schema, và DTO riêng của domain đó. Một media không bao giờ gọi trực tiếp vào service của history hay notification — thay vào đó chúng giao tiếp qua event hoặc import module cụ thể với forwardRef khi cần tránh circular dependency. Pattern này giảm coupling giữa các domain và cho phép test từng domain độc lập.

## Common Modules: Tầng Hạ Tầng Dùng Chung

Nằm trong thư mục `src/common/modules/`, các common module là các building block có thể tái sử dụng được import vào các resource module khi cần, tương tự shared library trong kiến trúc microservice. Có thể phân nhóm chúng theo chức năng.

**Nhóm Redis.** `RedisCacheModule` wrap `REDIS_URL` với `@tirke/node-cache-manager-ioredis` adapter và expose `RedisCacheService` — được dùng chủ yếu bởi `AuthModule` để lưu và đọc refresh token. `Redis2ndCacheModule` wrap `REDIS_2ND_URL` tương tự nhưng phục vụ application cache tầng hai — kết quả `findUserAuthGuard()` (user profile + permission array), rate limit counter, và lookup nặng khác được lưu ở đây với TTL ngắn hơn (5–10 phút) so với refresh token (30 ngày). Việc tách thành hai module riêng biệt thay vì dùng chung một `ioredis` client có lợi ích: nếu second-level cache Redis bị overload hoặc cần flush, có thể restart `Redis2ndCacheModule` mà không ảnh hưởng đến auth session đang hoạt động.

`RedisPubSubModule` là module dynamic (dynamic module pattern của NestJS) với hai factory method: `register()` và `registerAsync()`. Module này sử dụng `DiscoveryModule` của NestJS để scan toàn bộ provider đang registered và tìm những method được đánh dấu bằng decorator `@OnPubSubMessage(channel)` — cơ chế này tương tự `@EventEmitter()` nhưng thông qua Redis Pub/Sub. `VideoCancelModule` trong Transcoder dùng cơ chế tương tự để lắng nghe channel `video-cancel`. `LocalCacheModule` đơn giản hơn: dùng `cache-manager` với store in-memory, phù hợp cho các lookup tần suất cao nhưng dữ liệu tương đối tĩnh (ví dụ: danh sách genre, chapter type) mà không cần round-trip Redis.

**Nhóm Infrastructure.** `PermissionsModule` chỉ export `PermissionsService` với hai method: `scanPermission()` (trả về mảng `UserPermission[]` từ tập roles của user) và `hasPermission()` (bitwise check trên permission field của từng role). Module này có dependency rất nhẹ và được import rộng rãi — `AuthModule`, `UsersModule`, `RolesModule` đều cần nó. `FilerModule` là abstraction layer cho toàn bộ tác vụ liên quan đến file: upload vào storage provider được cấu hình, generate signed URL, delete file. `FilerService` inject `SettingsModule` (để đọc storage provider đang active) và `ExternalStoragesModule` (để lấy credentials của provider đó), sử dụng `forwardRef()` để giải quyết circular dependency vì `ExternalStoragesModule` cũng import `FilerModule`.

**Nhóm Storage Adapters.** DaPlex hỗ trợ đa dạng cloud storage provider thông qua các module chuyên biệt: `AzureBlobModule` (Azure Blob Storage + Azure CDN), `CloudflareR2Module` (Cloudflare R2 S3-compatible), `S3Module` (AWS S3 generic), `OneDriveModule`, `GoogleDriveModule`, và `DropboxModule`. Ngoài ra còn có `ImgurModule` và `ImagekitModule` cho image hosting, `YoutubeModule` cho external stream source. Mỗi module wrap SDK của provider tương ứng và expose một service interface nhất quán — `FilerService` sẽ delegate đến đúng adapter tùy theo provider được cấu hình trong `Settings` document.

**Nhóm Metadata Scanner.** `TmdbScannerModule` gọi TMDB API (The Movie Database) để lấy metadata phim và series: poster, backdrop, plot, cast, rating, năm sản xuất. `TvdbScannerModule` làm tương tự cho TVDB (The TV Database), tập trung vào series và episode metadata. Cả hai dùng `HttpModule` của NestJS (wrap `axios`) và được import bởi `MediaScannerModule` — resource module expose `/api/media-scanner` endpoint để admin trigger scan thủ công hoặc tự động.

## AppSocketModule và WsAdminGateway

`AppSocketModule` chỉ import một module duy nhất: `WsAdminModule`.

> Quyết định tách WebSocket ra module riêng thay vì đặt vào `AppRoutingModule` phản ánh sự khác biệt về protocol: WebSocket không phải HTTP và không phù hợp với `RouterModule.register()` dùng cho REST. `AppSocketModule` là điểm mount rõ ràng cho tất cả WebSocket gateway hiện tại và tương lai.

`WsAdminGateway` là Socket.IO gateway duy nhất trong hệ thống, định nghĩa namespace `ws/admin`. Decorator `@WebSocketGateway({ namespace: 'ws/admin', cors: { origin: '*' }, transports: ['websocket'] })` cấu hình ba điểm quan trọng: namespace phân biệt kết nối admin khỏi WebSocket namespace khác nếu có; `cors: { origin: '*' }` cho phép kết nối từ mọi origin (khác với CORS của REST được giới hạn bởi `ORIGIN_URL`); `transports: ['websocket']` tắt long-polling fallback, ép client phải dùng WebSocket thuần — điều này cần thiết cho Socket.IO Redis adapter khi scale multi-instance.

Xác thực WebSocket trong `handleConnection()` khác với xác thực REST. Client gửi access token qua `socket.handshake.auth['token']` (không phải HTTP header), và gateway gọi `authService.verifyAccessToken(accessToken)` — phương thức này chỉ verify JWT signature và expiry, không check Redis, không check `banned` flag. Nếu token hợp lệ, client được join vào room `user_id:${user._id}` tự động; nếu không, gateway gọi `client.disconnect(true)` để đóng kết nối ngay lập tức. Room `user_id:${user._id}` là cơ chế target push notification đến đúng user khi có encode job update, thay vì broadcast đến mọi client.

Gateway cũng expose hai message handler: `room:join` và `room:leave` cho phép client admin đăng ký nhận update của job cụ thể (ví dụ: join room `job:${jobId}` khi mở màn hình quản lý encode, leave room khi đóng màn hình). Cơ chế này tránh việc mỗi client admin nhận mọi encode event — chỉ nhận event của những job mà admin đang theo dõi.

Đáng chú ý là Socket.IO Redis adapter (cho phép WebSocket event được broadcast qua nhiều API instance) hiện đang bị comment out trong `main.ts`:

```typescript
// const redisIoAdapter = new RedisIoAdapter(app);
// await redisIoAdapter.connectToRedis();
// app.useWebSocketAdapter(redisIoAdapter);
```

Điều này có nghĩa là ở trạng thái hiện tại, `REDIS_IO_URL` chưa được sử dụng cho WebSocket scaling — chỉ cần thiết khi triển khai multi-instance API sau load balancer. Trong deployment VPS đơn, Socket.IO dùng in-process event emitter mặc định là đủ.

## Tổng Hợp: Luồng Request Từ Đầu Đến Cuối

Để hình dung toàn bộ tầng backend API hoạt động, xem xét luồng một request `GET /api/media/:id` từ khi đến server đến khi trả response:

Fastify nhận request HTTP, gọi middleware chain (CORS check → cookie parse → auth header extract). `AuthGuard` được trigger đầu tiên (nếu route có `@UseGuards(AuthGuard)`): gọi `authService.verifyAccessToken()` → sau đó `findUserAuthGuard()` (check Redis2ndCache trước, nếu miss thì query MongoDB + chạy `scanPermission()`). Nếu user bị ban, trả 403. Sau khi `request.user` được set, `RolesGuard` (nếu route có `@RolesGuardOptions()`) kiểm tra `user.granted` array với `includes()` và `some()`. Request hợp lệ được dispatch vào `MediaController.findOne()`, controller gọi `MediaService.findById()`, service truy vấn MongoDB qua Mongoose model thuộc connection `DATABASE_A`. Response được serialize thành JSON bởi Fastify (với BigInt → string patch), thêm CORS header, trả về client.

Luồng này diễn ra trong dưới 20ms cho một cache hit (Redis2ndCache trả kết quả findUserAuthGuard), hoặc dưới 50ms khi cần query MongoDB. Fastify JSON serialization nhanh hơn Express nhờ schema-based serialization thay vì dynamic stringify — đây là lợi ích throughput thực tế của việc chọn Fastify ở tầng framework.
