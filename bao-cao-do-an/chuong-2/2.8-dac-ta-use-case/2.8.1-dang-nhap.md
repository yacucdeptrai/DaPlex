# 2.8.1. UC-U-02 — Đăng Nhập và Quản Lý Phiên

| Thuộc tính | Giá trị |
|---|---|
| **Mã Use Case** | UC-U-02 |
| **Tên** | Đăng nhập và quản lý phiên |
| **Tác nhân chính** | Người dùng thông thường, Quản trị viên |
| **Điều kiện tiên quyết** | Tài khoản đã tồn tại và đã xác thực email (`verified: true`) |

| **Hậu điều kiện (thất bại)** | Không có token nào được tạo; trạng thái hệ thống không thay đổi |

---

## Mô Tả Tổng Quan

Use Case này bao phủ toàn bộ vòng đời quản lý phiên đăng nhập của DaPlex: từ xác thực credentials ban đầu, phát hành token, tự động gia hạn phiên khi access token hết hạn, đến kết thúc phiên chủ động. Cơ chế phiên của DaPlex dựa trên mô hình **dual-token**: access token ngắn hạn cho từng request và refresh token dài hạn được lưu trong Redis để gia hạn phiên.

---

## Luồng Chính — Đăng Nhập

**Bước 1:** Người dùng gửi `POST /auth/sign-in` với body `{ email, password }` qua giao diện Angular.

**Bước 2:** `AuthController` nhận request và chuyển đến `AuthService.authenticate()`. Service tra cứu user document trong MongoDB theo `email` với projection đầy đủ, bao gồm `populate('roles')` để lấy thông tin vai trò.

**Bước 3:** `AuthService` so khớp mật khẩu bằng `bcrypt.compare()` với hash đã lưu trong user document. Nếu không khớp → ném `HttpException 401` với `StatusCode.INCORRECT_PASSWORD`.

**Bước 4:** Nếu mật khẩu đúng, `createJwtToken(user)` được gọi, thực hiện song song hai thao tác:
- Tạo **access token** bằng `jwtService.signAsync({ _id: user._id.toString() }, { secret: ACCESS_TOKEN_SECRET, expiresIn: ACCESS_TOKEN_EXPIRY })` — JWT HS256 với payload chứa duy nhất `_id` của user.
- Tạo **refresh token** bằng `nanoid(32)` — chuỗi ngẫu nhiên 32 ký tự, **không phải JWT**. Refresh token được lưu vào Redis (qua `RedisCacheService`) tại key `REFRESH_TOKEN:{token}` với value `{ _id: user._id }` và TTL bằng `REFRESH_TOKEN_EXPIRY`.

**Bước 5:** API trả về đối tượng `Jwt` chứa access token, refresh token, thời hạn, và `UserDetails` (thông tin user đã được `plainToInstance` serialize). `AuthController` đặt refresh token vào HTTP-only cookie với `httpOnly: true`, `sameSite: 'strict'`, domain kiểm soát bởi `COOKIE_DOMAIN`.

**Bước 6:** Angular HTTP interceptor lưu access token vào bộ nhớ in-memory (không lưu localStorage để tránh XSS), đính kèm vào header `Authorization` của mọi request tiếp theo.

---

## Luồng Tự Động Gia Hạn Phiên (Refresh)

**Điều kiện kích hoạt:** API trả về 401 cho một request có access token đã hết hạn (sau `ACCESS_TOKEN_EXPIRY` giây).

**Bước 1:** Angular HTTP interceptor bắt 401 response và tự động gọi `POST /auth/refresh-token`, trình duyệt tự đính kèm refresh token cookie.

**Bước 2:** `AuthService.refreshToken()` đọc refresh token từ cookie, tra cứu key `REFRESH_TOKEN:{token}` trong Redis. Nếu key không tồn tại (đã hết hạn hoặc bị revoke) → 401 `TOKEN_REVOKED`.

**Bước 3:** Nếu key tồn tại, Redis key cũ được đặt TTL về 60 giây (grace period để xử lý đồng thời nhiều request refresh). Service tra cứu user document mới nhất từ MongoDB (`findUserById` với `includeRoles: true`), sau đó gọi `createJwtToken()` tạo cặp token mới — thực hiện **token rotation**: refresh token cũ sẽ hết hạn sau 60 giây, refresh token mới được tạo với TTL đầy đủ.

**Bước 4:** Angular interceptor nhận token mới, retry request gốc bị 401, tiếp tục luồng xử lý bình thường mà người dùng không hay biết.

---

## Luồng Xác Thực Mỗi Request (AuthGuard)

Mọi endpoint yêu cầu xác thực đều đi qua `AuthGuard` trước khi vào handler:

**Bước 1:** `AuthGuard.canActivate()` đọc access token từ `request.headers.authorization`.

**Bước 2:** `AuthService.verifyAccessToken()` xác minh chữ ký JWT với `ACCESS_TOKEN_SECRET`. Nếu token hết hạn hoặc chữ ký sai → 401.

**Bước 3:** `AuthService.findUserAuthGuard(userId)` được gọi, sử dụng `Redis2ndCacheService.wrap()` với TTL 300 giây để cache kết quả. Lần đầu (cache miss): truy vấn MongoDB với `findOneAndUpdate` để cập nhật `lastActiveAt` đồng thời lấy dữ liệu user. Lần sau (cache hit): trả về từ Redis trong vài millisecond.

**Bước 4:** `permissionsService.scanPermission(authUser)` được gọi bên trong `findUserAuthGuard()` sau khi dữ liệu user được truy vấn từ MongoDB. Hàm duyệt qua danh sách `UserPermission` enum, kiểm tra từng quyền xem `user.roles` có role nào chứa bit tương ứng không, và trả về mảng `UserPermission[]` gồm các quyền user có. Kết quả này được gán vào `authUser.granted` và được `RolesGuard` đọc ở bước kiểm tra quyền tiếp theo.

**Bước 5:** Nếu `user.banned === true` → 403 `USER_BANNED`, từ chối ngay lập tức bất kể token còn hợp lệ.

---

## Luồng Đăng Xuất

**Bước 1:** Người dùng chọn đăng xuất trên Angular, app gửi `POST /auth/revoke-token` kèm refresh token cookie.

**Bước 2:** `AuthService.revokeToken()` xóa key `REFRESH_TOKEN:{token}` khỏi Redis. Access token hiện tại vẫn hợp lệ cho đến khi hết hạn tự nhiên (`ACCESS_TOKEN_EXPIRY` giây), nhưng không thể gia hạn thêm.

**Bước 3:** Angular xóa access token khỏi bộ nhớ, chuyển hướng về trang đăng nhập.

---

## Luồng Thay Thế và Ngoại Lệ

**[2a] Email không tồn tại:** `AuthService.authenticate()` không tìm thấy user document → 401 `EMAIL_NOT_EXIST`. Không tiết lộ email có tồn tại hay không trong message (response code đủ để phân biệt nhưng không đủ cho enumeration attack).

**[3a] Mật khẩu sai:** `bcrypt.compare()` trả về false → 401 `INCORRECT_PASSWORD`.

**[Refresh-2a] Refresh token hết hạn hoặc bị xóa:** Key không tồn tại trong Redis → 401 `TOKEN_REVOKED`. Angular interceptor không retry mà chuyển hướng người dùng đến trang đăng nhập.

**[AuthGuard-2a] Token giả mạo hoặc hỏng:** `jwtService.verifyAsync` ném exception → `AuthGuard` bắt và ném 401 `UNAUTHORIZED`. Exception gốc được logger ghi lại nhưng không expose ra client.

**[AuthGuard-3a] User bị xóa khỏi DB:** `findUserAuthGuard` trả về null sau cache miss → 401 `UNAUTHORIZED_NO_USER`. Cache hit có thể che khuất trường hợp này trong tối đa 300 giây.

> Điểm thiết kế đáng chú ý trong Use Case này là **refresh token không phải JWT** mà là chuỗi ngẫu nhiên `nanoid(32)` được lưu trong Redis. Cách thiết kế này cho phép **revoke tức thì**: xóa key Redis là đủ để vô hiệu hóa phiên ngay lập tức, không cần chờ JWT hết hạn. Đây là lựa chọn thiết thực cho hệ thống self-hosted nơi quản trị viên có thể cần ngắt phiên của một tài khoản cụ thể ngay lập tức. Đổi lại, mỗi refresh yêu cầu một round-trip đến Redis — đây là chi phí chấp nhận được vì refresh chỉ xảy ra mỗi `ACCESS_TOKEN_EXPIRY` giây (5 phút theo cấu hình mặc định), không phải mỗi request.
