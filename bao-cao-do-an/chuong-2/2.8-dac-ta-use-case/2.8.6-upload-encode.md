# 2.8.6. UC-A-02 — Upload File Nguồn và Kích Hoạt Encode

| Thuộc tính | Giá trị |
|---|---|
| **Mã Use Case** | UC-A-02 |
| **Tên** | Upload file nguồn và kích hoạt encode |
| **Tác nhân chính** | Quản trị viên (quyền `MANAGE_MEDIA`); DaPlex-Transcoder (tác nhân hệ thống) |
| **Điều kiện tiên quyết** | Media document đã tồn tại; ít nhất một `ExternalStorage` đã cấu hình |
| **Hậu điều kiện (thành công)** | DASH segments và MPD file trên cloud storage; Media document cập nhật trạng thái `completed`; stream URL có thể truy cập |
| **Hậu điều kiện (thất bại)** | Job được đánh dấu `failed` trong BullMQ; Media document cập nhật trạng thái tương ứng; file tạm được dọn |

---

## Mô Tả Tổng Quan

Đây là Use Case phức tạp nhất trong hệ thống, kéo dài qua hai service (API và Transcoder), ba cơ chế giao tiếp (HTTP, BullMQ, Redis Pub/Sub), và có thể mất từ vài phút đến hàng giờ tùy kích thước file và codec. Use Case bắt đầu đồng bộ (HTTP upload) và chuyển sang hoàn toàn bất đồng bộ sau khi Job được đẩy vào hàng đợi BullMQ.

---

## Luồng Chính — Upload File

**Bước 1:** Quản trị viên chọn file video nguồn trên giao diện Angular và bắt đầu upload. Angular gọi `POST /media/:id/movie/source` (phim lẻ) hoặc `POST /media/:id/tv/episodes/:episode_id/source` (tập series) dưới dạng multipart upload theo từng chunk. API nhận chunk và ghi vào bộ nhớ tạm phía server.

**Bước 2:** API tạo một `UploadSession` document để theo dõi tiến trình upload. Mỗi chunk nhận được được ghi nhận, giúp có thể tiếp tục nếu kết nối bị gián đoạn. `session_id` được trả về cho client sau chunk đầu tiên.

**Bước 3:** Khi toàn bộ file đã upload xong, quản trị viên xác nhận bằng cách gọi `POST /media/:id/movie/source/:session_id` (endpoint confirm) kèm tham số encode: `{ codec, storageId, encodingSettingId }`. Tham số này chỉ định codec đích (H264/H265/VP9/AV1), `ExternalStorage` provider để upload kết quả, và bộ cài đặt encode (`EncodingSetting` document).

**Bước 4:** API gọi `MediaService` để đẩy job vào BullMQ queue tương ứng với codec: `h264`, `h265`, `vp9`, hoặc `av1`. Job payload chứa đường dẫn file nguồn tạm, `storageId`, `mediaId`, `episodeId`, và các tùy chọn encode. Media document được cập nhật trạng thái `processing`.

---

## Luồng Chính — Encode (DaPlex-Transcoder)

**Bước 5:** Consumer tương ứng trong DaPlex-Transcoder (`VideoCosumerH264`, `VideoCosumerH265`, `VideoCosumerVP9`, hoặc `VideoCosumerAV1`) nhận job từ BullMQ queue. Mỗi consumer kế thừa abstract class `BaseVideoConsumer` (vốn kế thừa từ `WorkerHost` của BullMQ) và định nghĩa phương thức `process(job)` — phương thức này gọi `this.videoService.transcode(job, VideoCodec.H264)` (hoặc codec tương ứng) để bắt đầu pipeline encode.

**Bước 6:** `VideoService.transcode(job, codec)` bắt đầu pipeline encode gồm bốn giai đoạn:

- **Giai đoạn 1 — Tải file nguồn:** `RcloneHelper.download()` tải file từ bộ nhớ tạm API về máy Transcoder (hoặc từ linked source trên cloud storage nếu là `linked-source`). Rclone được dùng thay vì HTTP download trực tiếp để hỗ trợ nhiều protocol và provider.

- **Giai đoạn 2 — Encode với FFmpeg:** FFmpeg được gọi với tham số tương ứng codec. Ví dụ H.264: `ffmpeg -i input -c:v libx264 -preset slow -crf 18 -c:a libfdk_aac -vbr 5 -ac 2 output_[resolution].mp4` cho mỗi resolution trong bộ cài đặt. Nhiều resolution được encode tuần tự hoặc song song tùy cấu hình. Tiến trình FFmpeg được parse từ stderr để tính `progress` percentage.

- **Giai đoạn 3 — Đóng gói DASH với MP4Box:** `MP4Box -dash 6000 -profile onDemand -out manifest.mpd [input files...]` tạo MPD manifest kiểu `SegmentBase` và các MP4 fragment. Flag `-profile onDemand` tạo self-contained MP4 files với `moov` box ở đầu file, phù hợp cho progressive download và DASH.

- **Giai đoạn 4 — Upload lên Cloud Storage:** `RcloneHelper.upload()` đẩy toàn bộ DASH artifacts (MPD và các MP4 segment) lên `ExternalStorage` đã chỉ định. Rclone tự động chọn protocol upload phù hợp với provider (S3 API, Google Drive API, WebDAV, v.v.).

**Bước 7:** Trong suốt quá trình encode, Consumer phát `BullMQ job progress events` với `progress` (0–100%), `stage` (download/encode/package/upload), và thông tin chi tiết giai đoạn hiện tại. DaPlex-API lắng nghe events này qua `MediaConsumer` và đẩy đến client qua WebSocket (`ws/admin` namespace).

**Bước 8:** Khi upload hoàn tất, `DaplexApiService` trong Transcoder gọi API endpoint để cập nhật Media document với: đường dẫn MPD trên cloud storage, codec, resolution list, và trạng thái `completed`. File tạm trên máy Transcoder được xóa.

---

## Luồng Hủy Job

**Điều kiện kích hoạt:** Quản trị viên bấm nút Hủy trên giao diện admin trong khi job đang chạy.

**Bước 1:** Angular gọi API endpoint hủy job. API publish message `{ jobId }` lên Redis Pub/Sub channel `video-cancel`.

**Bước 2:** `DaPlex-Transcoder` đang lắng nghe channel `video-cancel`. Khi nhận message, nó gọi `VideoService.cancel(jobId)` để dừng FFmpeg process hiện tại (kill process) và dừng upload đang chạy.

**Bước 3:** Job được đánh dấu `cancelled` trong BullMQ. Media document được cập nhật trạng thái `cancelled`. File tạm được dọn.

---

## Luồng Upload Linked Source

Thay vì upload file mới, quản trị viên có thể đăng ký file đã có trên cloud storage qua `POST /media/:id/movie/linked-source` với `{ storageId, path }`. API xác minh file tồn tại tại đường dẫn đã cung cấp, sau đó tạo Job với path này thay vì path file tạm.

---

## Luồng Thay Thế và Ngoại Lệ

**[3a] File nguồn hỏng (corrupt):** FFmpeg báo lỗi tại giai đoạn encode → job được đánh dấu `failed` với error message. `DaplexApiService` cập nhật Media document trạng thái `failed`. BullMQ retry theo cấu hình backoff (job failed 3 lần → moved to failed queue, không retry thêm).

**[4a] Cloud storage đầy hoặc lỗi kết nối:** Rclone upload thất bại → exception được throw lên Consumer → job `failed` với retry. Quản trị viên cần kiểm tra cấu hình storage.

**[6-encode] Hết dung lượng đĩa Transcoder:** FFmpeg không thể ghi output → exception → job `failed`. Cần theo dõi disk usage trên máy Transcoder.

> Use Case này minh họa rõ nhất nguyên tắc thiết kế cốt lõi của DaPlex: **tách biệt orchestration (API) và execution (Transcoder)**. API không thực thi encode — nó chỉ nhận yêu cầu, tạo job, và theo dõi tiến trình. Transcoder không nhận lệnh trực tiếp từ HTTP — nó chỉ lắng nghe hàng đợi. Hai service giao tiếp qua ba kênh bất đồng bộ (BullMQ job, BullMQ events, Redis Pub/Sub), tạo ra một kiến trúc loose-coupled có thể scale độc lập và chịu lỗi tốt hơn so với kiến trúc RPC đồng bộ.
