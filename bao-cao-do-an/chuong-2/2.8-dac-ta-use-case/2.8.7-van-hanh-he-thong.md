# 2.8.7. UC-A-07 đến UC-A-10 — Vận Hành Hệ Thống

Nhóm use case này mô tả các tác vụ vận hành dành cho quản trị viên hệ thống. Bốn use case bao gồm: quản lý tài khoản người dùng, quản lý vai trò và phân quyền, cấu hình cloud storage provider, và xem nhật ký kiểm toán.

---

## UC-A-07: Quản Lý Tài Khoản Người Dùng

| Thuộc tính | Giá trị |
|---|---|
| **Mã Use Case** | UC-A-07 |
| **Tên** | Quản lý tài khoản người dùng |
| **Tác nhân chính** | Quản trị viên (quyền `MANAGE_USERS`) |
| **Điều kiện tiên quyết** | Đã đăng nhập với quyền `MANAGE_USERS` |
| **Hậu điều kiện (thành công)** | User document được cập nhật; cache `USER_AUTH_GUARD:{id}` bị xóa để có hiệu lực ngay |

### Luồng Xem Danh Sách Người Dùng

**Bước 1:** Quản trị viên truy cập trang quản lý người dùng. Angular gọi `GET /users` với phân trang và filter tùy chọn (tìm theo username, email, trạng thái banned/verified). API trả về danh sách user với thông tin tài khoản: username, email, trạng thái `verified`, `banned`, danh sách vai trò đã gán, và `lastActiveAt`.

### Luồng Khóa/Mở Khóa Tài Khoản

**Bước 1:** Quản trị viên chọn tài khoản muốn khóa và gửi `PATCH /users/:id` với `{ banned: true }`.

**Bước 2:** `UsersService.update()` cập nhật `banned: true` trong User document. Sau đó gọi `authService.clearCachedAuthUser(id)` để xóa cache `USER_AUTH_GUARD:{id}` trong Redis2ndCacheService.

**Bước 3:** Ở request tiếp theo của người dùng bị khóa, `AuthGuard.findUserAuthGuard()` gặp cache miss → tra cứu MongoDB → nhận user mới nhất với `banned: true` → throw 403 `USER_BANNED`. Hiệu lực khóa xảy ra trong tối đa một request sau khi cache bị xóa — gần như ngay lập tức.

**Bước 4:** Mở khóa tài khoản thực hiện tương tự với `{ banned: false }`. Cache cũng được xóa để hiệu lực ngay.

### Ràng Buộc Bảo Vệ

Quản trị viên **không thể** khóa tài khoản Server Owner (`owner: true`). `UsersService` kiểm tra điều kiện này và từ chối với 403 nếu cố tình. Quản trị viên cũng không thể cập nhật tài khoản có quyền cao hơn chính mình (ví dụ: quản trị viên chỉ có `MANAGE_USERS` không thể sửa tài khoản có `ADMINISTRATOR`).

## UC-A-08: Quản Lý Vai Trò và Phân Quyền

| Thuộc tính | Giá trị |
|---|---|
| **Mã Use Case** | UC-A-08 |
| **Tên** | Quản lý vai trò và phân quyền |
| **Tác nhân chính** | Quản trị viên (quyền `MANAGE_ROLES`) |
| **Điều kiện tiên quyết** | Đã đăng nhập với quyền `MANAGE_ROLES` |
| **Hậu điều kiện (thành công)** | Role document được tạo/cập nhật; người dùng được gán/thu hồi vai trò; cache bị invalidate |

### Luồng Tạo Vai Trò

**Bước 1:** Quản trị viên tạo vai trò mới qua `POST /roles` với `{ name, color, permissions, position }`. Trường `permissions` là số nguyên chứa bitwise OR của các `UserPermission` flag muốn gán cho vai trò (ví dụ: `MANAGE_MEDIA | MANAGE_USERS = 0x2 | 0x8 = 0xA = 10`). Trường `position` xác định thứ tự ưu tiên khi hiển thị danh sách vai trò của người dùng.

**Bước 2:** `RolesService.create()` lưu Role document vào MongoDB. Role document chứa `name`, `color` (hex code cho giao diện), `permissions` (số nguyên bitfield), và `position`.

### Luồng Gán Vai Trò Cho Người Dùng

**Bước 1:** Quản trị viên xem danh sách người dùng hiện có vai trò cụ thể qua `GET /roles/:id/users`.

**Bước 2:** Quản trị viên gán hoặc thu hồi vai trò từ một hoặc nhiều người dùng đồng thời qua `PATCH /roles/:id/users` với `{ userIds: [...], action: 'add' | 'remove' }`.

**Bước 3:** `RolesService.updateUsers()` cập nhật `roles` array trong các User document liên quan. Cache `USER_AUTH_GUARD:{id}` của từng user bị ảnh hưởng được xóa qua `authService.clearCachedAuthUser()`.

**Bước 4:** Ở request tiếp theo của người dùng vừa được cập nhật vai trò, `AuthGuard` load lại dữ liệu từ MongoDB với `populated roles` mới, `permissionsService.scanPermission()` tính lại mảng `granted` (danh sách `UserPermission` values từ các roles mới). Quyền mới có hiệu lực ngay.

### Luồng Cập Nhật và Xóa Vai Trò

Cập nhật tập quyền của vai trò qua `PATCH /roles/:id` với `{ permissions }` mới. Xóa vai trò qua `DELETE /roles/:id` — vai trò bị xóa tự động được gỡ khỏi tất cả User document đang sở hữu nó. Cache của tất cả user bị ảnh hưởng cần được invalidate (được xử lý trong `RolesService.remove()`).

## UC-A-09: Cấu Hình Cloud Storage Provider

| Thuộc tính | Giá trị |
|---|---|
| **Mã Use Case** | UC-A-09 |
| **Tên** | Cấu hình cloud storage provider |
| **Tác nhân chính** | Quản trị viên (quyền `ADMINISTRATOR`) |
| **Điều kiện tiên quyết** | Đã đăng nhập với quyền `ADMINISTRATOR`; thông tin kết nối provider (credentials, bucket/container) sẵn sàng |
| **Hậu điều kiện (thành công)** | ExternalStorage document được tạo/cập nhật; provider sẵn sàng nhận DASH upload từ Transcoder |

### Luồng Thêm Provider

**Bước 1:** Quản trị viên tạo provider mới qua `POST /external-storages` với `{ name, provider, credentials }`. Trường `provider` nhận giá trị từ `CloudStorage` enum, cùng các module bổ sung (azure-blob, cloudflare-r2, firebase):

| Giá trị enum | Số | Provider |
|---|---|---|
| `CLOUDINARY` | `1` | Cloudinary |
| `IMAGEKIT` | `2` | ImageKit |
| `GOOGLE_DRIVE` | `3` | Google Drive |
| `IMGUR` | `4` | Imgur |
| `DROPBOX` | `5` | Dropbox |
| `ONEDRIVE` | `6` | OneDrive |
| `S3` | `7` | Amazon S3 (và các provider tương thích S3) |
| `FILER` | `8` | Filer |

**Bước 2:** `ExternalStoragesService.create()` lưu `ExternalStorage` document với credentials được mã hóa hoặc tham chiếu đến secret store (tùy cấu hình). `ExternalStorage` document được DaPlex-Transcoder tải khi xử lý encode job để lấy thông tin kết nối upload.

**Bước 3:** Quản trị viên kiểm tra kết nối provider. Nếu thành công, `storageId` này có thể được chọn khi trigger encode (UC-A-02 bước 3).

### Luồng Cập Nhật và Xóa

Cập nhật credentials (ví dụ: rotate API key) qua `PATCH /external-storages/:id`. Xóa provider qua `DELETE /external-storages/:id` — chỉ xóa được nếu không có encode job đang active dùng provider này.

## UC-A-10: Xem Audit Log

| Thuộc tính | Giá trị |
|---|---|
| **Mã Use Case** | UC-A-10 |
| **Tên** | Xem nhật ký kiểm toán (audit log) |
| **Tác nhân chính** | Quản trị viên (quyền `ADMINISTRATOR`) |
| **Điều kiện tiên quyết** | Đã đăng nhập với quyền `ADMINISTRATOR` |
| **Hậu điều kiện** | Không thay đổi dữ liệu — audit log là read-only |

### Luồng Chính

**Bước 1:** Quản trị viên truy cập trang audit log. Angular gọi `GET /audit-log` với filter tùy chọn (theo user, theo loại hành động, theo khoảng thời gian) và phân trang. Hoặc dùng `GET /audit-log/cursor` cho phân trang cursor khi log lớn.

**Bước 2:** API trả về danh sách `AuditLog` document, mỗi document chứa: `user` (người thực hiện), `action` (loại hành động), `targetId` (resource bị ảnh hưởng), `changes` (mảng `AuditLogChange` với `field`, `oldValue`, `newValue`), và `createdAt`.

**Bước 3:** Quản trị viên xem chi tiết một log entry qua `GET /audit-log/:id` để xem diff đầy đủ của tất cả field thay đổi.

### Tính Toàn Vẹn Audit Log

Audit log là append-only — không có endpoint DELETE hay PATCH trên `/audit-log`. Ngay cả Server Owner cũng không thể xóa log entries qua API công khai.

> Điều này đảm bảo tính toàn vẹn của nhật ký kiểm toán, cần thiết cho các môi trường yêu cầu accountability (trường học, tổ chức có nhiều admin).

Các hành động được ghi log bao gồm: tạo/cập nhật/xóa Media, tạo/cập nhật/xóa User, thay đổi vai trò, thay đổi cấu hình cloud storage. Hành động đọc (GET) không được ghi log để tránh bloat.
