# 4.3.1. Cấu Trúc Dự Án NestJS

Mục này mô tả cách tổ chức mã nguồn DaPlex-API theo kiến trúc module của NestJS — từ cấu trúc thư mục tổng quan đến các quyết định thiết kế schema, decorator, enum, và pagination. Nắm vững cấu trúc dự án giúp hiểu rõ vị trí và trách nhiệm của từng thành phần trước khi đi sâu vào logic nghiệp vụ. Nội dung bao gồm tổ chức thư mục, Mongoose schema với Snowflake ID, custom decorators, hệ thống enum tập trung, và hai chiến lược pagination.

---

## Tổ Chức Thư Mục Tổng Quan

Dự án DaPlex-API tổ chức mã nguồn theo kiến trúc module-based của NestJS, nhưng không sử dụng cấu trúc phân cấp module lồng nhau sâu — thay vào đó áp dụng mô hình phẳng với một `AppModule` root và các resource module ngang hàng nhau đăng ký tập trung trong `AppRoutingModule`. Cấu trúc `src/` gồm tám thành phần chính: các file root module (`app.module.ts`, `app-routing.module.ts`, `app-socket.module.ts`, `main.ts`), thư mục `resources/` chứa 17 feature module, thư mục `common/` chứa infrastructure dùng chung, thư mục `schemas/` chứa Mongoose schema, thư mục `decorators/` chứa custom NestJS decorators, thư mục `enums/` chứa constant enum, thư mục `utils/` chứa utility function, và thư mục `types/` chứa TypeScript ambient declarations.

> Quyết định đặt tất cả Mongoose schema trong một thư mục `schemas/` phẳng (thay vì phân tán trong từng resource module) có ưu điểm là dễ tìm và dễ maintain quan hệ giữa các schema. Khi `MediaModule` cần import `User` schema để populate `addedBy`, hoặc `HistoryModule` cần import `Media` để populate media details, chúng đều import từ cùng một nơi: `import { User, UserSchema } from '../../schemas'`. Nhược điểm là thư mục `schemas/` trở thành điểm phụ thuộc của nhiều module, nhưng vì schema là read-only definition (không chứa business logic), coupling này không gây vấn đề thực tiễn.

## Thư Mục `resources/` — Feature Modules

Mỗi resource module trong `src/resources/` tuân theo pattern nhất quán: file `*.module.ts` khai báo imports/providers/controllers/exports, file `*.controller.ts` định nghĩa HTTP endpoints, file `*.service.ts` chứa business logic, thư mục `dto/` chứa DTO cho request body và query, thư mục `entities/` chứa class cho response (serialize bằng `class-transformer`), và thư mục `guards/` nếu module có guard riêng.

Phân biệt rõ ràng giữa DTO và entity là điểm mạnh của kiến trúc này: DTO validate **input** từ client (dùng `class-validator` decorator như `@IsString()`, `@IsEmail()`, `@MinLength()`), còn entity định nghĩa **output** đến client (dùng `class-transformer` decorator như `@Exclude()`, `@Expose()`, `@Transform()`). Ví dụ `AuthService.createJwtToken()` trả về `new Jwt(accessToken, refreshToken, expiry, plainToInstance(UserDetails, user))` — `plainToInstance` chạy transformer để loại bỏ các field nhạy cảm như `password`, `activationCode`, `recoveryCode` khỏi response ngay tại lớp service.

Một số resource module quan trọng cần chú ý thêm về cấu trúc:

- `MediaModule` là module phức tạp nhất, import 14 module khác (AuthModule, AuditLogModule, GenresModule, ProductionsModule, CollectionModule, TagsModule, ChapterTypeModule, HistoryModule, PlaylistsModule, RatingsModule, CloudflareR2Module, OnedriveModule, FilerModule, S3Module, WsAdminModule, SettingsModule, ExternalStoragesModule) cùng với 4 BullMQ queue cho H264/H265/VP9/AV1. Một số import dùng `forwardRef()` để phá circular dependency (ví dụ `GenresModule` và `MediaModule` đều cần reference đến schema của nhau). Module cũng khai báo 5 consumer: `MediaConsumerH264`, `MediaConsumerH265`, `MediaConsumerVP9`, `MediaConsumerAV1` (nhận kết quả từ Transcoder), và `MediaResultConsumer` (xử lý kết quả cuối cùng sau khi tất cả codec hoàn thành).
- `AuthModule` export `AuthService`, `PermissionsModule`, và `Redis2ndCacheModule` để các module khác (MediaModule, UsersModule...) có thể sử dụng `AuthGuard` và permission checking mà không cần import trực tiếp.

## Mongoose Schemas — Thiết Kế Schema với Snowflake ID

Thư mục `schemas/` chứa 40+ Mongoose schema, tất cả đều dùng `@nestjs/mongoose` decorator pattern. Đặc điểm nổi bật là **tất cả schema dùng `bigint` làm `_id`** thay vì MongoDB ObjectId mặc định:

```typescript
@Schema({ timestamps: true })
export class Media extends TrackableDoc<Media> {
  @Prop({ type: () => BigInt, required: true })
  _id: bigint;
  // ...
}
```

Schema không tự generate `_id` — service gọi `createSnowFlakeId()` trước khi save để gán `_id` thủ công. `createSnowFlakeId()` dùng thư viện `nodejs-snowflake` với `custom_epoch` (thời điểm bắt đầu của hệ thống DaPlex, nhỏ hơn timestamp bây giờ để giảm bit cần cho phần timestamp) và `instance_id` (machine ID phân biệt các server deploy song song). Snowflake ID là `bigint` 64-bit, lưu trong MongoDB dưới BSON `Int64` — đây là lý do `MongooseModule.forRootAsync()` cần `useBigInt64: true` để Mongoose không auto-cast về `number`.

Schema `Media` là ví dụ điển hình cho các pattern được dùng xuyên suốt codebase:

**Discriminated union**: `Media` có field `type` ('movie' | 'tv'), field `movie` (subdocument `Movie`) và field `tv` (subdocument `TVShow`). Thay vì hai collection riêng, tất cả media đều trong cùng collection `medias`, phân biệt bằng `type`. Cách này đơn giản hóa query cross-type (ví dụ lấy tất cả media của một genre).

**Mixed reference**: Các array như `genres`, `studios`, `producers`, `tags`, `credits` dùng kiểu `Mixed` với `ref` thay vì `ObjectId`:
```typescript
@Prop({ type: [{ type: MongooseSchema.Types.Mixed, ref: 'Genre' }] })
genres: Types.Array<Genre>;
```
Đây là pattern để lưu cả embedded document lẫn reference ID trong cùng một field — khi `populate()`, Mongoose thay reference bằng full document; khi không populate, array chứa bigint ID. Điều này cần thiết vì `_id` là `bigint`, không phải ObjectId, nên không thể dùng `ObjectId ref` thông thường.

**`_translations`**: Schema `Media` có field `_translations: Translations<TranslatedMedia>` lưu phiên bản dịch của các field text (title, overview, slug) cho từng ngôn ngữ — ví dụ `_translations.vi.title` chứa tiêu đề tiếng Việt. Cách tiếp cận nhúng translation vào document thay vì tạo collection riêng giảm số lượng query khi load media detail (một query thay vì hai).

**Full-text search index**: `MediaSchema.index({ slug: 'text', '_translations.vi.slug': 'text' })` tạo MongoDB text index bao gồm cả slug tiếng Anh lẫn tiếng Việt, cho phép `$text: { $search: keyword }` tìm kiếm đa ngôn ngữ.

## Custom Decorators — Tích Hợp NestJS và Class-Validator

Thư mục `src/decorators/` chứa hai nhóm decorator:

**NestJS param/metadata decorators**: `@AuthUser()` là `createParamDecorator` extract `request.user` từ Fastify request context — controller dùng `@AuthUser() user: AuthUserDto` để nhận user đã xác thực mà không cần inject `AuthService`. `@RateLimitOptions({ ttl, limit, catchMode, continueWithCaptcha })` là metadata decorator gắn config lên handler method để `RateLimitInterceptor` đọc qua `Reflector`. `@AuthGuardOptions({ anonymous: true })` cho phép route chấp nhận cả request có và không có token — guard sẽ set `user.isAnonymous = true` thay vì throw 401.

**Custom class-validator constraints**: `UsernameExistConstraint`, `EmailExistConstraint`, `ReCaptchaConstraint`, `IsISO6391Constraint`, `GenreExistConstraint`, `ProductionExistConstraint` là các NestJS injectable class implement `ValidatorConstraintInterface`. Mỗi constraint được đăng ký với `useContainer(app.select(AppModule))` ở `main.ts` để class-validator có thể resolve chúng từ DI container — cho phép inject `UserModel`, `HttpService`... vào validator. Ví dụ `UsernameExistConstraint.validate()` query database để kiểm tra username đã tồn tại chưa, trả về lỗi `{ code: StatusCode.USERNAME_EXIST }` trong `defaultMessage()`. Decorator sử dụng: `@UsernameExist({ each: false })` trong `SignUpDto` — validation xảy ra tại lớp DTO, trước khi request đến service.

## Hệ Thống Enums Tập Trung

Thư mục `src/enums/` tập trung 30+ enum constant, bao gồm:

**`StatusCode`** — numeric code cho mọi loại lỗi nghiệp vụ: `EMAIL_NOT_EXIST`, `INCORRECT_PASSWORD`, `TOKEN_REVOKED`, `USER_BANNED`, `NULL_AUTHORIZATION`, `UNAUTHORIZED`, `USERNAME_EXIST`, `TOO_MANY_REQUESTS_CAPTCHA`, `TOO_MANY_REQUESTS_TTL`... Frontend nhận code này để hiển thị thông báo lỗi đúng ngôn ngữ mà không cần parse message string từ server.

**`CachePrefix`** — string prefix cho Redis key: `REFRESH_TOKEN`, `AUTH_USER`, `RATE_LIMIT`, `MEDIA_STREAM`... Định nghĩa tập trung tránh typo khi construct Redis key trong nhiều service.

**`TaskQueue`** — tên BullMQ queue: `VIDEO_TRANSCODE` (base name, append codec suffix như `video-transcode:h264`), `VIDEO_TRANSCODE_RESULT`. Dùng cùng giá trị này ở cả API lẫn Transcoder để đảm bảo hai service đăng ký đúng queue.

**`MongooseConnection`** — `DATABASE_A` và `DATABASE_B`, dùng làm tham số trong `@InjectModel(Model.name, MongooseConnection.DATABASE_A)` để inject đúng Mongoose connection.

**`VideoCodec`** — `H264 = 1`, `H265 = 2`, `VP9 = 4`, `AV1 = 8` (bitmask, khớp với tên queue của Transcoder: `video-transcode:1`, `video-transcode:2`, `video-transcode:4`, `video-transcode:8`).

**`UserPermission`** — bitmask: `MANAGE_MEDIA = 0x1`, `MANAGE_USERS = 0x2`, `ADMINISTRATOR = 0x4`... Bitmask cho phép gán nhiều permission với một số duy nhất: role có `permissions = 0x3` có cả `MANAGE_MEDIA` và `MANAGE_USERS`.

## Pagination Utilities — Hai Chiến Lược Truy Vấn

DaPlex-API cung cấp hai class paginator trong `src/utils/`, mỗi class sinh MongoDB aggregation pipeline bằng cách gọi `.build()`:

**`MongooseCursorPagination`** dùng cho các list cần infinite scroll (history, notification, search results). Thay vì page number, client gửi `pageToken` encode thông tin về vị trí cuối list (ví dụ ID của item cuối cùng). Pipeline dùng `$facet` với hai stage: `stage1` đếm tổng records (`$group: { count: $sum: 1 }`), `stage2` filter theo cursor và lấy `limit + 1` records — nếu có đúng `limit + 1` records thì `hasNextPage = true`. Cách này tránh `$skip` lớn gây full collection scan trên collection nhiều triệu records.

**`MongooseOffsetPagination`** dùng cho các list admin cần nhảy trang (genres, productions, users, audit-log). Pipeline truyền thống với `$skip` và `$limit`, kết quả trả về `{ totalPages, totalResults, results, page }`. Phù hợp với collection kích thước vừa phải (vài nghìn đến vài chục nghìn records) nơi `$skip` performance không phải vấn đề.

Cả hai class đều hỗ trợ: `search` (full-text hay partial match), `filters` (arbitrary MongoDB filter expression), `sortQuery` (client-provided sort string được validate qua `sortEnum` whitelist trước khi convert sang `{field: 1|-1}`), và `fields` projection. `.buildLookup()` variant của `MongooseOffsetPagination` nhận thêm `LookupOptions[]` để thêm `$lookup` stage join collection liên quan vào kết quả.

## Cấu Hình TypeScript Backend

File `tsconfig.json` backend khác frontend ở một số điểm quan trọng: `"experimentalDecorators": true` và `"emitDecoratorMetadata": true` (cần thiết cho NestJS DI metadata), `"target": "ES2022"`, `"module": "commonjs"` (không phải ESM — Node.js CJS là runtime target). `skipLibCheck: true` bỏ qua type errors trong `node_modules`, phổ biến trong NestJS project vì một số third-party types không luôn sạch. `strict: true` bật tất cả strict checks, bao gồm `strictNullChecks` — service methods khai báo rõ ràng `Promise<User | null>` khi query có thể không tìm thấy, buộc caller kiểm tra null trước khi dùng.
