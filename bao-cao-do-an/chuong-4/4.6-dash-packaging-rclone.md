# 4.6. Tích Hợp rclone — Lớp Trừu Tượng Cloud Storage

Mục này trình bày cách DaPlex-Transcoder tích hợp rclone như một lớp trừu tượng thống nhất cho mọi thao tác với cloud storage, thay thế các SDK riêng lẻ của từng provider. Giải pháp này cho phép hỗ trợ đa provider mà không làm phức tạp codebase, đồng thời đảm bảo credentials được quản lý an toàn và token OAuth2 được duy trì tự động.

---

## Lý Do Chọn rclone Thay Vì SDK Thuần

DaPlex-Transcoder không dùng AWS SDK, Microsoft Graph SDK, hay Google Drive SDK để upload file encode kết quả. Thay vào đó, toàn bộ thao tác với cloud storage được thực hiện qua **rclone** — công cụ command-line mã nguồn mở hỗ trợ hơn 70 backend khác nhau (S3-compatible, OneDrive, Google Drive, Dropbox, FTP, local filesystem, và nhiều hơn) với một giao diện thống nhất.

> Quyết định này giải quyết một vấn đề thiết kế cốt lõi: DaPlex cho phép admin cấu hình bất kỳ provider storage nào từ admin dashboard — nếu dùng SDK riêng cho từng provider, Transcoder phải import và maintain code cho tất cả provider đó. Với rclone, Transcoder chỉ cần biết cách tạo rclone config và gọi rclone binary — mọi logic giao tiếp với provider được rclone xử lý.

> Trade-off của cách tiếp cận này: rclone là external process thay vì in-process library — mỗi lần upload/download phải spawn một child process, có overhead khởi động nhất định. Với file video lớn (hàng GB), overhead này không đáng kể so với thời gian transfer thực sự. Với file nhỏ như thumbnail hay manifest JSON (< 1 MB), overhead spawn process có thể đáng kể nếu số lượng file lớn. Tuy nhiên DaPlex giảm thiểu số lần gọi rclone bằng cách dùng `rclone move --include` để move cả thư mục `thumbnails/` trong một lệnh thay vì từng file riêng.

---

## Dynamic Config Generation — Không Lưu Credentials Tĩnh

Thay vì có một file `rclone.conf` tĩnh với credentials cố định, DaPlex tạo config **động per-job** trong `ensureRcloneConfigExist()`. Method này kiểm tra file config hiện tại bằng `fileHelper.findInFile(configFile, '[${storage}]')` — tìm section header `[storageId]` trong config file. Nếu không tìm thấy, service query MongoDB để lấy `ExternalStorage` document, gọi `decryptToken()` để decrypt `clientSecret` (đã AES encrypt at rest), rồi `rcloneHelper.createRcloneConfig(externalStorage)` tạo config block mới và append vào file config.

Cơ chế lazy append này đảm bảo file config chỉ chứa remote cần thiết cho job hiện tại, và credentials được refresh từ MongoDB mỗi khi cần. Nếu admin thay đổi OAuth token trong khi job đang queue, lần encode tiếp theo sẽ đọc token mới từ database. Config file được đặt tại đường dẫn từ env var `RCLONE_CONFIG_FILE` — thường là một file trong `data/` directory của Transcoder, bên ngoài git repo.

---

## createRcloneConfig — Sinh Config Theo Provider

`createRcloneConfig()` trong `RcloneHelper` tạo config text theo format INI của rclone, khác nhau tùy `storage.kind`. Với **OneDrive** (`kind = 6`):

```ini
[{storageId}]
type = onedrive
client_id = {clientId}
client_secret = {clientSecret}
token = {"access_token":"...","token_type":"Bearer","refresh_token":"...","expiry":"..."}
drive_id = {driveId}
drive_type = business
no_versions = true
```

`token` là JSON string chứa OAuth2 token — rclone đọc format này trực tiếp. `drive_type = business` chỉ định OneDrive for Business (Microsoft 365), khác với OneDrive personal. `no_versions = true` tắt versioning của SharePoint để tránh tốn dung lượng khi upload file lớn bị overwrite.

Với **Google Drive** (`kind = 3`), config tương tự nhưng `type = drive` và `root_folder_id` thay vì `drive_id`. Với **S3-compatible** (`kind = 7`) và **Filer** (`kind = 8`), `createS3RcloneConfig()` tạo **hai remote lồng nhau**:

```ini
[{storageId}_s3]
type = s3
provider = Other
access_key_id = {clientId}
secret_access_key = {clientSecret}
endpoint = https://s3.example.com/
region = auto

[{storageId}]
type = alias
remote = {storageId}_s3:{bucket}/{folderPrefix}
```

Remote thứ nhất (`_s3` suffix) là S3 raw remote với endpoint và credentials. Remote thứ hai là `alias` — một wrapper cho phép đặt root tại một subfolder cụ thể của S3 bucket. Kỹ thuật alias remote này quan trọng vì `createRcloneMoveArgs()` dùng `{storageId}:{mediaId}/{streamId}` làm destination — với alias, `{storageId}:` đã tự động map về `{bucket}/{folderPrefix}`, nên path đích thực tế là `{bucket}/{folderPrefix}/{mediaId}/{streamId}`. Admin chỉ cần nhập bucket và folder prefix một lần khi cấu hình storage, Transcoder không cần biết chi tiết này.

## uploadMedia — Upload Tuần Tự Với Cancel Support

`uploadMedia()` bọc `rclone move` trong một Promise và tích hợp cơ chế cancel. Lệnh rclone được spawn với `--config`, `--low-level-retries 5`, `-v --use-json-log`, `--stats 3s`, và `move {source} {destination}`. Flag `--use-json-log` khiến rclone output progress theo format JSON thay vì text thuần — `parseRcloneUploadProgress()` parse JSON log này và extract `msg` field để hiển thị progress trên console.

`--stats 3s` khiến rclone emit progress stats mỗi 3 giây. Mỗi stats message chứa tốc độ transfer, phần trăm hoàn thành, và tổng bytes. Hàm `parseRcloneUploadProgress()` clean message bằng regex — loại bỏ newline, tab, và các dòng "Checking/Transferring" verbose để chỉ giữ summary line.

`createCancelJobChecker()` — một setInterval polling mỗi 5 giây — kiểm tra `CanceledJobIds[]` trong khi rclone đang chạy. Nếu job bị cancel, checker gọi `rclone.kill('SIGINT')`, rclone nhận signal và dừng upload đang dở. Khác với FFmpeg (dùng stdin 'q'), rclone không hỗ trợ graceful shutdown qua stdin nên SIGINT là cách duy nhất. Upload dở sẽ tạo ra file chưa hoàn chỉnh trên storage — đây là acceptable vì toàn bộ thư mục `{mediaId}` sẽ bị xóa khi job bị cancel ở phía API.

rclone dùng `move` thay vì `copy` vì file encode kết quả trên local disk không còn cần thiết sau khi upload — move tiết kiệm bước xóa file sau này. Trường hợp đặc biệt với thumbnail là `rclone sync` (khi `job.data.update = true`) thay vì `move` — update encode dùng sync để đảm bảo thư mục thumbnails trên storage đồng bộ chính xác với local, xóa thumbnail cũ nếu số lượng thay đổi.

---

## findUploadedFiles — Xác Minh Sau Upload

Sau khi tất cả encode và upload hoàn thành cho một codec, `transcode()` gọi `findUploadedFiles()` để verify tổng số file trên storage khớp với `totalExpectedFiles`. Method này gọi `rclone lsjson {remote}:{parentFolder} --files-only --exclude {thumbnailFolder}/**` — list tất cả file trong thư mục `{mediaId}`, loại trừ thumbnails. `lsjson` trả về JSON array, rclone spawn với cancel checker interval 500ms (nhanh hơn thông thường vì list operation rất nhanh).

Một điểm xử lý đặc biệt: khi exit code là `3`, method trả về empty array thay vì reject — rclone exit code 3 nghĩa là "directory not found", tình huống hợp lệ nếu storage chưa tạo thư mục. `totalExpectedFiles` được tính là `expectedVideoFiles + 1 source + 3 audio` — 3 audio files vì mỗi track encode ra hai codec (AAC và Opus) cộng với một track surround, nhưng thực tế con số 3 cố định này đến từ giả định default (surround AAC + stereo AAC + opus). Nếu thiếu file, service retry tối đa 5 lần — xử lý latency của S3-compatible listing API (eventual consistency trên một số provider như Backblaze B2).

---

## Cron Token Refresh — Duy Trì Kết Nối OAuth2

`@Cron('0 0 */5 * *')` — cron job chạy mỗi 5 ngày — gọi `handleInactiveRefreshToken()`. Method này dùng `rcloneHelper.findAllRemotes()` để list toàn bộ remote đã cấu hình trong file config (`rclone listremotes`), sau đó `getRemoteTypes()` (`rclone listremotes --long`) để biết type của từng remote. Chỉ remote có `type = onedrive` mới được refresh — Google Drive và S3 không cần (S3 key không expire; Google Drive token được refresh tự động bởi rclone khi dùng).

Cơ chế refresh: với mỗi OneDrive remote, gọi `rclone about {remote}` — lệnh này query storage usage info, nhưng tác dụng phụ là buộc rclone refresh OAuth token nếu gần hết hạn và ghi token mới vào config file.

> Đây là "trick" dùng side effect của rclone's auto-refresh: thay vì implement OAuth2 PKCE flow thủ công, DaPlex chỉ cần gọi bất kỳ authenticated rclone operation nào để trigger refresh. OneDrive access token có TTL 1 giờ, refresh token TTL 14 ngày — nếu Transcoder không chạy job nào trong 14 ngày, refresh token expire. Cron 5 ngày đảm bảo token được refresh định kỳ ngay cả khi không có job.
