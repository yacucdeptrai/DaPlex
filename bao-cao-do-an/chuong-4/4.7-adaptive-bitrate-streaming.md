# 4.7. Adaptive Bitrate Streaming — Tái Tạo Manifest và Phát Video Phía Client

Mục này trình bày cách DaPlex triển khai Adaptive Bitrate Streaming hoàn toàn phía client, không phụ thuộc vào server để sinh manifest. Kiến trúc bao gồm lớp `StreamManifestHelper` tái tạo MPD/HLS từ JSON, tích hợp Vidstack với dash.js, và các cơ chế chọn chất lượng video, audio track, codec, cũng như render phụ đề qua WebAssembly.

---

## Kiến Trúc Manifest Client-Side — Loại Bỏ Server-Side XML Generation

DaPlex áp dụng một quyết định thiết kế không phổ biến trong lĩnh vực video streaming: **manifest DASH và HLS không được sinh ra phía server** mà được tái tạo hoàn toàn phía client từ dữ liệu JSON có cấu trúc. Khi người dùng mở trang xem video, Angular frontend tải file `manifest_{codec}.json` từ CDN — file này chứa đầy đủ thông tin về tất cả track video, track audio, byte range của từng segment và init range. Lớp `StreamManifestHelper` trong Angular đọc JSON này và xây dựng MPD hoặc m3u8 ngay trên trình duyệt, rồi truyền trực tiếp vào Vidstack/dash.js dưới dạng parsed object hoặc data URI.

> Quyết định này xuất phát từ một quan sát quan trọng: với CMAF single-file (mô hình "onDemand" của MP4Box), toàn bộ nội dung video nằm trong một file MP4 duy nhất — các "segment" thực chất chỉ là byte range của file đó. Manifest chỉ là một bản đồ ánh xạ timestamp → byte offset, hoàn toàn có thể được sinh từ dữ liệu đã lưu sẵn trong JSON. Nếu server phải sinh MPD XML mỗi lần player khởi động, đó là một HTTP round-trip không cần thiết và tạo ra một endpoint thêm cần maintain. Thay vào đó, `manifest_{codec}.json` là file tĩnh, có thể CDN-cache vĩnh viễn, và tất cả logic dựng manifest nằm trong Angular bundle — stateless và không cần server tham gia.

## StreamManifestHelper — Ba Phương Thức Tái Tạo

`StreamManifestHelper` cung cấp ba phương thức tái tạo manifest từ cùng một `StreamManifest` JSON object, phục vụ ba mục đích khác nhau trong pipeline phát video.

Phương thức `convertToM3U8(manifest, baseUrl, options)` sinh **HLS master playlist** dưới dạng data URI (`data:application/x-mpegurl;charset=UTF-8,...`). Playlist bắt đầu với `#EXTM3U` và `#EXT-X-VERSION:7`, sau đó với mỗi entry trong `manifest.audioTracks`, một `#EXT-X-MEDIA` line được tạo với `TYPE=AUDIO`, `GROUP-ID="audio"`, và `URI` trỏ đến một data URI chứa HLS audio playlist nội tuyến. Nếu `!options.opus`, track audio có codec Opus bị loại khỏi output. Track audio có số kênh vượt quá 6 (`channels > 6`) cũng bị bỏ qua vì most browser HLS implementations không hỗ trợ multi-channel audio qua MSE. Với mỗi video track, `#EXT-X-STREAM-INF` line được tạo kèm `BANDWIDTH`, `RESOLUTION`, `CODECS`, và `AUDIO="audio"`. `generatePlaylist()` là hàm nội bộ tạo từng HLS media playlist — nó dùng `#EXT-X-MAP:URI="...",BYTERANGE="{length}@{offset}"` cho init segment, rồi mỗi segment dùng `#EXT-X-BYTERANGE:{length}@{offset}` theo chuẩn HLS byte-range. Toàn bộ playlist này được encode thành data URI và nhúng trực tiếp vào master playlist — không có bất kỳ HTTP request phụ nào khi player parse master playlist.

Phương thức `convertToMPD(manifest, baseUrl, options)` sinh **MPD XML string** theo chuẩn DASH. Phương thức này dùng `lodash groupBy` để gom các `videoTracks` theo `codecID` — video H264, H265, và AV1 mỗi loại tạo thành một `AdaptationSet` riêng với `mimeType="video/mp4"` và `codecs` tương ứng. Nếu `!options.av1`, AdaptationSet AV1 bị loại. Bên trong mỗi AdaptationSet, từng track (độ phân giải khác nhau) là một `Representation` với `bandwidth`, `width`, `height`. `SegmentBase indexRange="{start}-{end}"` và `Initialization range="{start}-{end}"` dùng byte range từ JSON. Với audio, mỗi track là một AdaptationSet với `mimeType="audio/mp4"`, kèm `lang` attribute nếu có. Track audio với `channels >= 7` bị loại khỏi DASH MPD vì browser dash.js không decode multi-channel AAC qua MSE trong hầu hết trường hợp.

Phương thức `convertToParsedDash(manifest, baseUrl, options)` là phương thức quan trọng nhất về mặt kỹ thuật: nó sinh **dash.js native parsed manifest object** thay vì XML string. dash.js thông thường nhận URL, tải MPD XML, parse thành JavaScript object rồi mới phát. `convertToParsedDash()` bỏ qua toàn bộ bước đó bằng cách trực tiếp xây dựng JavaScript object theo đúng schema mà dash.js parser output — `Period`, `AdaptationSet`, `Representation`, `SegmentBase` với `Initialization` và `indexRange` như các plain JavaScript object. `parsedManifest.url = baseUrl.replace(':path', '')` thiết lập base URL để resolve relative URLs bên trong manifest. Khi object này được truyền vào dash.js, player bắt đầu phát ngay lập tức mà không cần bất kỳ network request nào để tải manifest — tất cả data đã sẵn sàng. Đây là kỹ thuật tương tự như Server-Side Rendering nhưng áp dụng cho media manifest: giảm cold-start latency đáng kể vì player không phải chờ tải và parse XML.

## Vidstack + dash.js — Tích Hợp Hai Lớp

`BaseVideoPlayerComponent` trong Angular dùng **Vidstack** như UI layer (controls, progress bar, quality selector) và **dash.js** như MSE engine (adaptive streaming, codec negotiation). Đây là kiến trúc hai lớp: Vidstack quản lý UI state và cung cấp reactive store; dash.js xử lý toàn bộ việc tải segment, decode, và ABR algorithm.

Khi Vidstack provider được khởi tạo và detect là DASH media, `BaseVideoPlayerComponent` inject dash.js vào Vidstack qua `provider.library = DashMediaPlayer`. Thay vì Vidstack tải MPD URL thông thường, component truyền parsed manifest object từ `convertToParsedDash()` vào dash.js API trực tiếp. Configuration của dash.js được set qua `provider.config`:

```typescript
{
  streaming: {
    abr: { autoSwitchBitrate: { audio: false } },
    trackSwitchMode: { video: 'alwaysReplace', audio: 'alwaysReplace' },
    saveLastMediaSettingsForCurrentStreamingSession: false
  }
}
```

| Cấu hình | Giá trị | Ý nghĩa |
|---|---|---|
| `autoSwitchBitrate.audio` | `false` | Tắt ABR cho audio track — audio phải được select thủ công bởi user; nếu để `true`, dash.js có thể tự động switch sang audio track bitrate thấp hơn, không mong muốn khi user đang nghe một ngôn ngữ cụ thể |
| `trackSwitchMode` | `'alwaysReplace'` | Khi switch track, dash.js tải và thay thế buffer ngay lập tức thay vì chờ buffer cũ hết — người dùng thấy hiệu ứng switch nhanh hơn |
| `saveLastMediaSettingsForCurrentStreamingSession` | `false` | Tắt cơ chế lưu trữ session của dash.js — DaPlex tự quản lý state qua Angular state management nên không cần dash.js duplicate việc này |

## ABR Video Quality — Automatic và Manual Selection

`changeVideoQuality(height: number)` trong `BaseVideoPlayerComponent` xử lý cả hai chế độ quality selection. Khi `height === 0`, đây là "Auto" mode: `player.qualities.autoSelect()` cho phép Vidstack/dash.js tự chọn quality dựa trên bandwidth estimation. Khi `height > 0`, method tìm quality object trong Vidstack store bằng cách match `quality.height === height`, rồi set `quality.selected = true` — Vidstack propagate selection này xuống dash.js, dash.js ngừng ABR và lock vào representation cụ thể.

`subscribeMediaStore()` thiết lập reactive subscriptions đến Vidstack store:

```typescript
effect(() => {
  const qualities = player.qualities;
  this.videoQualities = [...qualities].reverse(); // hiển thị 4K → 480p
});

effect(() => {
  const audioTracks = player.audioTracks;
  this.audioTracks = [...audioTracks];
});
```

`[...qualities].reverse()` đảo thứ tự vì Vidstack expose qualities từ thấp đến cao nhưng UI convention hiển thị từ cao xuống thấp. `player.audioTracks` là danh sách AudioTrack objects — mỗi track có `label` (được Transcoder đặt theo format `"{language} - {codec} - {channels}ch"`), `language`, và `selected`. Subscription này reactive theo Vidstack signal system — khi dash.js cập nhật danh sách quality (ví dụ khi bandwidth giảm đột ngột và một số quality bị loại tạm), Angular tự động re-render danh sách.

## Audio Track Selection và Codec Detection

`changeAudioTrack(index: number)` select audio track dựa trên index. Method đọc `track.label.split(' - ')[2]` để extract codec string từ label (phần thứ ba, ví dụ "AAC" hay "Opus"). Track được select bằng `audioTrack.selected = true`. Vì `autoSwitchBitrate.audio = false`, dash.js sẽ giữ track được select cho đến khi user thay đổi — không có ABR override nào.

Codec selection logic ở phía component quyết định file manifest nào (`manifest_h264.json`, `manifest_h265.json`, hay `manifest_av1.json`) được load khi khởi tạo player. Logic này kiểm tra `options.av1` (user opt-in từ settings) và browser codec support qua `MediaSource.isTypeSupported('video/mp4; codecs="av01..."')`. H265 cũng tương tự. Kết quả là default luôn là H264 (broad compatibility), và H265/AV1 chỉ được dùng khi browser support đã được verify runtime. Điều này khác với server-side content negotiation vì không cần server biết client capabilities — client tự resolve.

Với Opus audio, `convertToM3U8(..., { opus: false })` và `convertToMPD(..., { opus: false })` cho phép filter Opus tracks nếu browser không support Opus trong MP4 container. Safari và một số older browsers không decode `audio/mp4; codecs="opus"` qua MSE dù họ support Opus trong Ogg. DaPlex kiểm tra `MediaSource.isTypeSupported('audio/mp4; codecs="opus"')` trước khi enable Opus option.

## Subtitle Rendering — LibASS qua WASM

`BaseVideoPlayerComponent` tích hợp `LibASSTextRenderer` để render phụ đề định dạng SSA/ASS — format phụ đề phức tạp với font positioning, animation, và karaoke effects. Thư viện jassub compile LibASS (C library) sang WebAssembly, cho phép chạy LibASS nguyên bản trong trình duyệt. `LibASSTextRenderer` nhận Vidstack player instance và một canvas element; jassub render ASS subtitle frame lên canvas, overlay lên video element.

Option `offscreenRender` được disable trên mobile vì OffscreenCanvas API không có đủ performance trên mobile GPU. `loadSubtitleFontList` nhận danh sách font URL để tải custom font — quan trọng với phụ đề tiếng Việt hay tiếng Nhật cần font đặc biệt. Khi track phụ đề được chọn, component tải file `.ass` từ CDN và pass cho LibASS renderer; khi track thay đổi, renderer được reset và tải file mới. VTT (WebVTT) subtitle thông thường được Vidstack handle native không qua jassub — jassub chỉ dành cho SSA/ASS.
